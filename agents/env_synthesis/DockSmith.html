DockSmith

Scaling Reliable Coding Environments via an Agentic Docker Builder
↗ arXiv:2602.00592 Qwen3-Coder-30B-A3B Multi-Docker-Eval SOTA · 2026.02
DockSmith 的核心主张：Docker 环境构建不只是预处理步骤，而是一种需要专门训练的 Agentic 能力。这项能力包含长时程工具调用、依赖冲突推理和失败恢复——而这些 skill 并非 Docker 独有，会迁移到 SWE、Terminal 等下游任务。基于 SWE-Factory 风格流水线（扩展了 Loop-Detection Controller 和 Cross-Task Success Memory），收集 30K+ 可执行验证的 Docker 构建轨迹，训练 Qwen3-Coder-30B-A3B 达到 Multi-Docker-Eval 39.72% F2P / 58.28% Commit Rate，并在 SWE-bench Verified（+2.25pt）、SWE-bench Multilingual（+2.09pt）、TerminalBench 2.0（+3.37pt）上同步提升。
01

核心重构：Docker 构建作为 Agentic 能力

传统视角把 Docker 环境构建视为「数据准备」的工程杂活——弄好就扔，不值得深入研究。DockSmith 打破了这个假设。

🔍
为什么 Docker 构建是 Agentic 任务？

一个真实的 Docker 构建任务，agent 需要：

· 长时程工具调用：搜索 CI 配置 → 分析依赖清单 → 生成 Dockerfile → 执行构建 → 解析错误 → 修改 → 重试……往往需要 10–30+ 轮迭代
· 跨层依赖推理：OS 包（apt）→ 语言运行时版本 → 项目依赖（pip/npm/cargo）→ 测试框架，四层之间存在隐式约束
· 失败恢复：构建错误往往级联出现，需要判断根因（是 apt 源问题还是版本冲突）而不是逐行修补
· 不确定性处理：相同语言在不同 repo 里构建路径完全不同

这些特性让 Docker 构建与 SWE（bug 修复）在 skill 结构上高度重叠——都需要「读懂 codebase → 定位问题 → 迭代修复 → 验证」的循环能力。这正是 DockSmith 迁移效应的机制来源。

💡
DockSmith 的关键 insight：Docker 构建的训练 signal 和 SWE 的训练 signal 共享底层 reasoning skill。因此，在 Docker 数据上训练不会只改善 Docker 任务，而是会提升模型整体的「在执行反馈中迭代修复」能力。这一发现改变了我们对 mid-training 数据多样性的理解。
02

四 Agent 流水线

DockSmith 基于 SWE-Factory 风格的流水线，由四个专职 agent 协作完成 Docker 环境构建，每个 agent 负责单一职责。

Agent 01
Context Retrieval Agent
通过迭代工具调用检索 repo 中的关键信息：

· 依赖清单（requirements.txt, go.mod, Cargo.toml…）
· CI 配置（.github/workflows/, .travis.yml…）
· 测试入口点（pytest 配置, Makefile…）

职责边界：只读取信息，不生成任何输出。
Agent 02
Dockerfile Agent
基于 Context Retrieval 的信息和执行反馈，合成或修补 Dockerfile。

核心难点：在「正确性」和「最简性」之间权衡——过度安装依赖会掩盖真实错误，缺少依赖则直接失败。

接收 Test Analysis Agent 的结构化诊断作为修补指导。
Agent 03
Eval Script Agent
生成可执行的工作区配置 + 测试调用脚本，在容器内运行。

需要正确处理：路径设置、环境变量注入、测试框架参数、超时控制。

eval script 的错误（eval_patch_error）在 DockSmith 训练后下降了 79.1%——是改善最显著的错误类别。
Agent 04
Test Analysis Agent
执行构建/测试流水线，将原始日志提炼为结构化失败摘要，供下一轮修复迭代使用。

输出结构化 JSON：{layer, root_cause, suggested_fix, is_terminal}

is_terminal=true：标记「无解」情形（依赖 404、编译器版本不存在），提前退出防止无限循环。

↑ structured failure summary (JSON) ─────────────────────────── feedback routing
· Dockerfile 不变时：复用已构建 image，只重跑 eval script → 5× 速度提升
· Loop-Detection Controller 监控此环，防止无效循环（见 §03）
· Cross-Task Memory 在 repo 间复用成功方案（见 §03）
03

两大创新机制

DockSmith 在 SWE-Factory 的基础上新增了两个机制，专门针对 Docker 构建的两类特有失败模式。

🔁
Loop-Detection Controller
防死循环

问题来源：Docker 构建中常见的「死循环」模式——agent 在同一个错误上反复尝试相同修法，消耗大量 token 却毫无进展。典型案例：

· Dockerfile Agent 反复修改同一行 apt-get install 却没有改包名
· Eval Script Agent 在同一测试路径上循环添加/删除参数
· 两个 agent 互相推卸责任（Dockerfile 说是 script 问题，script 说是 Dockerfile 问题）

工作机制：控制器持续监控 agent 调用序列 和 执行失败签名（error type + error location 的哈希）。
检测条件：同一 agent 组合 + 同一失败签名，连续出现 N 轮仍无改善。
介入方式：强制多样化——激活替代 agent 或切换策略（如放弃当前 Dockerfile，从 CI 配置重新生成）。

训练后，diag_loop_error（循环诊断错误）下降 59.8%，diag_vague_error（模糊诊断）下降 71.7%。

🧠
Cross-Task Success Memory
跨任务知识复用

问题来源：不同 repo 在构建上往往有相似的依赖模式（同一生态的项目通常用同一套工具链）。SWE-Factory 的 memory pool 只在单个任务的多次尝试之间共享经验，无法跨 repo 复用。

工作机制：将已验证成功的 (Dockerfile, eval_script) 对持久化存储，并以 repo 特征（语言 + 依赖栈 + 构建工具）为索引。
当处理新 repo 时，系统检索最相似的历史成功案例，以轻量级 demonstration 的形式注入当前 agent 的 context（类似 few-shot）。

效果：减少「重新发明轮子」——agent 不需要从零学 Go modules 的构建方式，而是直接参考已验证过的类似 repo 的 Dockerfile 结构。显著降低 docker_env_error（−48.7%）。

ℹ
Loop-Detection 和 Cross-Task Memory 是互补的：前者处理「当前 repo 内的失败」，后者处理「跨 repo 的知识迁移」。两者都是为了提升轨迹质量——生成的训练数据中更少包含无效循环，更多包含高质量的问题诊断和修复推理。
04

训练数据三层筛选

原始 repo 池：15,000+ 个 repo（≥500 stars，≥200 forks），覆盖 10 种语言（Python · PHP · TypeScript · Go · C++ · JavaScript · Java · Rust · C · Ruby）。从 ~200,000 个候选 PR 中筛选，只保留 merged 且包含实质代码变更 + 测试修改的 PR。

L1
Cross-Language Balancing — 跨语言均衡
不同语言的 repo 数量差距悬殊（Python 远多于 Ruby/Rust）。如果不控制，模型会严重偏向 Python，在其他语言上泛化差。

解法：对每种语言设置 token 贡献上限，防止长尾语言在 batch 中占比过低。每批次从 10 种语言均匀采样，语言分布在训练过程中保持稳定。
↓
L2
Redundancy Filtering — 冗余轨迹过滤
过滤掉低质量轨迹，判断标准（满足任一即过滤）：

· 反复调用同一 agent 模块超过阈值次数（说明陷入无效循环，Loop-Detection 没有完全解决）
· 对话轮数过多（超长轨迹通常包含大量无效尝试）
· 消息数量异常多（消息膨胀通常来自错误处理不当）

目标：保留「以最少步骤解决问题」的高效轨迹，让模型学习紧凑的问题解决策略而非低效的反复试错。
↓
L3
Deduplication — 评估集去污染
在 repo 级别去重，与所有评估 benchmark 做交叉比对：

· Multi-Docker-Eval 的 40 个 repo
· SWE-bench Verified 的 repo 集
· SWE-bench Multilingual 的 repo 集
· TerminalBench 2.0 的 repo 集

相关 repo 的所有数据从训练集中移除，确保评估结果不含数据泄露。
05

复杂度课程学习

这是 DockSmith 数据构建的精华：不是把所有过滤后的轨迹平等对待，而是根据 Dockerfile 复杂度 对训练实例分组，并用 课程学习 策略混合采样。

📐
复杂度打分公式
Acceptance Shaping
complexity score · Dockerfile d 
Score(d) = 0.5 · L(d)  # 非空行数（规模代理）
         + 5.0 · R(d)  # RUN 指令数（操作步骤数）
         + 3.0 · P(d)  # distinct apt packages 数（依赖复杂度）

三个维度分别捕捉不同层次的复杂度：L 反映文件规模，R 反映操作步骤的多少，P 反映依赖管理的难度。RUN 指令权重最高（5×），因为每个 RUN 都可能引入新的错误点。

Easy : Medium : Hard = 1 : 2 : 2——刻意过采样中等和困难实例，让模型在有挑战性的环境里多练，而不是被大量简单实例稀释学习信号。

07

迁移学习：Docker ↔ SWE / Terminal 双向增益

这是 DockSmith 最反直觉也最有价值的发现：在 Docker 构建数据上训练，可以提升 SWE 和 Terminal 任务的表现；反过来，SWE 数据也可以提升 Docker 构建表现。

实验方式：以 token 级别混合 SWE 轨迹和 Docker 轨迹，测试不同混合比例（SWE : Docker）。

SWE : Docker 比例 SWE-bench Verified SWE-bench Multilingual TerminalBench 2.0
Only SWE（baseline） 49.65% 31.83% 10.67%
1 : 0.5 51.75% 33.33% 14.04% ▲
1 : 1 51.90% ▲ 33.92% ▲ 11.38%
1 : 2 49.55% 31.75% 11.52%

峰值增益：SWE.V +2.25pt，SWE.M +2.09pt，Terminal +3.37pt。

⚠
混合比例有「甜区」：1:0.5 ~ 1:1 之间效果最好。Docker 数据过多（1:2）会稀释 SWE skill，导致 SWE 性能反而下降。这个 trade-off 对我们的 mid-training 数据混合策略有直接参考价值。

反向实验（Docker 数据 + 少量 SWE）也成立：

训练数据 SWE.V SWE.M Terminal Multi-Docker-Eval
Docker only 33.50% 18.25% 7.16% 34.43%
Docker + 0.5× SWE 47.45% 31.00% 10.11% 35.63%

加入少量 SWE 数据后，Docker 任务本身也提升了（34.43% → 35.63%）——SWE 的 code-reading skill 迁移到了 Docker 构建的依赖分析上。

09

对我们的启发

D^env 构建可以加入 Loop-Detection 机制
我们的 D^env 数据收集流水线（daVinci-Env 风格）同样存在 agent 死循环问题。DockSmith 的 Loop-Detection Controller（监控 agent 调用序列 + 失败签名哈希，相同组合 N 轮无改善则强制多样化）可以直接迁移到我们的环境构建流水线，减少无效轨迹数量，提高数据质量。
复杂度课程学习的采样策略值得参考
DockSmith 的 Score(d) = 0.5·L + 5·R + 3·P 公式和 Easy:Medium:Hard = 1:2:2 采样比是一个经过消融验证的配方。对我们的 D^env 实例，可以设计类似的复杂度分层：pass@1 在 25%–75% 之间的实例（参考 daVinci-Env 的 1-2/4 策略），在这个区间内再按 Dockerfile/issue 复杂度做二次分层，进一步优化学习信号密度。
混合比例甜区：D^ctx : D^env ≈ 1 : 0.5 ~ 1 : 1
DockSmith 的混合实验表明，适度混合（1:0.5 ~ 1:1）是最优区间，过多 Docker/Env 数据反而会损害 SWE 性能。这直接映射到我们的 Proposal：D^ctx（PR 数据）和 D^env（执行轨迹数据）的混合比例应该在 1:0.5 ~ 1:1 左右，不应让 D^env 占比过高。这是有定量实验支撑的建议，而不只是直觉。
Docker 构建能力 = 通用 Agentic 基础能力，不只是预处理
如果我们在未来需要自己构建环境（而不是直接复用 OpenSWE），应该考虑先训练一个 DockSmith 风格的「环境构建 agent」，而不是用 GPT-4.1 或 Claude 直接做环境构建。DockSmith 已经证明：在这个 task 上专门训练的 30B 模型可以超过 Claude-Sonnet-4，且构建出的 skill 会迁移到主任务上，一份数据两份收益。
Cross-Language Balancing 是多语言 D^env 的必须项
如果 Phase 1 D^env 要覆盖多语言（参考 TerminalTraj 的 8 语言设计），必须做跨语言均衡，否则 Python 数据会主导训练，其他语言能力形同虚设。DockSmith 用 per-language token 上限解决这个问题，我们可以在数据混合配方里设置类似的语言分布约束（例如每种语言不超过总 D^env token 的 30%）。
→
建议行动：DockSmith 的模型和 Docker 构建轨迹已公开。优先考虑：① 用 DockSmith 驱动我们自己的环境构建流水线（替代 GPT-4.1）；② 将 DockSmith 的 Loop-Detection 逻辑移植到 daVinci-Env 风格的流水线中；③ 在 D^ctx : D^env 混合比例设计时，参考 1:0.5 的甜区结论。
