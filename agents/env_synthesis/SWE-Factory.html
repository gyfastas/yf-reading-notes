SWE-Factory

Your Automated Factory for Issue Resolution Training Data and Evaluation Benchmarks
↗ arXiv:2506.10954 github.com/DeepSoftwareAnalytics/swe-factory SE Data Factory · 2025.06
SWE-Factory 是第一个开源的、全自动、跨语言的 GitHub issue resolution 数据构建流水线。它针对现有 SWE-bench 风格流水线中的三大瓶颈——二进制测试文件缺失、环境构建依赖大量人工、fail2pass 验证需要手写 parser——分别提出了自动化解决方案：Binary Test File Recovery、SWE-Builder（四 Agent 多智能体系统）和 Exit-Code-Based Log Parsing。在 671 个真实 issue（4 种语言）上的实验表明，SWE-Builder 以 GPT-4.1 mini 为基座可达到 50.2% F2P / 64.8% Output Rate，平均成本仅 $0.047/instance；exit-code 验证方法的 F1 高达 0.99。此外，基于 SWE-Factory 收集的 2,809 条轨迹进行 SFT，可使 Qwen2.5-Coder-14B 在 SWE-bench Verified 上的 resolve rate 从 5.8% 提升到 21.0%。
01

三大瓶颈

传统的 GitHub issue resolution 数据集构建遵循 SWE-bench 的四阶段流水线：Repository Selection → Issue Data Collection → Evaluation Environment Construction → Fail2Pass Validation。SWE-Factory 指出其中存在三个尚未被系统解决的关键痛点。

🗂️
P1 · 二进制测试文件缺失
数据收集

当 PR 的 test patch 包含二进制文件（如 .png, .tif, .zip）时，GitHub API 返回的 patch 中这些文件的内容为空（显示为 "Binary files ... differ"）。这导致：

· 测试缺少必要的输入资源；
· git apply 无法成功应用 test patch， codebase 卡在 base commit。

在 Pillow 中 30.43% 的实例受此影响，matplotlib 中 22.10%。这些 issue 被静默丢弃，限制了数据的多样性，尤其对多模态理解任务不利。

🔧
P2 · 评估环境构建依赖大量人工
环境构建

不同仓库使用不同语言、依赖管理工具和测试框架；同一仓库的不同版本也可能需要不同的安装命令。现有方法需要为每个 task instance 手动编写 Dockerfile 和测试脚本，人力成本极高。

📋
P3 · Fail2Pass 验证需要手写 Parser
验证

现有 fail2pass 验证阶段需要研究者手动 inspect 测试日志，并为不同仓库、不同测试框架编写定制的日志解析代码。当日志格式随项目演进发生变化时，parser 需要不断重写。

02

SWE-Builder：四 Agent 多智能体系统

为了解决 P2（环境构建），SWE-Factory 提出了 SWE-Builder——一个 LLM-based 多智能体系统，通过四个专职 agent 的协作自动完成评估环境的构建与测试脚本的生成。

Agent 01
Repository Explorer
自动收集仓库的关键构建信息：

· 环境依赖（requirements.txt, pom.xml 等）
· 测试命令（pytest, mvn test 等）
· 文档与安装脚本（CONTRIBUTING.md）

通过 browse_file, browse_dir, search_keyword 等工具迭代检索，直到收集到足够信息。
Agent 02
Environment Manager
基于 Repository Explorer 的信息生成或更新 Dockerfile，将复杂的环境配置脚本化为可复现的容器镜像。

保留生成历史，在失败时可用前序版本作为 fallback，确保连续性。
Agent 03
Test Manager
生成在容器内运行的测试执行脚本。

职责包括：处理二进制文件变更（插入下载/删除命令）、路径设置、环境变量注入，以及最关键的一步——在脚本末尾输出唯一的 exit code 标记：

rc=$?; echo "OMNIGRIL_EXIT_CODE=$rc"

这为后续全自动验证提供了标准化接口。
Agent 04
Test Analyst
执行构建与测试，提炼失败摘要并 orchestrate 下一轮迭代。

使用 build_image, start_container, run_eval 验证环境。若失败，分析日志并生成 targeted optimization plan，召回相关 agent 继续迭代。最多迭代 5 轮。
ℹ
Evaluation Environment Memory Pool：SWE-Builder 维护一个 per-repo 的记忆池，复用同一仓库此前成功的环境配置。消融实验表明，移除 Memory Pool 后 F2P Rate 平均下降 5.2%，且处理时间延长。
💡
DockSmith 正是在 SWE-Builder 的这一架构上做了扩展：保留了 Context Retrieval (Explorer)、Dockerfile Agent (Environment Manager)、Eval Script Agent (Test Manager) 和 Test Analysis Agent 的四 Agent 分工，同时新增了 Loop-Detection Controller 和 Cross-Task Success Memory，将 per-task memory 升级为跨 repo 复用。→ 查看 DockSmith 精读笔记
03

Exit-Code-Based Fail2Pass 验证

为了解决 P3（验证），SWE-Factory 提出了一种全新的全自动验证方法：利用主流测试框架（pytest、Jest、Maven 等）都通过 exit code 报告测试结果这一事实，直接从测试脚本捕获 exit code，而无需解析五花八门的日志文本。

🎯
标准化接口：OMNIGRIL_EXIT_CODE

Test Manager 被 prompt 要求在脚本末尾显式捕获并输出 exit code：

rc=$?
echo "OMNIGRIL_EXIT_CODE=$rc"

外部验证器只需要 grep 这一固定标记，即可 100% 准确地知道测试是 pass (exit=0) 还是 fail (exit≠0)。

为了验证 exit-code 方法的准确性，研究者对 1,201 个 task instance 进行了人工评估：三位经验丰富的研究者独立 inspect 测试日志，通过讨论解决分歧，建立 ground truth。结果如下：

100%
Precision
98.3%
Recall
0.991
F1 Score
0
False Positives

16 个 False Negative 全部源于 LLM 生成的测试脚本错误（4 个标识符拼写错误，12 个因误加 set -e 导致脚本提前终止），而非 exit code 本身的不可靠。手动修正这些脚本错误后，F1 可达 1.0。

✓
这一发现极具工程价值：只要强制规范测试脚本的输出接口，exit code 就是判断测试状态的黄金标准。它完全消除了为不同框架/仓库维护定制 parser 的人力负担。
04

实验与消融研究

数据集：SweSetupBench —— 从 12 个知名开源仓库中分层采样得到的 671 个 issue，覆盖 Python、Java、TypeScript、JavaScript 四种语言。

基座模型：GPT-4.1 mini、Kimi-K2、DeepSeek-V3-0324。

主实验：SWE-Builder 效果

ModelF2P Rate (↑)Output Rate (↑)Time (min↓)Cost ($↓)
GPT-4.1 mini50.2% (337/671)64.8%26.30.047
Kimi-K247.8% (321/671)63.2%30.20.056

GPT-4.1 mini 在 F2P 和 Output Rate 上均表现最佳；DeepSeek-V3 虽然 F2P 略低，但性价比最高（$0.037/instance，约 11.4 valid instances per dollar），是大规模数据构建的优选。

⚠
Execution Feedback 是最关键的组件：移除后 F2P Rate 从 50.2% 暴跌至 0.3%。这说明「构建环境 → 执行测试 → 分析错误 → 迭代修复」的闭环反馈是 SWE-Builder 生效的核心机制。没有真实执行反馈，仅靠静态代码审查几乎无法构建出可用的评估环境。
05

SFT 探索：数据质量 = 模型能力

为了验证 SWE-Factory 收集的数据能否真正提升模型的软件工程能力，作者用 Kimi-K2 + DeepSWE agent 框架在 2,809 个自动收集的 Python task instance 上生成了轨迹，并对 5 个模型做了 3 epoch 的全参数 SFT。

ModelBase RRSFT RRΔ
Qwen2.5-Coder-14B5.8%21.0%+15.2
Qwen2.5-Coder-7B4.4%13.6%+9.2
Qwen2.5-Coder-3B0.0%3.4%+3.4
Qwen3-8B3.4%16.2%+12.8
Llama-3.1-8B1.2%5.2%+4.0

所有模型在 SWE-bench Verified 和 SWE-bench Lite 上均有显著提升，Empty Patch Rate 和 Tool Call Failure Rate 同步下降。这说明 SWE-Factory 不仅是一个数据构建工具，更是一个可扩展的模型能力增强引擎。

06

对我们的启发

Docker 环境构建可直接继承 SWE-Builder 的四 Agent 架构
DockSmith 已经证明这条路径的正确性。如果我们要构建自己的环境构建流水线，Repository Explorer → Environment Manager → Test Manager → Test Analyst 的分工模式是一个经过实验验证的可靠起点。DockSmith 在此基础上新增的 Loop-Detection 和 Cross-Task Memory 可以视为「第二代」扩展。
Exit-Code 验证法是环境数据建设的「基础设施」
与其为每个 repo/测试框架写定制 parser，不如强制规范测试脚本输出统一的 exit code 标记。这是从「工程脚本」向「自动化工厂」质变的关键一步。F1=0.99 的验证精度足以支撑大规模无人值守的数据生产。
Memory Pool 值得从 per-repo 升级到 cross-repo
SWE-Factory 的 Memory Pool 只在一个 task 的多次尝试间复用成功经验。DockSmith 的 Cross-Task Memory 将其扩展到了 repo 间——这是 F2P 提升的重要来源。我们在设计 D^env 收集流水线时，应尽早引入跨仓库环境配置复用机制，减少重复劳动。
Execution Feedback 是不可妥协的核心
消融实验中，移除 Execution Feedback 后 F2P 几近归零。这提醒我们：任何环境构建 agent 的 prompt / 训练数据都必须包含「构建 → 执行 → 观察错误 → 迭代」的完整闭环。仅让模型「猜测」Dockerfile 而不与真实执行环境交互，是行不通的。
Binary Test File 修复是个容易被低估的数据增量来源
在 Pillow 等视觉/多模态相关 repo 中，超过 20% 的 issue 因二进制 test patch 缺失而被丢弃。若我们要收集 VLM-related SE 数据，这一修复步骤将直接决定数据集的大小与多样性。
→
建议行动：① 在 D^env 流水线中强制采用 OMNIGRIL_EXIT_CODE 风格的 exit-code 验证接口，替代人工 parser；② 以 SWE-Builder 四 Agent 架构为起点设计环境构建模块，逐步叠加 Loop-Detection 与 Cross-Task Memory；③ 如果涉及图像/二进制测试资源，优先集成 Binary Test File Recovery 模块。
