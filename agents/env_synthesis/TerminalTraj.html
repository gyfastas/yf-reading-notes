TerminalTraj

Large-Scale Terminal Agentic Trajectory Generation from Dockerized Environments
↗ arXiv:2602.01244 🤗 m-a-p/TerminalTraj ⌥ Wusiwei0410/TerminalTraj Manchester + IQuest · 2026.02
本文提出 TerminalTraj——一个大规模 terminal agentic 轨迹生成框架。核心贡献：针对 terminal 任务的两大固有挑战（Executability 和 Verifiability），设计了一套「先建 Docker 环境，再生成 Docker-aligned 任务，再合成带验证代码的轨迹」的三阶段流水线。最终构建 32,325 个独立 Docker images（8 种编程语言），生成 50,733 条经可执行验证器验证的 terminal 轨迹，覆盖 8 个任务 domain。基于此训练的 TerminalTraj-32B（Qwen2.5-Coder backbone）在 TerminalBench 1.0 达 35.30%、TerminalBench 2.0 达 22.00%，为 sub-100B 模型 SOTA，并显著提升 test-time scaling 能力。
01

两大核心挑战

与 SWE（代码修复）任务不同，Terminal 任务的训练数据生成面临两个独特的结构性困难：

Challenge 01
Executability
Terminal 任务（编译 Go 项目、安装 C++ 库、配置 PostgreSQL……）涵盖 8 种编程语言，且每种语言又有多个版本。不同任务的构建工具链和依赖版本往往冲突，无法共享同一个执行环境。

解法：为每个任务实例单独构建一个 Docker image，确保任务在其专属、确定性的环境中可执行。最终构建了 32,325 个 Docker images。
Challenge 02
Verifiability
SWE 任务的验证相对统一——运行 test suite，看 pass/fail。但 Terminal 任务的输出极度异构：
· 构建产物（binary 是否存在、版本是否正确）
· 文件系统变更（文件内容、目录结构）
· 进程状态（服务是否启动）
· 日志内容（特定字符串出现）

没有统一 checker 能覆盖所有情形。解法：为每个任务合成专属的可执行验证代码，而不依赖统一规则。
⚡
这两个挑战的本质是 SWE 方法论无法直接迁移到 Terminal 领域的原因。TerminalTraj 的设计选择——32K 独立容器 + per-task 验证代码——正是为了在 terminal domain 重建 SWE 里 fail2pass test 所提供的那种确定性验证保障。
02

三阶段流水线

TerminalTraj 采用「环境优先」策略——先建可执行环境，再基于环境反向设计任务，再合成轨迹。这与 daVinci-Env 和 SWE-Smith 的核心思路一致。

S1
Repository Filtering — 模型驱动质量评分
从 GitHub 海量仓库出发，不依赖启发式规则，而是让模型对每个 repo 进行质量评分。

评分维度：README 完整性、依赖可安装性、任务生成潜力、代码活跃度、构建文档丰富性。

过滤后得到覆盖 8 种编程语言 的高质量源仓库池：Python · Go · C++ · Java · Rust · TypeScript · Bash · Ruby。
↓
S2
Docker Environment Construction — 每任务独立容器
对筛选后的 repo，自动生成并构建 Docker image。关键工程细节：

· 语言版本矩阵管理：同一语言的不同版本需独立镜像（如 Python 3.9 vs 3.11，Go 1.19 vs 1.22）
· 构建工具链覆盖：go build / cmake+make / cargo / mvn / npm / pip 等
· 网络隔离：执行环境无外网访问，依赖在构建时固定
· 存储预算：每个容器 200GB 存储上限

最终产出：32,325 个 Docker images，公开 5,660 个含完整 Docker 环境的实例。
↓
S3
Task Instance Generation — Docker-Aligned 任务设计
基于已构建的 Docker 环境，反向设计任务描述——而非先写任务再建环境。

每个任务实例包含三要素：
① 自然语言描述（query）：清晰说明目标，如「编译 Prometheus Blackbox Exporter v0.19.0」
② Docker 环境绑定：任务与特定容器一一对应
③ 可执行验证代码（success condition）：成功判断逻辑，覆盖 8 个 domain
↓
S4
Trajectory Synthesis — Agent 多轮执行 + 验证筛选
以 Qwen2.5-Coder 作为 backbone agent，在 Docker 容器中通过多轮 Shell 命令交互完成任务。

· 轨迹长度：2~376 轮对话（平均复杂度适中）
· 命令格式：JSON 批处理 {"commands": ["cmd1", "cmd2"]}
· 验证：任务完成后运行专属验证代码，pass 则保留，fail 则丢弃
· 最终：50,733 条经验证轨迹（公开 20,000 条）
ℹ
「环境优先」的反向设计策略（先有 Docker → 再生成任务）比「先写任务再建环境」的前向策略更 scalable：任务的可行性在设计阶段就被 Docker 环境保证，避免了大量「任务描述了但环境跑不起来」的浪费。
03

per-task 可执行验证器

TerminalTraj 解决 Verifiability 挑战的核心机制：为每个任务生成专属验证代码，而非依赖通用 checker。验证代码随任务一起合成，定义了该任务的 success condition。

验证代码根据任务类型采用不同策略：

🔧
Build 类任务：检查 binary 是否存在 + 版本号是否正确。如「构建 Prometheus Blackbox Exporter」→ 验证 ./blackbox_exporter --version 输出含 0.19.0。
📁
File Ops 类任务：检查文件存在性、内容包含特定字符串、目录结构符合预期。验证器以 Python 脚本形式运行，精确比对文件状态。
📦
Package 安装类任务：验证 pkg-config --modversion libname 输出正确版本，或 python -c "import pkg; print(pkg.__version__)" 通过。
🐍
Python Scripting 类任务：运行生成的脚本，对比 stdout 是否符合预期输出格式（正则匹配或精确匹配）。
📊
Log Analysis 类任务：运行分析脚本，验证提取出的信息（如 top-5 错误类型）是否在期望范围内。
⚠
per-task 验证器的代价是构建成本高——每个实例都需要合成并测试专属验证代码。这也是为什么公开数据集只有 5,660 条（含 Docker）而非全部 50,733 条的原因之一：带完整 Docker 环境的数据分发成本更高。

轨迹数据格式 · 真实样例

数据集字段：query（任务描述）和 messages（多轮对话）。先看 System Prompt，再看各 domain 的真实样例。

system prompt（所有样例共用） 
role: "system"
content: "You are an AI assistant tasked with solving command-line tasks
  in a Linux environment. You will be given a task description and
  the output from previously executed commands. Your goal is to
  solve the task by providing batches of shell commands.

  Format your response as JSON with the following structure:
  {\"commands\": [\"cmd1\", \"cmd2\", ...]}

  Guidelines:
  - Provide commands that are executable in a Linux terminal
  - You can provide multiple commands in a single response
  - Analyze the outputs of previous commands to inform next steps
  - If a command fails, diagnose the issue and try an alternative
  - Complete the task efficiently with minimal unnecessary steps"

Log Analysis · Bash Android logcat 崩溃重启分析 复杂度 ★★★★
query（节选）
Analyze the Android system log at /app/system_logs/logcat.txt to identify
persistent applications automatically restarted after a crash.

Pattern to find:
  ActivityManager: Start proc [proc] for service [svc]: pid=[n] uid=[n]
  PRECEDED by:
  ActivityManager: Process [proc] (pid [n]) has died. scheduling restart
    of crashed service [svc] in [delay]ms

Write the process names to /results/restarted_persistent_apps.txt (one per line).
典型 assistant 轮次
# 轮 1：探索日志格式
{"commands": [
  "head -50 /app/system_logs/logcat.txt",
  "grep -c 'has died' /app/system_logs/logcat.txt"
]}

# 轮 2：提取崩溃→重启对（Python 比 grep 更适合跨行模式）
{"commands": [
  "python3 - <<'EOF'\nimport re\nwith open('/app/system_logs/logcat.txt') as f:\n    lines = f.readlines()\nrestarted = set()\nfor i, line in enumerate(lines):\n    m = re.search(r'Process (\\S+) \\(pid', line)\n    if m and 'has died' in line and 'scheduling restart' in line:\n        proc = m.group(1)\n        # 向后找 Start proc\n        for j in range(i+1, min(i+20, len(lines))):\n            if proc in lines[j] and 'Start proc' in lines[j]:\n                restarted.add(proc)\n                break\nwith open('/results/restarted_persistent_apps.txt','w') as f:\n    f.write('\\n'.join(sorted(restarted)))\nprint(f'Found {len(restarted)} processes')\nEOF"
]}
# 返回: Found 7 processes ✓

# 返回: John age 30 ✓
格式设计要点
📋
命令批处理：每轮输出 {"commands": [...]}，允许一轮执行多条命令，减少 round-trip。简单任务 2 轮完成，复杂任务可达 376 轮。
🔄
累积 stdout/stderr：user 的 content 始终包含所有历史命令输出（不只是上一轮）。agent 可回溯任意历史状态做决策，但代价是 context 会随轨迹增长膨胀。
🔁
错误感知 + 重试：从样例 2（Cap'n Proto）、3（PostgreSQL）、6（nlohmann）可以看到，agent 在命令失败后会诊断错误并换策略重试，而不是重复同一命令。这正是训练数据希望教给模型的核心 behavior。
📐
任务难度差异大：从「写一个文件（2 轮）」到「跨行正则分析 logcat（多轮）」，数据集覆盖了从 trivial 到 complex 的全谱系。这与 DockSmith 的 curriculum 思路一致——难度分布对训练质量有直接影响。

07

与 SWE 方法的对比

TerminalTraj 是 SWE 数据构建范式向 Terminal domain 的迁移。两者共享「环境优先 + 可执行验证」的核心理念，但在具体实现上有重要差异：

维度 TerminalTraj daVinci-Env (SWE)
任务类型 Terminal / CLI 命令执行 代码 bug 修复（patch）
输出异构性 极高（binary/文件/日志/进程） 统一（test pass/fail）
验证机制 per-task 可执行验证代码 fail2pass test patch
环境构建驱动 模型质量评分 → Docker build 4-agent 协作（含 Test Analysis）
难度过滤 验证通过率隐式过滤 显式 1-2/4 pass rate 过滤
Docker 镜像数 32,325 个 45,320 个
轨迹数 50,733 条 ~13,000 条（高过滤）
公开范围 20,000 条轨迹 + 5,660 含 Docker ~13,000 轨迹 + 45,320 环境
语言覆盖 8 种语言（多语言设计） 主要 Python
ℹ
daVinci-Env 在质量把控上更严格（显式难度过滤保留 ~9K/45K 环境），轨迹质量更高但数量更少。TerminalTraj 保留了更多轨迹（50K），质量控制依赖可执行验证器的通过率，属于量大质保策略。两者思路各有侧重，对我们的 D^env 构建都有参考价值。
08

对我们的启发

Phase 2 VLM Agent：Terminal 任务 D^env 的直接参考
Phase 2 中如果要构建 VLM agent 的 D^env，Terminal/CLI 任务是比 SWE 更接近 VLM 真实使用场景（截图操作、命令行界面）的 domain。TerminalTraj 的「环境优先 + per-task 验证器」框架可以直接借鉴：先建 Docker 环境 → 基于环境设计 VLM 交互任务 → 合成带验证的轨迹。
多语言覆盖 = 更强泛化（ScaleEnv 的结论在 Terminal 上得到验证）
TerminalTraj 的 8 语言设计，结合 DockSmith 的 transfer 实验，共同验证了 domain 多样性提升 agent 泛化能力。对我们的 D^env 数据：如果只覆盖 Python，可能导致 agent 在 Go/C++ 等场景的表现显著弱于预期。应在 Phase 1 D^env 设计阶段就规划多语言覆盖。
命令批处理格式值得参考
{"commands": ["cmd1", "cmd2"]} 的 JSON 批处理格式有两个优势：① 减少对话轮次，降低序列长度；② 形成结构化的决策单元，便于后续做 step-level 的 reward shaping。我们的 D^env 轨迹格式设计时，批处理命令比单步命令更紧凑，值得在 Action space 设计中考虑。
公开数据可直接用于 D^env 的 warm-up 训练
TerminalTraj 公开了 20,000 条经验证的轨迹（HuggingFace: m-a-p/TerminalTraj），其中 5,660 条带完整 Docker 环境。这部分数据可直接作为 Phase 1 D^env 的初始数据，快速验证 Terminal 任务的 mid-training 效果，再基于此结果决定是否自行构建更大规模的 D^env。
「环境范式可移植性」是关键结论
TerminalTraj（Terminal）和 daVinci-Env（SWE）证明了同一套「环境优先 + 可执行验证」范式在不同 domain 都成立。DockSmith 进一步证明 Docker 构建能力和代码能力共享 skill。这意味着我们的 D^env 构建不必局限于 SWE 任务——Terminal 任务、甚至 GUI 任务都可以用类似框架构建，且不同 domain 的训练数据有正向 transfer 效应。
