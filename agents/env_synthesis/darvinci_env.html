daVinci-Env

Open SWE Environment Synthesis at Scale
↗ arXiv:2603.13023 🤗 GAIR/OpenSWE ⌥ GAIR-NLP/OpenSWE GAIR-NLP · 2026.03
本文提出 OpenSWE——一个大规模、全自动的 SWE 训练环境构建系统。核心贡献：通过四 agent 协作流水线，从 572,114 个 GitHub PR 中自动构建出 45,320 个可执行 Docker 环境（覆盖 12,800 个 repo），并进一步通过 difficulty-centric filtering 筛选出 ~9,000 个高质量环境用于生成 ~13,000 条训练轨迹。在此基础上训练的 OpenSWE-32B / 72B 在 SWE-Bench Verified 上分别达到 62.4% / 66.0%，同时显著提升了数学推理（+12pt）和科学（+5pt）等 out-of-domain 能力。
01

四个专职 Agent

⚙ 所有 Agent 统一由 DeepSeek-V3.2 驱动
Agent 01
Repo Exploration Agent
从本地 repo checkout 中提取 setup/test 相关信息。

仅暴露三个受限 API：browse（结构探索）、search（定位配置文件）、digest（提取文件内容）。

cost-aware 策略：默认浅探索，仅在 Test Analysis Agent 明确请求时才深入获取额外文件，节省 token。
Agent 02
Dockerfile Agent
生成可复现的 Docker build 文件。

使用预置 base image openswe-python-{ver}（覆盖 Python 2.7 ~ 3.14，内置 conda + testbed env + 包镜像）。用本地 bare repo cache + COPY 代替运行时 git clone，彻底绕开 GitHub rate limit。

硬性禁令：FORBIDDEN to run tests in the Dockerfile。
Agent 03
Evaluation Script Agent
生成 bash 脚本验证 patch 正确性。

只跑 targeted tests，不跑全量 test suite。可合成原 PR 里没有的新测试用例——这是与 SWE-Bench 静态 fail2pass 脚本的本质区别。

禁止 set -e（会截断 error code），必须输出 OPENSWE_EXIT_CODE=$rc。
Agent 04
Test Analysis Agent
分析执行结果，判断 pass/fail 并精确路由反馈。

Pass 时：检查有无硬编码 exit code 或跳过真实测试的 shortcut。
Fail 时：诊断根因并路由到对应 agent，无解情形（依赖 404、版本不存在）直接 is_finish=true 提前退出。

输出结构化 JSON：{is_finish, dockerfile_fb, eval_fb, explore_fb}
ℹ
四个 agent 共享同一个 LLM 后端（DeepSeek-V3.2），但接受不同的 system prompt 和工具集约束，形成明确的职责边界。每个 agent 只在自己的职责范围内决策，避免单一大 agent 的幻觉叠加问题。
02

六步构建流水线

S1
GitHub PR 四层漏斗过滤
从原始 572,114 个 GitHub PR 开始，四层串行过滤：
① repo ≥ 5 stars  ·  ② 主语言为 Python  ·  ③ 有 linked issue（含描述文字） ·  ④ PR 不能只改 *tests* / *spec* / *e2e* 目录
↓
S2
Repo Exploration
Agent 接收 repo 名称、commit/version、patch 提示的文件线索，通过三个受限 API 按需探索。捕获 Python 特定框架（poetry, uv）和测试框架信息，输出精确的依赖版本（如 flask==2.0.3）和测试命令。
↓
S3
Dockerfile 构建
基于探索结果生成 Dockerfile。关键工程决策：
· 预置 base image 避免重复安装基础依赖
· Layer 顺序优化：stable 层放前面充分利用 Docker cache；dev 依赖层独立，便于廉价迭代
· 强制 pip install -e . 防止加载已安装的过期包
↓
S4
Evaluation Script 生成
生成的脚本必须遵循严格格式，用 heredoc 注入 patch，输出结构化分隔符与机器可读 exit code：
eval_script.sh · 必须包含的固定结构 
# patch via heredoc（随机分隔符，防止内容冲突）
git apply -v - <<'EOF_114329324912'
<patch content>
EOF_114329324912

# 结构化输出分隔符（regex 解析用）
echo ">>>>> Start Test Output"

# 只跑 targeted tests，不跑全量 test suite
# 可合成原 PR 里没有的新 test case
pytest tests/test_target.py -v

rc=$?                                 # 立即捕获 exit code
echo ">>>>> End Test Output"
echo "OPENSWE_EXIT_CODE=$rc"           # machine-readable signal

# ⚠ 严禁 set -e（会在 test fail 时截断 exit code 输出）
S5
双条件 Rule-Based 验证
每个环境执行两次，必须同时满足：
① 仅 test patch：exit code ≠ 0（测试应该失败，证明 bug 存在）
② 完整 fix patch：exit code = 0（测试应该通过，证明 fix 有效）

优化：Dockerfile 不变时复用已构建 image，只重跑 eval script → 5× 速度提升。
容器资源上限：4 CPU / 24GB RAM / 200GB 存储。
↓
S6
Test Analysis 诊断与路由
解析 OPENSWE_EXIT_CODE=X 标记（regex），缺失则触发 failure。
诊断失败原因并精确路由反馈，或标记无解案例直接退出（Dockerfile 问题 / Script 问题 / 不可解）。
03

迭代反馈路由环

四个 agent 形成一个带精确路由的迭代修复环，而不是简单的串行流水线。失败时反馈被定向发送给负责该问题的 agent，避免无效重试。

Repo Exploration
→
Dockerfile
→
Eval Script
→
Execute
→
Test Analysis
Dockerfile 错误 ────────────────────→ Dockerfile Agent 重生成
Script 错误 ────────────────────→ Eval Script Agent 重生成
需要更多文件 ────────────────────→ Repo Exploration Agent 深入
无解情形 ────────────────────→ is_finish=true，提前退出
⚠
无解情形的识别：依赖包 404、Python 版本不存在（如 numpy ≤ 1.8）、conflicting dependencies 等情形，Test Analysis Agent 会标记 is_finish=true 直接终止，不浪费资源继续重试。网络问题则设置 is_finish=false 重试。
04

Difficulty-Centric Filtering

45,320 个通过验证的环境并不都适合训练——太简单的和太难的都会降低学习效率。论文设计了一套基于 pass rate 的难度校准机制：

⚙
用 GLM-4.7 在 OpenHands 或 SWE-Agent scaffold 上对每个环境采样 4 条轨迹（temperature = 1.0，200k context window，max 300 steps），统计成功条数作为难度代理指标。

任何 agent 都无法解决
1/4
保留 ✓
适合难度
有挑战，但可解
2/4
保留 ✓
适合难度
有挑战，但可解
3–4/4
丢弃
过于简单
对训练无增益
✓
这一机制解决了两类核心质量风险：PR-Issue 不对齐（patch 没有真正解决描述的 issue）和Triviality（issue 描述直接泄露了解决方案）。通过 agent 的实际表现来过滤，比规则过滤更鲁棒。

过滤结果：45,320 → ~9,000 高质量环境，用于生成 ~13,000 条 curated trajectories。yield rate ≈ 20%。

05

多层质量保障

①
Rule-Based 双条件验证：fail2pass 通过 exit code 确认，而非 log 字符串匹配（更鲁棒）。OPENSWE_EXIT_CODE=X 标记缺失即视为失败。
②
Shortcut 检测：Test Analysis Agent 在 pass 后主动检查是否存在硬编码 exit code 或绕过真实测试执行的逻辑。
③
Action Masking：训练轨迹中含 formatting error 或操作错误的 step 做 loss masking，不参与梯度计算，仅保留正确动作的训练信号。
④
Reward Hacking 过滤：轨迹中含 git pull 的 bash action 直接移除——这是典型的作弊行为（通过拉取远程已有 fix 绕过真实修复）。
⑤
Benchmark 去污染：SWE-rebench 和 SWE-Bench Verified 中的实例全部从训练集移除，防止测试集泄露。
06

基础设施

组件规格 / 方案
计算集群 64 节点 ECS，每节点 32 vCPU / 128GB RAM / 4TB SSD / 20Gbps 网络
构建耗时 ~2 周完成 45,320 个环境构建
任务调度 File-based 消息队列（共享文件系统），数据并行，节点间无强耦合（非 MPI/Ray）
进程恢复 systemd 服务自动重启，防止单节点故障影响全局进度
容器管理 自动 Docker 资源清理守护进程，防止"僵尸容器"积累占用磁盘
监控 Prometheus + Grafana 实时监控构建进度、成功率、资源使用
Image 复用 Dockerfile 不变时直接复用已构建 image，仅重跑 eval script → 5× 提速
网络隔离 本地 bare repo cache + COPY 代替 git clone，消除 GitHub rate limit 依赖
ℹ
构建系统选择 file-based 消息队列而非 MPI/Ray，有意为之——牺牲一些调度效率换来极高的容错性：任何节点崩溃都不影响其他节点，任务状态持久化在文件系统中，重启即可续跑。

08

对我们方案的启发

直接复用 OpenSWE 环境，跳过 $891K 构建成本
45,320 个 Docker 环境已开源（🤗 GAIR/OpenSWE）。我们不需要重新构建——直接在这批环境上运行 rollout，用自己的 agent 生成轨迹。相当于 免费得到了 Phase 1 D^env 的执行基础设施。
1-2/4 Pass Rate 过滤标准可直接移植
难度校准的核心操作数：用小模型（GLM-4.7 级别）采样 4 次，保留 1-2 次成功的实例。不需要大规模计算，在我们的 D^env 构建里可以用 Qwen3-VL 自身做 difficulty estimation，成本可控。
Dockerfile 构建经验直接可借鉴
预置 base image、本地 repo cache、pip install -e . 这套工程实践是他们踩坑总结的。我们的 D^env 如果需要自建环境，直接 follow 这套范式，避免重复踩坑。特别是 5× 加速的 image 复用策略，对 eval script 迭代优化很有价值。
git pull reward hacking 过滤要提前设计
论文明确指出轨迹中 git pull 的 bash action 需要过滤——这是 agent 通过拉远程仓库已有 fix 作弊的行为。在我们的 D^env rollout pipeline 里，这类 reward hacking 检测需要从设计阶段就纳入，而不是事后补救。
Evaluation Script 可合成新测试用例是关键差异化
与 SWE-Bench 静态 fail2pass 不同，daVinci-Env 的 Eval Script Agent 可以合成原 PR 里没有的新测试用例。这意味着即使原始 PR 的测试覆盖不足，也能构建有效验证环境。如果我们需要从 D^ctx 的 PR 数据中提取验证信号，这个思路值得参考。
数据量 vs 难度筛选的 trade-off
从 45,320 → 9,000 只保留了 20%，但最终模型效果（66.0%）超越了使用更多但未筛选数据的方案。这验证了 "质量 > 数量" 的核心判断——我们在 D^env 阶段宁可少，要精，不要盲目追求 token 量。
