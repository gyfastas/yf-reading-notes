1. 一句话总结

通过合成可控实验，论文解耦了 pre-training / mid-training / RL 三阶段各自对推理泛化的因果贡献：RL 只在模型"能力边缘"有效，mid-training 是被严重低估的关键桥梁，process reward 能缓解 reward hacking。

2. 核心 Insights

Takeaway 1 RL 只在模型的「能力边缘」产生真正的能力增益

背景：论文用合成数学题来做��验。每道题本质是一个依赖图（DAG），图中边的数量 op 代表需要多少步算术运算才能得出答案。op=2 是两步简单题，op=20 是二十步复杂链式推理。Pre-training 阶段只让模型学了 op=2-10（最多 10 步）的题。

为什么用 pass@128：pass@1 衡量的是"一次做对"的概率（采样效率），而 pass@128 衡量的是"给 128 次机会，至少做对一次"的概率——本质上测试这个能力在不在模型的能力空间里。如果 pass@128 从 0 变成非零，说明模型学会了之前根本做不到的事。

RL 真正扩展能力的两个条件：

  1. 条件 1：任务不能被 pre-training 充分覆盖（不能太简单）
    模型在 pre-training 中学了 op=2-10 的题，这些题已经被学透了（pass@128 ≈ 99.5%）。对这些题做 RL，只能把 pass@1 从 87% 提到 95%（更准了），但 pass@128 纹丝不动——因为模型本来就会，RL 只是让它更稳定地一次命中，没有产生新能力。
    → 通俗理解：对一个已经会做一位数加法的学生做强化训练，他做对的速度变快了，但你不能说他学会了新技能。
  2. 条件 2：任务不能远超模型当前水平（不能太难）
    对于 op=15-20 的题（远超 pre-training 范围），base model 的 pass@128 ≈ 0%——给 128 次机会也做不出来。在这些题上做 RL 几乎没效果，因为 RL 依赖模型偶尔能"蒙对"来获得正向 reward，如果完全做不到，就收不到任何有效梯度信号。
    → 通俗理解：让一个小学生直接做微积分题，做一万道也学不会——完全超出探索范围。

真正有效的区间："能力边缘"（edge of competence）

只有 op=11-14 的题——模型 pass@1 ≈ 0%（单次几乎做不对）但 pass@128 > 0%（128 次里偶尔能蒙对）——RL 才产生真正的能力增益：pass@128 从 ~30% 飙升到 ~72%。这个区间恰好在模型能力的"边界"上：够得着但还不稳，RL 能把偶尔的成功强化放大成稳定的能力。

能力边缘图示： 
难度    pass@1  pass@128   RL 效果              区间名
─────────────────────────────────────────────────────
op=2-10   87%    99.5%    只磨尖 pass@1，pass@128 不变    ID（太简单）
op=11-14   5%    30%      pass@128: 30→72%，真正新能力    Edge（边缘）✓
op=15-20   0%     ~0%     几乎无效果                      Hard（太难）

统一了两派观点：学术界关于"RL 到底能不能超越 base model"有争论。认为"RL 只是 sharpener"的人（Yue et al., Zhao et al.）实验对象是 ID 任务（模型本来就会的题），所以看到 pass@128 不动；认为"RL 能创造新能力"的人（Liu et al., Sun et al.）实验对象是 OOD 任务（模型原本做不好的题），所以看到了真正的泛化。两边都对，只是在看不同的难度区间。

Takeaway 2 RL 的跨 context 泛化需要 pre-training 种下「种子」

Context B（长尾模板）在 pre-training 中的暴露比例实验：

Pre-training 中 context B 比例RL 后 context B 上 pass@128结论
0%~0%完全不泛化
0.1%~0%不够
1%~60%���破临界点，强泛化
10%~60%收益饱和

关键发现：RL 无法凭空创造能力，但 pre-training 只需要极稀疏的「原子推理基元」种子（≥1%），RL 就能将其放大组合成复杂 OOD 解法。种子不需要复杂，只要基元到位。

Takeaway 3 Mid-training 是被严重低估的关键阶段

在固定 compute budget 下对比 5 种 mid-training / RL 配比（β=0 纯 mid, β=1 纯 RL, 以及 0.2/0.5/0.8 三种混合）：

目标最优配比说明
OOD-edge (op=11-14) 可靠性Full mid + Light RL (β=0.2)Mid-training 建立 prior，少量 RL 对齐即可
OOD-hard (op=15-20) 突破Light mid + Heavy RL (β=0.8)Mid 种种子，把 budget 留给 RL 探索

Mid-training + RL 组合比纯 RL 平均高 +10.8%（同等 compute 下）。Mid-training 的角色是分布桥梁：缩小 pre-training 与 RL 数据分布的 gap，让 RL 更高效。

Takeaway 4 Process reward 减少 reward hacking 并提升推理忠实性

复合奖励 R = α·R_out + (1-α)·R_pv（outcome + process verification）：

  • 纯 outcome reward → 模型走捷径（正确答案但推理链错误 = reward hacking）
  • 加入 process verification → pass@1 提升 4-5%，结构错误（dependency mismatch）减少 28%
  • 最严格版（R_out only if R_pv=1）效果最好，但更难优化
  • 推荐：0.2 × outcome + 0.8 × process 是实际最佳平衡点

3. 实践指导

Guidance 1 RL 数据设计：瞄准模型能力边缘

过滤 RL 数据集，只保留 pass@1≈0 且 pass@k>0 的任务。太简单的（模型已解决）浪费 compute，太难的（pass@k=0）贡献零梯度。可以迭代式地做 curriculum：模型变强后重新评估"边缘"在哪。

Guidance 2 Pre-training 数据：广撒基元种子，不需要复杂样本

不需要在 pre-training 中塞大量复杂示例，只要确保原子推理基元（基本规则、基本技能）在长尾 context 中有 ≥1% 的覆盖。RL 会自动把这些基元组合出复杂解法。优先追求覆盖面而非深度。

Guidance 3 Compute 分配：Mid-training 装 prior，RL 做探索

把 mid-training 定位为 "安装 prior"，RL 定位为 "放大探索"。目标越难 → RL 占比越大；目标越近 → mid-training 占比越大。别把全部 budget 给 RL —— mid-training 是最被低估的杠杆。

Guidance 4 奖励设计：混合 outcome + process 信号

稀疏 outcome reward 容易被 hack。将 dense process-level 验证混入 reward function，推荐 0.2 outcome + 0.8 process，在 process supervision 质量高的前提下效果最好。

4. 对 ThothVL Agentic Mid-training 的启示

启示 1 VLM Agentic 任务也有「能力边缘」

对于 VLM agent（如 web browsing, GUI 操作），RL 数据不应均匀分布在所有难度。应先评估 base model 在各环境上的 pass@k，然后只在"边缘区"采样 RL trajectory。

启示 2 Mid-training 对 Agentic 能力尤其关键

论文在合成数学上验证了 mid-training 的桥梁作用。类比到 agentic 场景：base VLM 可能有视觉理解但无 tool-use/multi-turn 能力。Mid-training 用 SFT 注入 agent 基元（ReAct 格式、CodeAct 格式、tool call schema），然后 RL 在真实 env 中 fine-tune 组合策略。不做 mid-training 直接 RL 大概率 reward 信号太稀疏。

启示 3 1% 基元覆盖 → 自己构造长尾 context

论文发现 1% 的长尾 context 暴露就足够。对 VLM：只需在 mid-training 中混入少量（~1%）目标 env 的 interaction trace（比如 WebShop 的 search→click 序列），RL 就有可能泛化到未见过的网站。不需要每个 env 都大量标注。

启示 4 Process reward 对 agentic task 更自然

Agentic 任务天然有 step-level signal（每步 env.step() 返回 reward/info），比数学推理的 process verification 更容易获取。GiGPO 的 step-level 信用分配 + 论文的 process reward 思路高度互补。

5. 实验框架设计

论文最大的方法论贡献是建了一个完全可控的合成实验框架，解决了现实 LLM 训练中 "pre-training 数据不透明、无法归因" 的根本问题。

5.1 合成数据生成（基于 GSM-Infinite）

核心思路：用依赖图（DAG）生成推理问题，精确控制每一个维度。

dependency graph G = (V, E)
  V = ���量节点（每个有数值）
  E = 依赖边（算术操作: +, -, ×, mod, ...）
  v* = 答案节点

复杂度 = op(G) = |E| = 边数 = 推理步数

contextual template τ（如 "animals-zoo", "teachers-school"）
  → 将抽象 DAG 渲染成自然语言数学题

训练/测试数据完全不重叠:
  Pre-training: op=2-10, context A
  Mid-training: op=11-14, context A
  RL:           op=7-20, context A+B (多种配比)
  Test:         op=2-20, context A+B

5.2 两个泛化维度

维度含义怎么测
Extrapolative (Depth)能否解比训练更深的推理链？Pre-train 在 op≤10，测 op=11-20
Contextual (Breadth)能否迁移到没见过的表面形式？Pre-train 在 context A，测 context B（相同 DAG 结构但不同模板）

5.3 Process-Verified 评估

不只看最终答案，还解析模型输出为预测依赖图 Ĝ，与 gold 图 G 对比每个节点的依赖关系和数值。Process accuracy = 所有节点准确率的平均。只有 ProcessAcc(Ĝ, G) = 1 且 â = a* 才算完全正确。

6. 实验 1: RL 何时真正超越 Base Model?（Section 3）

Setup

  • Pre-training: 100M 参数 Qwen2.5-style 模型，10B token，op=2-10（ID 任务饱和）
  • RL: GRPO，200K 样本，4 种 RL 数据配置：op=7-10 (ID), op=9-12 (混合), op=11-14 (edge), op=17-20 (hard)
  • 指标: pass@k (k=1,2,4,...,128)，在 ID/OOD-edge/OOD-hard 三个区间分别评估

关键结果

评估区间RL 效果分析
ID (op=2-10)pass@1 ↑，pass@128 不变RL 只做了 sharpening（采样效率提升），没有新能力
OOD-edge (op=11-14)pass@128 显著 ↑ (+42%)真正的能力增益，RL 数据在 edge (op=11-14) 时效果最大
OOD-hard (op=15-20)pass@128 小幅 ↑RL 有少量溢出泛化，但增幅远小于 edge

核心洞察: pass@1 提升 ≠ 新能力。只有 pass@128 提升才是真正的能力边界扩展。RL 数据必须对准模型的 "edge of competence"。

7. 实验 2: Pre-training 暴露如何影响 RL 泛化?（Section 4）

Setup

  • 固定 RL 配方（50% context A + 50% context B, op=2-20）
  • 变量：pre-training 中 context B 原子基元（op=2）的混入比例：0%, 0.1%, 1%, 10%

关键结果

在 context B 上的 pass@128 表现：

Pre-training B 比例    RL 后 Context B pass@128
       0%                    ≈ 0%          ← 完全不泛化
     0.1%                    ≈ 0%          ← 仍然不泛化
       1%                   ~60%           ← 临界突破！
      10%                   ~60%           ← 饱和

Figure 5 拓扑相似度分析：简单任务（op=2-10）上模型复制 context A 的推理结构到 B；复杂任务（op=11-20）上模型开始创造新结构。这说明 RL 不只是记忆迁移，还能在基元基础上做组合创新。

8. 实验 3: Mid-training 与 RL 的 Compute 分配（Section 5）

Setup

固定总 compute budget T，通过参数 β ∈ [0,1] 分配给 mid-training 和 RL：

T_mid = (1-β)·T    T_RL = β·T

β=0:   Full mid-training, no RL
β=0.2: Light RL (80% mid + 20% RL)
β=0.5: Medium RL
β=0.8: Heavy RL (20% mid + 80% RL)
β=1:   Full RL, no mid-training

Mid-training: 1B token SFT on op=11-14; RL: GRPO, batch 1024, 100 steps on op=11-14

RL 的 compute 用 T_RL ≈ (5/3)·N·r·L 折算（N=样本数, r=6 rollout, L=2048 token）

关键结果

任务区间最优配比对比纯 RL 改善
OOD-edge (op=11-14) pass@1Full mid + Light RL (β=0.2)—
OOD-edge (op=11-14) pass@128Full mid + Light RL (β=0.2)—
OOD-hard (op=15-20) pass@1Heavy RL (β=0.8)—
OOD-hard (op=15-20) pass@128Heavy RL (β=0.8)—
全区间 平均Mid+RL 混合+10.8%

核心结论：任何配比的 mid+RL 都比纯 RL 好。目标越接近 → mid 越重（安装 prior）；目标越难 → RL 越重（放大探索）。

9. 实验 4: Process Reward 减少 Reward Hacking（Section 6）

Setup

复合奖励函数：

R = α·R_out + (1-α)·R_pv

R_out = 1 if final answer correct, else 0
R_pv  = process-verified accuracy (每步推理是否正确)

四种配置:
  1.0 Outcome                   (α=1.0, 纯 outcome)
  0.2 Outcome + 0.8 Process     (α=0.2)
  0.5 Outcome + 0.5 Process     (α=0.5)
  1.0 Outcome if Process=1      (严格版: R_out only when R_pv=1)

关键结果

奖励配置pass@1 (op=2-14)pass@1 (op=15-20)dependency mismatch 减少
纯 Outcome70.7%5.6%—
0.2 Out + 0.8 Process+4.7%+4.3%-28%
0.5 Out + 0.5 Process+4.6%+3.4%-18.6%
Out if Process=1+5.2%+4.1%-27.9%

Process reward 同时提高了 accuracy 和推理忠实性（结构错误大幅减少），从根本上改变了 RL 的优化方向：从"找答案捷径"转向"学正确推理路径"。
