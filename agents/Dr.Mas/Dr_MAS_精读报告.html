1. 核心贡献速览

研究背景：多智能体LLM系统（MAS）通过角色专业化实现复杂推理和工具使用，但RL后训练存在严重的训练不稳定性问题。

核心问题

将GRPO（Group Relative Policy Optimization）直接扩展到多智能体场景时，会出现梯度范数爆炸（gradient-norm explosion），导致训练崩溃。

Dr. MAS的核心解法

Agent-wise Advantage Normalization：每个智能体使用自身的奖励统计量（均值µk、标准差σk）进行优势归一化，而非全局统计量。

实验效果

任务 指标 提升幅度
Math avg@16 / pass@16 +5.6% / +4.6%
Search avg@16 / pass@16 +15.2% / +13.1%
额外收益：基本消除梯度尖峰

2. Agentic RL与End-to-End Multi-Agent System知识梳理

2.1 Agentic RL基础概念

Agentic RL指的是让LLM作为智能体（agent）在环境中自主决策、执行动作、接收反馈并通过强化学习优化策略的范式。

特性 传统RLHF Agentic RL
奖励来源 人类偏好/RM模型 可验证信号（答案正确性）
动作空间 文本生成 工具调用、多轮交互、代码执行
轨迹长度 单轮/短序列 多轮、长程决策
典型方法 PPO、DPO GRPO、RLVR
RLVR (Reinforcement Learning from Verifiable Rewards): 利用可自动验证的信号（如数学题答案、代码执行结果）进行RL训练，无需人工标注偏好。

2.3 Multi-Agent RL训练的关键挑战

  1. 异构数据分布：不同智能体被调用频率不同，样本量|Yk|差异大
  2. 角色分化：Solver/Verifier/Answer等不同角色的奖励分布差异显著
  3. 信用分配：多轮交互中如何归因最终奖励到各智能体的动作
  4. 系统复杂性：多模型协同训练、资源调度、编排管理

3. Dr. MAS算法详解

3.1 问题诊断：为什么GRPO在多智能体场景不稳定？

Aiglobal = (Ri - µ) / σ

GRPO的问题：在多智能体系统中，不同Agent的奖励分布差异很大：

  • Solver：直接生成答案，奖励方差大（对/错）
  • Verifier：判断逻辑，奖励相对稳定
  • Searcher：检索信息，奖励取决于检索质量
梯度范数爆炸的根源：使用全局baseline (µ, σ) 时，某些Agent的奖励分布远高于/低于全局均值 → 优势估计系统性偏差 → 方差不匹配 → 梯度范数爆炸

3.2 Dr. MAS的解决方案

Ai,kagent = (Ri - µk) / σk
1
Agent-wise Normalization：每个Agent只在自己被激活的steps上计算统计量
2
消除放大因子：(σ²k + (µk - µ)²) / σ² = 1

3.3 算法伪代码

# Dr. MAS 核心算法
def compute_dr_mas_advantages(trajectories, agent_ids):
    advantages = {}

    for k in agent_ids:
        # 收集智能体k的所有奖励
        Y_k = [(i, R_i) for i, traj in enumerate(trajectories)
               if agent_k_activated_in(traj, k)]

        # 计算智能体k自身的统计量
        mu_k = mean([R for _, R in Y_k])
        sigma_k = std([R for _, R in Y_k])

        # 计算agent-wise优势
        for i, R_i in Y_k:
            A_ik = (R_i - mu_k) / (sigma_k + eps)
            advantages[(i, k)] = A_ik

    return advantages

6. 核心要点总结

理论贡献

  1. 首次理论分析了GRPO在多智能体场景下的不稳定性根源
  2. 严格证明了梯度范数爆炸的条件：(σ²k + (µk - µ)²)/σ² → ∞
  3. 提出简单有效的Agent-wise Normalization，有理论保证地消除了不稳定性

适用场景

✅ 推荐使用

  • 多角色协同的MAS
  • 异构模型配置
  • 长程多轮交互
  • 观察到梯度尖峰

⚠️ 收益有限

  • 单智能体设置
  • 角色差异很小
  • 极短程交互

9. 训练方式与数据集详解

9.1 Multi-Agent是一起训练的吗？

是的！Dr. MAS是端到端（End-to-End）联合训练所有智能体。

训练流程

  1. 分布式Rollout采集：所有Agent在同一个batch中交互，生成长序列轨迹
  2. 共享Reward：整个trajectory获得一个terminal reward (0/1)，所有steps共享
  3. Agent-wise Advantage：各Agent使用自己的奖励统计量计算advantage
  4. 分组更新：共享参数的Agent一起更新，非共享的独立更新

9.4 Verifier和Searcher的奖励机制

核心设计：所有Agent共享同一个Trajectory-level Reward，没有per-agent或per-step的独立reward。 
# Math任务奖励计算
def compute_math_reward(trajectory, ground_truth):
    final_answer = extract_final_answer(trajectory)
    return 1 if verify_answer(final_answer, ground_truth) else 0

# 所有steps共享同一个R
# Solver step1: R=1
# Verifier step2: R=1  (即使拒绝错误答案)
# Solver step3: R=1
# Verifier step4: R=1  (批准正确答案)

为什么这样有效？通过Group Relative比较，即使共享R，每个Agent也能学到"相对于自己其他尝试的表现"。

12. 上下文隔离与Loss计算机制

12.1 Verifier和Searcher的上下文如何隔离？

答案：通过文本形式的memory_context传递，不需要mask！

核心设计

Composite Prompt = env_prompt + team_context + role_instruction

1. env_prompt: 任务描述 + 交互历史（memory_context）
2. team_context: 队友的输出（以文本形式）
3. role_instruction: 当前Agent的角色定义

12.2 关键洞察

❌ 错误的理解

  • 认为Verifier和Searcher共享一个大的Transformer
  • 需要在同一个forward pass中处理
  • 需要考虑attention mask隔离

✅ 正确的理解

  • 两个独立的模型（可能共享参数）
  • 每个Agent独立的forward pass
  • 上下文通过文本传递

12.3 在算Loss时怎么处理？

# 按Agent分组收集数据
agent_batches = {
    "Verifier": [{"prompt": ..., "output": ..., "R": 1}, ...],
    "Searcher": [{"prompt": ..., "output": ..., "R": 1}, ...]
}

# 每个Agent独立计算自己的PPO loss
for agent_id, data in agent_batches.items():
    mu_k = mean([d["R"] for d in data])
    for item in data:
        A = (item["R"] - mu_k) / sigma_k
        # 标准的PPO-clip loss
        loss = -min(ratio * A, clip(ratio) * A)
🔑
关键结论：每个Agent是独立的模型调用，不需要mask，因为本来就是分开的forward pass。
