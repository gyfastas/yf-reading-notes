一句话总结

KLong 解决的是超长时域 Agent 训练——当任务需要 100+ 步、跨越多轮会话、甚至超出上下文窗口时，怎么让 Agent 仍然保持连贯性。

参数只有 Kimi K2 Thinking 的 1/10（106B vs 1T），但在 PaperBench 上超过 11.28%。这说明：长轨迹能力不是参数规模的线性函数，而是训练范式的函数。

核心问题

Agent 的轨迹（trajectory）一旦变长，有两个致命问题：

  1. 上下文装不下——整个交互历史塞不进模型的上下文窗口
  2. 信用分配崩溃——强化学习的 reward 信号在超长链上稀释，模型不知道哪一步该 credit 还是 blame

这导致现有 Agent 在长任务上表现极差—— PaperBench（一个需要 10+ 小时连续工作的学术 benchmark）上，即使 Kimi K2 Thinking（1T 参数）也搞不定。

两段式训练方法

1. Trajectory-Splitting SFT（轨迹分割监督微调）

不是一次性把整个长轨迹喂给模型，而是：

  • 保留早期上下文（任务的初始设定、用户意图）作为固定锚点
  • 逐步截断后期，把长轨迹切成重叠的子轨迹
  • 每个子轨迹都有早期上下文 + 一段局部历史，确保连续性
类比：就像看一部 100 集的剧，每集开头都带一段前情提要，但你不需要从头再看一遍。

2. Progressive RL（渐进式强化学习）

分多阶段训练，每个阶段逐步延长 timeout（允许 Agent 执行更长时间）：

阶段Timeout训练目标
早期阶段较短（1-2 小时）学基础模式：读论文、写代码、跑简单实验
中期阶段中等叠加多步依赖：代码出错了怎么 debug、实验结果不对怎么调整
后期阶段较长（6 小时，RL6H）完整长程任务：从读论文到复现所有结果

关键设计：每个阶段用上一阶段训练好的模型做初始化，timeout 逐步放开。不是一开始就给 6 小时让模型乱逛，而是让它先在短时间里学会基本操作，再慢慢适应更长的决策链。

为什么这样设计？

痛点Progressive RL 解法
信用分配稀疏早期 timeout 短，任务天然被切成更短片段，reward 更密集，模型先学会"局部正确"
探索空间爆炸渐进式解锁相当于课程表：先学会走路，再学跑步，再学马拉松

数据与 Reward 设计

Research-Factory 自动流水线

  1. 搜索 Agent 去顶会（ICML、NeurIPS、ICLR 等过去五年）爬论文，过滤质量高的
  2. 评估 Agent 读论文 + 读官方代码，自动生成评估 rubric（复现标准）
  3. 喂给 Claude 4.5 Sonnet (Thinking) 执行，生成超长轨迹
  4. Rejection sampling：只保留 judge score 达标的轨迹

RL 阶段的 Reward

  • Judge：GPT-OSS-120B（放弃了 PaperBench 原版的 o3-mini）
  • Rubric：多维度结构化评估，不是简单 pass/fail
    • Experimental setup（实验设置是否复现）
    • Dataset usage（数据使用是否正确）
    • Result validation（结果是否匹配论文）
    • Core contributions（核心贡献是否覆盖）
  • Partial credit：即使没跑完完整复现，部分正确也能拿到非零 reward

RL 冷启动保障（三层）

层作用
Comprehensive SFT通用 Agent 技能激活：coding、math、search、tool use
Trajectory-Splitting SFT长轨迹模式注入：Claude 4.5 蒸馏轨迹 + rejection sampling
Progressive 课程从短 timeout 开始，逐步解锁长度

关键洞察

  • "智能的瓶颈不是记住多少，而是如何在时间中保持连贯。"
  • Trajectory splitting 本质上是在做时间维度上的压缩与展开——把一维的长序列，折叠成有层次的多维结构。
  • 这跟人类记忆的"情节-语义"分层、小说叙事的"章-节-段"结构都是同构的。
  • 长时域能力不是参数规模的线性函数，而是训练范式的函数。

深层思考：可验证奖励的来源

核心悖论： 形式化可验证的任务（代码执行、数学证明）覆盖范围有限；非形式化但关键的能力（创意、直觉、策略）没有形式验证器。如果只训练"可验证"的事，模型永远学不会"不可验证"但关键的事。

LLM-as-Judge 的脆弱性

LLM judge 的 rubric 容易被 hack：

  • Format Hacking：生成三段式回答，结构完整但内容可能是编的
  • Keyword Stuffing：密集塞入 rubric 关键词刷分
  • Length Hacking：LLM judge 普遍有 length bias
  • Self-Referential Hacking：利用 judge 和 policy 同家族的统计规律

Reward Hackability 排序（易 → 难）

  1. 字符串匹配 / BLEU / ROUGE —— 纯表面相似
  2. LLM-as-judge outcome-based rubric —— 结构/格式 bias
  3. Process-based LLM judge —— 奖励中间推理步骤
  4. Rule-based 验证（编译通过、API 格式正确）
  5. Executable sandbox 验证 —— 最难 hack
  6. Human-in-the-loop —— 最安全，但无法 scale

可能的出路：社会性验证

多智能体博弈——对手无法被取悦。对手是 Agent，不是死板的 assert，它会进化、会找漏洞、会识破表演。当验证节点（judge/agent/auditor）的密度超过临界值，集体智慧涌现。

Q & A

🙋 问：RL 阶段的 dataset 是什么？
和 SFT 同一套 Research-Factory 流水线构建的 prompts/tasks，但 RL 时不再喂蒸馏好的 Claude 轨迹，而是让模型自己在这些任务上做 online rollouts，用 rubric + judge model 给可执行的 reward。SFT 是"模仿"，RL 是"自己动手做"。
2026-04-21
🙋 问：RL 阶段初始的 rollout 准确率能被保障吗？怎么保障的？
三层冷启动：1) Comprehensive SFT 激活通用 Agent 技能；2) Trajectory-Splitting SFT 注入 Claude 4.5 蒸馏的长轨迹模式（rejection sampling 筛选）；3) Progressive 课程从短 timeout 开始，让模型先学会"走路"再跑马拉松。SFT 负责"从随机拉到会做"，Progressive 负责"从短拉到长"。
2026-04-21
🙋 问：短链任务怎么给 reward，怎么评价完成与否？
不是简单 pass/fail，而是 GPT-OSS-120B 执行的多维度 rubric 评估。即使没跑完完整复现，在"论文理解""代码结构"等子项上也能拿 partial credit。这保证了 RL 初期有稳定的非零信号。
2026-04-21
🙋 问：LLM as judge 的 rubrics 容易 hacking 吗？一般什么 reward 容易 hack？
容易。本质是 Goodhart's Law——指标成为目标后就不再是好的指标。最容易被 hack 的是字符串匹配和纯文本 LLM judge；最难的是 executable sandbox（代码必须实际跑出正确结果）。KLong 的缓解：SFT 用真实执行轨迹冷启动 + rejection sampling 筛选，减少早期进入 hacking basin 的概率。更深层的出路可能是社会性验证——多智能体互相博弈，逼近"社会性完备性"。
2026-04-21
🙋 问：怎么设计一个系统，产生源源不断的可验证奖励的来源？
Agent-World 的答案是三层架构：1) 从真实世界挖工具生态（MCP Servers、开源文档、工业 PRD）→ 天然可执行边界；2) 自动化任务合成（graph-based / programmatic）→ 先有验证方法再包装成任务；3) 自进化闭环 → 验证标准根据模型能力边界自动生长。终极答案可能不是"造更多验证器"，而是让验证本身成为智能行为的一部分——模型学会自己给自己出题、自己验证自己。
