Code Agent Scaffold 演进地图

基于 arXiv:2604.03515v1《Inside the Scaffold》源码级分析，梳理 13 个开源 coding agent 框架的演进脉络： 从 SWE-agent 的 ACI 概念，到 OpenHands 的 Event Stream，再到 Agentless 的"无 Agent"设计—— 揭示编程语言选择、控制循环策略、工具设计理念等核心 trade-off。
01

时间线演进

2023 — 持续活跃
Aider
最早的实用化代码 Agent 工具之一。核心创新：RepoMap（基于 tree-sitter + PageRank 的代码库理解）、 多种编辑格式（whole/diff/SEARCH-REPLACE）、Git-native 设计、 Architect/Editor 双模型模式。工程导向，无学术论文，但实战表现优异（26.3% SWE-bench Lite）。
无学术论文 持续活跃
2024.05 — 维护模式
SWE-agent
Princeton NLP 推出，首次系统性地提出 ACI (Agent-Computer Interface) 概念—— 为 LLM 定制一套 shell 命令（open, scroll, search_file 等），使代码导航对 agent 更友好。 核心洞察：好的接口设计比更强的模型更能提升性能。但因设计局限（命令粒度粗、缺乏结构化输出）， 已进入 maintenance mode，由 mini-swe-agent 接替。
arXiv:2405.15793 维护模式
2024.07
Agentless
UIUC 提出，挑战"复杂 agent 循环是必需"的假设。无 agent loop， 采用三阶段流水线：Localization → Repair → Validation。 性能与 agent 方案相当（27.3%），但成本仅 1/6~1/10。证明：对于 bug fix 任务， LLM 本身已具备足够能力，无需自主探索环境。
arXiv:2407.01489 · ICSE 2025
2024.07 — 持续活跃
OpenHands (原 OpenDevin)
UIUC/CMU 等多机构联合开发，目前最主流的开源 code agent 框架。 核心设计：Event Stream 架构（统一事件总线解耦通信）、 可插拔 Runtime（Docker 沙箱）、CodeAct 范式（以可执行代码作为 action 载体）。 相比 SWE-agent 的 ACI，OpenHands 提供更细粒度的 Tool/Function Calling 机制， 支持 browser automation、IPython 内核等多模态能力。
arXiv:2407.16741 活跃开发
2024 — 持续活跃
Moatless Tools
基于 有限状态机 (FSM) 的代码编辑框架。将修复任务建模为显式状态转换： Search → Identify → PlanToCode → EditCode。 使用 Claude 4 Sonnet 在 SWE-bench Verified 上达 70.8%，成本仅 $0.63/instance。 也是 moatless-tree-search（MCTS for code editing）的基础框架。
活跃开发
2025 — 持续活跃
OpenCode
受 Claude Code 启发的开源实现。核心设计：Client/Server 架构（远程 agent 执行）、 Build vs Plan 双模式（执行型 vs 规划型 agent）、LSP 原生集成。 使用 TypeScript 实现，GitHub stars 增长最快（135k），代表 CLI 工具类别的成熟形态。
活跃开发
演进主线：从 SWE-agent 的命令式 ACI（粗粒度 shell 命令） → 到 OpenHands 的函数式 CodeAct（细粒度 tool calling + 可执行代码） → 再到 Agentless 的无状态流水线（证明复杂 agent loop 非必需） → Moatless 的显式 FSM（结构化状态转换，天然产出训练轨迹）。
02

架构分类学：12维度分析框架

基于 arXiv:2604.03515v1《Inside the Scaffold》的源码级分析，13个开源 coding agent 框架在12个架构维度上呈现连续谱分布， 而非离散类别。这些维度组织为三层：控制架构层、工具与环境接口层、资源管理层。

三层架构维度概览
Layer 1: Control Architecture（控制架构层）
• Control loop type: Fixed pipeline → Sequential ReAct → Phased loop → Tree search → Full MCTS
• Loop driver: User-driven → Scaffold-driven → LLM-driven
• Control flow implementation: While loop / Recursion / Graph-as-control-flow / Exception-based

Layer 2: Tool & Environment Interface（工具与环境接口层）
• Tool set design: 0 tools (Aider) → 37 action classes (Moatless Tools)
• Edit & patch format: Unified diff → String replacement / Search-replace
• Tool discovery: Static (startup) → Config-conditional → Dynamic (runtime)
• Context retrieval: Keyword search → AST-aware → Knowledge graph traversal
• Execution isolation: Host filesystem → Docker container → Shadow git / Sandbox

Layer 3: Resource Management（资源管理层）
• State management: Destructive overwrite → Flat list → Typed event log → Graph-scoped → Event-sourced
• Context compaction: None → Rule-based truncation → Structural isolation → LLM summarization
• Multi-model routing: Single model → Role-based → Classifier chain → Per-attempt cycling
• Persistent memory: None → Config file → LLM-writable rules → Full session persistence
核心发现：11/13 的 agent 采用多原语组合（composable loop primitives），而非单一控制结构。 五种 loop 原语作为可组合构建块：ReAct、generate-test-repair、plan-execute、multi-attempt retry、tree search。 Agent 架构空间是组合式而非类别式的。
03

ACI 设计演进：从命令到函数

ACI (Agent-Computer Interface) 是 Agent 与代码环境交互的抽象层。 不同框架对 ACI 的理解差异巨大，反映了从"命令式"到"函数式"的设计演进。

SWE-agent
2024.05
命令式 ACI：自定义 shell 函数作为 action

  • open <file> [line] — 100行窗口浏览
  • scroll_up/down — 翻页
  • search_file <pattern> — 文本搜索
  • edit <line> <content> — 行级编辑
局限：命令粒度粗，无结构化输出，parse 依赖正则，agent 难以精确控制。
Aider
2023+
结构化编辑协议：多种编辑格式适配不同模型能力

  • Whole — 输出完整文件内容
  • Diff — 类 unified diff 格式
  • SEARCH/REPLACE — 精确块级替换
  • RepoMap — 代码库拓扑理解
优势：编辑成功率高，Git-native 保证可追溯性。
OpenHands
2024.07
CodeAct + Function Calling：以代码为 action 载体

  • IPythonRunCellAction — 执行 Python 代码
  • CmdRunAction — 执行 shell 命令
  • BrowseURLAction — 浏览器自动化
  • FileRead/WriteAction — 文件操作
优势：细粒度、可组合、结构化输出，支持多模态。
Moatless Tools
2024+
FSM 状态转换：显式状态机定义合法操作

  • Search — 代码定位
  • Identify — 确认修改点
  • PlanToCode — 规划编辑
  • EditCode — 执行修改
优势：状态转换天然产出结构化轨迹，适合训练数据构建。
关键洞察：为什么 SWE-agent 的 ACI 被认为"落后"？
1. 输出格式：SWE-agent 使用纯文本命令（如"open file.py"）， 需要复杂的正则 parse；OpenHands/Aider 使用结构化 Function Call / XML， 天然可序列化、可验证。

2. 粒度控制：SWE-agent 的 edit 命令只能按行编辑； Aider 的 SEARCH/REPLACE 支持块级精确替换；OpenHands 的 CodeAct 支持任意 Python 代码执行。

3. 反馈闭环：SWE-agent 的 observation 是原始 shell 输出； OpenHands 的 observation 是结构化对象（包含 exit_code、formatted_output 等）， 便于 agent 做程序化处理。

4. 可扩展性：SWE-agent 添加新命令需要修改 Docker 镜像中的 bash 函数； OpenHands 添加新 tool 只需继承 Action 类，无需修改 runtime。
04

架构设计对比

Framework 核心范式 Agent Loop 关键创新 SWE-bench Lite
Aider Pair Programming 交互式会话 RepoMap, Architect/Editor, Git-native 26.3%
SWE-agent ACI / 命令式 迭代执行 Agent-Computer Interface (ACI) 概念 12.5%
Agentless 流水线 / 无 Agent 无 Localization→Repair→Validation 三阶段 27.3%
OpenHands CodeAct / 事件驱动 Event Stream 循环 Event Stream, CodeAct, 可插拔 Runtime ~20-30%
Moatless FSM / 状态机 状态转换 有限状态机, MCTS 搜索, tree-sitter 70.8%*

* Moatless 使用 Claude 4 Sonnet 在 SWE-bench Verified 上的结果

架构选择的关键 Trade-off
Agent Loop vs 无 Agent：Agentless 证明对于确定性任务（如 bug fix）， 无循环的流水线设计更经济；但对于探索性任务（如需求理解、架构设计）， OpenHands 的灵活 agent loop 更有优势。

通用性 vs 专用性：OpenHands 追求通用平台（支持 browser、多语言、多 agent 协作）； Aider/Moatless 专注于代码编辑这一核心能力的极致优化。

可解释性 vs 灵活性：Moatless 的 FSM 提供完全可解释的轨迹，天然适合训练数据； OpenHands 的自由度更高，但轨迹噪声也更大。

编程语言分布洞察
TypeScript (5个)：OpenCode, Gemini CLI, Cline 以及部分 Codex CLI。集中在CLI 工具类别， 注重 IDE 集成、交互式开发者体验。Node.js 生态适合构建富客户端，但 analysis framework 对其支持较弱。

Python (8个)：SWE-agent, OpenHands, Agentless, Moatless Tools, AutoCodeRover, DARS-Agent, Prometheus, mini-swe-agent。 集中在SWE-bench 评测类别，科学计算/ML 生态成熟，适合 benchmark 驱动的研究。 Python 框架的 AST 分析、tree-sitter 集成更深度。

Rust (1个)：Codex CLI 使用 Rust 实现核心，注重性能与安全 sandboxing。
设计思路分类
1. 用户驱动 (User-driven)：Aider 代表，外循环由用户发起，inner generate-test-repair 循环自主。 适合 pair programming 场景，用户作为安全边界。

2. Scaffold 驱动 (Scaffold-driven)：Agentless, AutoCodeRover 代表，脚手架控制时序，LLM 在固定点被调用。 阶段分离清晰，适合确定性任务。

3. LLM 驱动 (LLM-driven)：SWE-agent, OpenHands, Codex CLI 等代表，LLM 选择工具并控制探索。 灵活性高，适合复杂 open-ended 任务。

4. 混合驱动 (Hybrid)：Prometheus 代表，LangGraph 状态机中 LLM 驱动工具选择，但节点间转移由脚手架控制。
控制循环策略对比
Fixed Pipeline (Agentless)：10阶段独立脚本，JSONL 连接，无反馈循环。 最简单、最低成本，但无法适应复杂场景。

Sequential ReAct (SWE-agent, OpenHands, Cline)：标准 Thought-Tool-Observation 循环。 主流选择，平衡简单性与灵活性。

Phased Loop (AutoCodeRover, Prometheus)：不同阶段有不同工具访问权限。 AutoCodeRover 的 search-then-patch 分离；Prometheus 的 LangGraph 显式边。

Tree Search (DARS-Agent)：ReAct 步骤形成搜索树节点，分支点 LLM 选择。 在线决策 + 离线评估分离。

Full MCTS (Moatless Tools)：完整蒙特卡洛树搜索，Select-Expand-Simulate-Backpropagate。 数值奖励引导，visit counts 平衡探索与利用。
06

设计洞察与启示

对 Midtrain 数据合成的启示
1. Scaffold 选择策略：
• Phase 1 (大规模 rollouts)：推荐 Moatless Tools 或 Agentless——结构化轨迹、低成本、高成功率
• Phase 2 (VLM 多模态)：推荐 OpenHands——天然支持 screenshot、browser、IPython
• 迭代编辑能力：参考 Aider 的多种编辑格式设计，提高 apply 成功率

2. ACI 设计的演进方向：
• 从 SWE-agent 的文本命令 → 到 OpenHands 的函数调用
• 从行级编辑 → 到块级/AST 级编辑（参考 Aider/Moatless 的 tree-sitter 使用）
• 从无状态命令 → 到状态机显式转换（Moatless FSM）

3. 成本与质量权衡：
Agentless 提供了重要的 cost-efficient baseline——如果简单流水线就能达到 27.3%， 那么复杂的 agent loop 需要证明其带来的额外价值（如 handling 更复杂场景） 足以抵消成本增加（6-10x）。
ACI 设计原则总结
原则 1：结构化优先于文本——Function Call / XML 比自由文本命令更易 parse、验证、学习
原则 2：细粒度控制——块级/AST 级编辑比行级更精确，减少 apply 失败
原则 3：可观察性——每个 action 必须有结构化 observation，便于 agent 做程序化处理
原则 4：可组合性——Action 应该是原子化的，复杂行为通过组合而非单一大命令实现
原则 5：轨迹友好——状态转换应该是显式的，天然产出训练数据
关于 Claude Code 等商业/闭源 Agent
根据论文的 inclusion criteria，以下知名 agent 被排除在分析之外（Criterion 2: no readable source code）：

Claude Code — 以编译后的 TypeScript bundle 形式分发，无公开源码仓库。OpenCode 是受其启发的开源实现。

GitHub Copilot Workspace, Cursor, Windsurf — 商业 AI 代码编辑器，scaffold 代码不公开。

MASAI — 仓库仅包含 README 链接，无实现代码发布。

此外还有 Criterion 1 排除的通用框架：Open Interpreter, Deep Agents, MetaGPT, CrewAI（多 agent 协调框架而非单个 agent 架构）。
收敛与发散维度（来自论文的关键洞察）
收敛维度（已标准化）：
• Tool capability categories：所有 LLM-driven agent 都覆盖 read/search/edit/execute 四类操作
• Edit format：向 string replacement / search-replace 收敛，优于 line-number/diff
• Execution isolation：Benchmark agent 普遍采用 Docker container

发散维度（开放设计问题）：
• Context compaction：7种不同策略，无主导方案
• State management：从 destructive overwrite 到 event-sourced，spectrum 广泛
• Multi-model routing：从 single model 到 7-layer classifier chain，设计空间未收敛

这些发散维度代表了 agent 架构的活跃设计前沿，也是未来研究的投资方向。
