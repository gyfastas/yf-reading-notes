OpenCode

100% open source AI coding agent with TUI
⌥ anomalyco/opencode ⭐ 142k stars v1.4.3 · 2025.04
OpenCode 是开源社区对 Claude Code 的回应——"100% open source" 的 AI 编程代理。 由 neovim 用户和 terminal.shop 团队打造，采用客户端/服务器架构，支持 TUI 和桌面应用双前端。 核心特点是开箱即用的 LSP 支持和多模型兼容（Claude/OpenAI/Google/本地模型）， 内置 build（开发）和 plan（分析）两种可切换 agent 模式。
01

架构设计：客户端/服务器

OpenCode 采用 客户端/服务器架构，这是与其他代码 agent 框架最显著的区别。TUI 只是众多可能的前端之一，服务器可以独立运行并支持远程驱动。

OpenCode 分层架构
TUI Frontend or Desktop App or Custom Client
↕ JSON-RPC / WebSocket
OpenCode Server (核心逻辑，状态管理)
↕
Agent Core (build / plan 模式)
↕
@general Sub-Agent (复杂任务委派)
↕
LSP Integration (开箱即用)
◉ 客户端/服务器分离
与 Claude Code 的单体架构不同，OpenCode 的 TUI 只是前端，核心逻辑在服务器。 这意味着可以远程运行 agent，或构建自定义前端（如 Web IDE 集成）。
⌘ TUI 优先设计
由 neovim 用户打造，终端体验经过精心打磨。支持 vim 风格的快捷键、 分屏、快速导航等。同时也提供桌面应用（Electron）供偏好 GUI 的用户使用。
◆ 开箱即用的 LSP
内置 Language Server Protocol 支持，无需额外配置即可获得代码补全、 跳转到定义、重命名等 IDE 功能。这是与 SWE-agent/OpenHands 等框架的关键差异。
◎ 模型无关设计
不绑定任何模型提供商，支持 Claude、OpenAI、Google 以及本地模型（通过 Ollama 等）。 用户可以自由切换模型，甚至为不同任务配置不同模型。
02

Agent 模式：build vs plan

OpenCode 内置两种可切换的 agent 模式，通过 /mode 命令切换，满足不同场景需求：

🔨 build 模式（默认）
完整功能的开发 agent，可以执行所有操作：编辑文件、运行命令、使用工具等。 适合实际编码任务，具有完全的环境访问权限。

相当于 OpenHands 的 CodeActAgent 或 SWE-agent 的默认模式。
📋 plan 模式
只读分析 agent，用于代码探索、架构理解和任务规划。 只能查看文件，不能修改。适合在动手编码前理解代码库结构。

这是一个独特设计——其他框架（如 SWE-agent、OpenHands）没有内置的"只读模式"， 需要通过配置工具权限来模拟。
ℹ
模式切换的使用场景：用户可以先在 plan 模式下探索代码库， 理解问题后再切换到 build 模式执行修改。这种显式的"先规划后执行"的切换， 与 OpenHands 的隐式 delegation 或 SWE-agent 的单一直通模式形成对比。
子 Agent 系统：@general

OpenCode 还内置了 @general 子 agent，用于处理复杂任务。当主 agent 遇到需要多步骤或跨领域知识的任务时，可以委派给 @general 执行。

使用示例 
# 在 OpenCode TUI 中
# 切换模式
/mode plan          # 切换到只读分析模式
/mode build         # 切换回开发模式

# 调用子 agent
@general analyze this codebase structure  # 委派给子 agent

# 使用 LSP 功能
/goto definition    # 跳转到定义（LSP）
/rename             # 重命名符号（LSP）

定位分析
🎯
OpenCode 的定位是"开源版 Claude Code"，而非"研究框架"。

与 OpenHands/SWE-agent 相比：OpenCode 更注重终端用户体验和 IDE 级功能（LSP）， 而非研究可复现性或 trajectory 数据采集。

与 Claude Code 相比：OpenCode 是开源替代，支持多模型， 且架构更灵活（客户端/服务器分离）。

与 Aider 相比：两者都是终端工具，但 OpenCode 内置 LSP 和子 agent 系统， Aider 更专注于 Git-native 的代码编辑。
04

设计洞察与启示

1. 客户端/服务器架构的新可能性
OpenCode 的 C/S 架构启发了新的部署模式：远程 agent 服务。 可以在服务器上运行 OpenCode server，开发者通过本地 TUI 或 Web 界面连接。 这与传统"本地运行一切"的模式不同，适合云端开发环境（如 GitHub Codespaces 或自建云平台）。
2. LSP 作为 first-class citizen
OpenCode 将 LSP 视为核心能力而非可选插件，这表明 agent 需要 IDE 级别的代码理解能力。 对于代码 agent 研究，这意味着：① 可以直接利用现有 LSP 工具获取代码语义； ② 不需要从头实现符号解析、类型推断等功能。 对比：SWE-agent 使用基于文本的搜索，OpenHands 需要额外配置 LSP。
3. 显式模式切换 vs 隐式能力控制
OpenCode 的 build/plan 模式切换是一种显式意图表达，与 OpenHands 的隐式 delegation 形成对比。 优势：用户（或上层系统）明确知道 agent 当前的能力边界； 劣势：需要手动切换，不如自动 delegation 灵活。 对研究的启示：显式模式可以作为 trajectory 数据中的"阶段标记"，帮助模型学习何时应该规划、何时应该执行。
4. 开源社区对闭源工具的回应
OpenCode 142k stars 的流行度说明开发者对"开源 Claude Code"的强需求。 这验证了 代码 agent 工具化的趋势——不仅研究需要，日常开发也需要。 对于 agentic midtrain 研究，这意味着：训练数据不仅来自研究框架（SWE-agent/OpenHands）， 也来自生产级工具（OpenCode、Aider、Cursor 等），后者的轨迹可能更具多样性。
⚠
数据可获取性：OpenCode 作为终端工具，默认不保存详细的 trajectory 数据（如每步的 thought/action）。 如果希望用 OpenCode 采集训练数据，需要修改代码或 hook 其内部事件系统。 相比之下，OpenHands 和 SWE-agent 原生支持 trajectory 导出。
