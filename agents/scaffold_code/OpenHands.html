OpenHands

An Open Platform for AI Software Agents
↗ arXiv:2407.16741 ⌥ All-Hands-AI/OpenHands 📖 docs.all-hands.dev Wang et al. · 2024.07
OpenHands（原名 OpenDevin）是目前最主流的开源 code agent 框架，由 UIUC、CMU 等多机构联合开发。它提供了一套完整的 AI 软件开发 agent 构建、评测、部署平台，核心设计围绕三个维度：(1) Event Stream 架构——actions 与 observations 通过统一事件流解耦通信；(2) 可插拔 Runtime 沙箱——Docker 容器隔离执行环境，支持代码运行、文件操作、浏览器自动化；(3) CodeAct 范式——以可执行代码（而非文本指令）作为 action 载体，显著提升 agent 的工具使用能力。在 daVinci-Env 研究中，OpenHands 作为 scaffold 用于 difficulty-centric filtering（GLM-4.7 在 OpenHands 上采样轨迹），同时也是 Phase 2 VLM 评测的候选平台，因为它天然支持 multimodal 输入。
01

系统架构总览

OpenHands 采���分层解耦架构，将 LLM 推理、任务控制、环境执行三个关注点分离，通过 Event Stream 串联。

↕ prompt / completion

↕ Action / Observation

↕ execute

Core Component
AgentController
系统主循环。驱动 agent → action → runtime → observation 的迭代执行。管理最大步数限制、任务状态机（RUNNING / FINISHED / ERROR / PAUSED）、中断与恢复。每步更新 event stream。
Core Component
Event Stream
所有 Action 和 Observation 的统一总线。Agent 向 stream 写入 action，runtime 将执行结果作为 observation 写回。支持 event 持久化（重放、调试、训练数据提取）。历史通过 stream replay 重建上下文。
Core Component
Runtime Sandbox
Docker 容器提供隔离执行环境，内置 IPython kernel（Python 执行）、Bash shell（系统命令）、Playwright 浏览器（网页自动化）。支持文件系统读写、网络访问（可配置隔离）。
Core Component
LLM Abstraction
通过 litellm 统一接入 100+ 种 LLM 后端，包括 Claude、GPT、Gemini、Llama、Qwen 等。支持 vision 输入（截图 → base64 → multimodal prompt），是 VLM 评测的关键特性。
02

Event Stream 深度解析

Event Stream 是 OpenHands 最核心的设计决策——所有通信（Action + Observation）都以不可变事件的形式流经同一条总线，agent、controller、runtime 三方完全解耦，仅通过 stream 交互。

2.1  Event 基础数据结构  ·  openhands/events/event.py
openhands/events/event.py  ·  核心字段 
class Event:
    # 由 EventStream 在 add_event() 时分配，全局自增
    _id: int

    # datetime.now().isoformat()，add_event() 时打入
    _timestamp: str

    # AGENT | USER | ENVIRONMENT —— 谁发出这个事件
    _source: EventSource

    # ★ 最关键字段：指向触发此 Observation 的 Action.id
    #   Action 本身 cause=None，Observation.cause = 对应 Action.id
    _cause: int | None

    # LLM 费用追踪（token 数、cost）
    _llm_metrics: Metrics | None

    # 关联的 tool call 元信息（tool_call_id、function name 等）
    _tool_call_metadata: ToolCallMetadata | None

所有 Action 和 Observation 都继承自 Event。cause 字段是整个架构的灵魂——它把每对 (Action, Observation) 显式绑定成因果链，无需额外对齐。

2.2  EventStream.add_event()  ·  三步写入流程
openhands/events/stream.py  ·  add_event()（简化） 
def add_event(self, event: Event, source: EventSource) -> None:
    # ① 分配全局递增 ID + 打时间戳 + 记录 source（线程安全锁内）
    with self._lock:
        event._id        = self.cur_id
        self.cur_id     += 1
        event._timestamp = datetime.now().isoformat()
        event._source    = source

    # ② 序列化为 JSON + secret 脱敏 + 写磁盘（先于通知 subscriber！）
    data = event_to_dict(event)
    data = self._replace_secrets(data)       # token/密码替换为 ***
    self.file_store.write(filename, json.dumps(data))

    # ③ 放入内存队列，异步通知所有 subscriber（非阻塞）
    self._queue.put(event)
ℹ
写磁盘优先于通知：事件先持久化，再通知 subscriber。这意味着即使进程崩溃，已发生的轨迹也完整保存在磁盘。Session 可以从任意 event id 开始 replay，这正是训练数据采集的基础。
2.3  AgentController._step()  ·  单步推理循环
openhands/controller/agent_controller.py  ·  _step()（简化） 
async def _step(self) -> None:
    # 前置检查：state==RUNNING，且没有 pending action（等待 observation 中）
    if self.state.agent_state != AgentState.RUNNING: return
    if self._pending_action is not None:    return

    # 检查 stuck loop / 迭代次数上限 / budget 上限
    self._run_control_flags()

    # ★ 核心：调用 agent.step()，内部把 event 历史转成 LLM messages，发请求
    action = self.agent.step(self.state)

    # 安全检查（可配置的 security analyzer）
    if isinstance(action, BaseRunAction):
        await self._handle_security_analyzer(action)

    # 把 action 写入 EventStream —— 触发后续 observation 的到来
    self.event_stream.add_event(action, EventSource.AGENT)
    self._pending_action = action     # 标记为 pending，等 observation 配对后清除
openhands/controller/agent_controller.py  ·  _handle_observation()（核心逻辑） 
async def _handle_observation(self, observation: Observation) -> None:
    # observation.cause == pending_action.id → 配对成功，解除 pending 锁
    if self._pending_action and self._pending_action.id == observation.cause:
        self._pending_action = None    # ← 清除后 _step() 才能继续
        if self.state.agent_state == AgentState.USER_CONFIRMED:
            await self.set_agent_state_to(AgentState.RUNNING)
        return
    # cause 不匹配 → 可能是 delegate 子 agent 的 observation，忽略

2.5  与 Claude Code 的机制对应

Claude Code 做的是完全相同的事情，只是用 Anthropic API 的 tool_use 格式实现：

OpenHands · EventStream

→ 分配 id=5，写磁盘

CmdOutputObservation(cause=5)
→ observation 通过 cause 字段绑定 action

LLM 输出 tool_use block，携带 tool_use_id="toolu_abc"

✓

2.6  Event 之间的关系：三个层次

Session 是按 id 排序的 event 序列，但 event 之间的关系不止"先后顺序"，共有三个层次：

Layer 1 · 时序
id 单调递增
event.id 由 EventStream 全局分配，严格单调递增。时序 = id 大小。这是唯一的全序关系，可直接按 id 排序还原 session。
Layer 2 · 因果
cause 字段
observation.cause = action.id。这是 EventStream 层面的因果追踪，用于 session replay、训练数据提取、get_pairs_from_events() 工具函数。不直接用于 prompt 组装。
Layer 3 · API 配对
tool_call_id
tool_call_metadata.tool_call_id，来自 LLM API 返回的 tool_call 对象中的 id 字段。这才是 prompt 组装时的匹配键，由 LLM 分配而非 EventStream。
⚠
关键区分：cause 和 tool_call_id 是两套并行的绑定系统，追踪同一件事但服务于不同用途。
· cause：EventStream 内部语义，用于轨迹分析和训练数据对��
· tool_call_id：LLM API 协议要求，用于组装合法的 messages 列表
源码中 ConversationMemory.process_events() 只用 tool_call_id 匹配，完全不读 cause。

两阶段匹配逻辑（源码）
openhands/memory/conversation_memory.py  ·  process_events()（简化） 
# 两个暂存字典
pending_tool_call_action_messages = {}   # response_id → Message(role='assistant', tool_calls=[...])
tool_call_id_to_message         = {}   # tool_call_id  → Message(role='tool', ...)

# ── 第一阶段：遍历所有 events，分别暂存 ──
for event in events:
    if isinstance(event, Action):
        # CmdRunAction/IPythonRunCellAction 等 → 提取 LLM 原始 response，暂存
        tool_metadata = event.tool_call_metadata
        assistant_msg = tool_metadata.model_response.choices[0].message
        pending_tool_call_action_messages[tool_metadata.model_response.id] = Message(
            role='assistant',
            tool_calls=assistant_msg.tool_calls,   # 含 tool_call_id
        )
    elif isinstance(event, Observation):
        # Observation → 格式化内容，以 tool_call_id 为键暂存
        tool_call_id_to_message[event.tool_call_metadata.tool_call_id] = Message(
            role='tool',
            tool_call_id=event.tool_call_metadata.tool_call_id,
            content=event.to_agent_observation(),
        )

# ── 第二阶段：配对，按顺序输出 ──
for response_id, assistant_msg in pending_tool_call_action_messages.items():
    # 只有该 assistant message 的所有 tool_calls 都有对应 result 才输出
    if all(tc.id in tool_call_id_to_message for tc in assistant_msg.tool_calls):
        messages.append(assistant_msg)              # 先 assistant
        for tc in assistant_msg.tool_calls:
            messages.append(tool_call_id_to_message[tc.id])  # 后 tool results

设计的关键洞察：ConversationMemory 本质上是把 EventStream（时序日志）重新组���成合法的 LLM 对话格式。两个系统的关注点不同——EventStream 关心"发生了什么、谁触发了谁"，ConversationMemory 关心"怎么让 LLM 看到完整上下文"。cause 字段服务于前者，tool_call_id 服务于后者，互不干扰。
03

Action Space

OpenHands 定义了一套完整的结构化 Action 类型，覆盖代码执行、文件操作、网页交互、agent 控制等所有维度。

CmdRunAction
在 Bash shell 中执行命令。command 字段为 shell 命令字符串，timeout 可设超时。最常用的 action 类型，git/pytest/find 等均通过此执行。
IPythonRunCellAction
在 Jupyter IPython kernel 中执行 Python 代码。保持 kernel 状态（变量、import 持久），适合数据分析、逐步调试。CodeAct 的核心 action——工具调用通过 Python 函数实现。
BrowseURLAction
在 headless browser 中打开指定 URL，返回页面截图（PNG）+ DOM 结构 + 文本摘要。触发 BrowserObservation，包含可交互元素坐标。
BrowseInteractiveAction
执行浏览器交互指令序列（点击、输入、滚动、截图）。使用 Playwright/Puppeteer 驱动，支持动态页面。主要供 BrowsingAgent 使用，也支持 VLM 视觉理解。
FileReadAction
读取 sandbox 内文件内容，支持按行范围读取（start/end）。返回 FileReadObservation 含文件内容和 error 信息。
FileWriteAction
向 sandbox 内文件写入内容，支持完整覆盖或 append 模式。是代码编辑的主要路径之一（另一路径是通过 IPython/Bash 直接写）。
AgentDelegateAction
将子任务委托给另一个 agent（如将"搜索文档"子任务交给 BrowsingAgent）。DelegatorAgent 的核心机制，实现多 agent 协作分工。
AgentFinishAction
宣告任务完成，携带最终回答或总结（outputs 字段）。触发后 AgentController 将 session 状态置为 FINISHED。
MessageAction
agent 向用户或协作 agent 发送消息（非执行性）。用于请求澄清、汇报中间进���，或在 DelegatorAgent 框架内的 agent 间通信。
◆
Action 的双重语义：在 SWE 任务中，CmdRunAction 和 IPythonRunCellAction 约占 action 总量的 70%+。FileReadAction / FileWriteAction 也常见，但在 CodeAct 范式下，文件操作更倾向于通过 Python（open()）完成，保持一致性。
04

Agent 类型

OpenHands 提供多种 agent 实现，每种针对不同任务类型优化。实际部署中可单独使用，也可通过 DelegatorAgent 组合。

PRIMARY
CodeActAgent
OpenHands 的主力 agent，基于 CodeAct 论文（arXiv:2402.01030）设计。核心思路：以可执行代码（Python/Bash）作为 action 的载体，而非自然语言工具调用。

在每步推理中，LLM 输出一段 Python 或 Bash 代码，OpenHands 直接执行并返回 stdout/stderr。相比 JSON function calling，代码有更强的表达力（条件、循环、错误处理）。

工具使用方式：通过 from openhands.tools import * 导入工具包，以普通函数调用的方式使用（如 search_files("pattern")），LLM 无需学习特定的 tool call schema。

SWE 任务优化：内置 SWE-specific 工具集（文件树浏览、grep、git diff、patch apply），针对大型代码库导航做了 prompt 优化。
BROWSING
BrowsingAgent
专门针对网页浏览任务优化。使用 BrowseURLAction 和 BrowseInteractiveAction，每步接收页面截图 + DOM 结构。

支持 视觉理解：当 LLM 支持 vision 时（Claude / GPT-4V / Gemini），截图以 base64 图像形式传入 prompt，agent 直接"看图操作"。这是 VLM 评测的关键路径。

用途：WebArena、WebVoyager 等浏览器任务评测；也作为 CodeActAgent 的子 agent（通过 AgentDelegateAction 调用）完成"查阅文档/搜索信息"等子任务。
ORCHESTRATOR
DelegatorAgent
多 agent 编排层。接收高层任务后，分解为子任务并通过 AgentDelegateAction 分配给专门的子 agent（CodeActAgent 写代码、BrowsingAgent 查文档）。子 agent 完成后通过 AgentFinishAction 返回结果。

典型使用场景：复杂 SWE 任务（先 BrowsingAgent 查 API 文档，再 CodeActAgent 实现并测试）；或需要并行子任务的 multi-step 问题。
LEGACY
MonologueAgent / ReActAgent
早期实现，用于对比实验。MonologueAgent 基于 ReAct 框架（Reason → Act → Observe），以自然语言 "thinking" 作为中间步骤，性能显著弱于 CodeActAgent，现已基本不用于评测。
04b

Sub-Agent 管理机制

OpenHands 通过 AgentDelegateAction 实现多 agent 协作。最反直觉的设计：子 agent 不是并发运行的，而是父 agent 完全暂停、把控制权线性交接给子 agent，子 agent 结束后父 agent 恢复。

        budget_flag=self.state.budget_flag,           # ← 共享，子 agent 消耗父的 cost 配额
        start_id=self.event_stream.get_latest_event_id() + 1,
    )

    self.delegate = AgentController(
        agent=delegate_agent,
        event_stream=self.event_stream,  # ★ 与父 agent 共用同一个 EventStream
        initial_state=state,
        is_delegate=True,               # 不独立订阅 stream，由父 controller 转发事件
    )

    # 向共享 stream 发送任务消息（子 agent 的起点）
    if 'task' in action.inputs:
        self.event_stream.add_event(
            MessageAction(content='TASK: ' + action.inputs['task']),
            EventSource.USER,
        )

def on_event(self, event: Event) -> None:
    if self.delegate is not None:
        delegate_state = self.delegate.get_agent_state()

        if delegate_state not in (FINISHED, ERROR, REJECTED):
            # 子 agent 还在运行 → 把所有事件转发给子 controller，父 agent 不处理
            self.delegate._on_event(event)

def end_delegate(self) -> None:
    # ① 同步子 agent 消耗的 iteration 计数回父 agent
    self.state.iteration_flag.current_value = self.delegate.state.iteration_flag.current_value

    # ② 关闭子 controller
    self.delegate.close()

    # ③ 把子 agent 的结果封装为 AgentDelegateObservation，写入共享 stream
    delegate_outputs = self.delegate.state.outputs   # 来自子 AgentFinishAction
    content = f'{self.delegate.agent.name} finishes task with {delegate_outputs}'
    obs = AgentDelegateObservation(outputs=delegate_outputs, content=content)

    # ④ 把 obs 的 tool_call_metadata 指向原始 AgentDelegateAction（用于 prompt 组装配对）
    for event in reversed(self.state.history):
        if isinstance(event, AgentDelegateAction):
            obs.tool_call_metadata = event.tool_call_metadata
            break

    self.event_stream.add_event(obs, EventSource.AGENT)
    self.delegate = None   # ← 父 agent 恢复控制权

04c

Memory · 记忆机制

OpenHands 的 memory 模块位于 openhands/memory/，承担两项本质不同的职责：① 知识检索（首次用户消息时注入 workspace 上下文 + 关键词触发的领域知识）；② 上下文压缩（event 数量超出 context window 时，把历史浓缩成摘要）。二者通过不同的数据流路径工作，互不干扰。

SUBSYSTEM 1 · RETRIEVAL
Memory 类 — 知识检索
订阅 EventStream，监听 RecallAction，向流中注入 RecallObservation（repo 信息、microagent 知识）
SUBSYSTEM 2 · COMPRESSION
Condenser — 历史压缩
接受 View，判断是否超限，输出 Condensation（触发 agent 发射 CondensationAction 写回 EventStream）

4c.1 Memory 类 — 知识检索子系统

Memory（memory/memory.py）以 EventStreamSubscriber.MEMORY 身份订阅事件流，只处理 RecallAction 类型的事件。RecallAction 有两种 recall_type，触发不同的检索逻辑：

recall_type触发条件检索内容返回
WORKSPACE_CONTEXT source=USER（第一条用户消息） repo_name / repo_directory / branch；runtime web_hosts；repo 级 microagent 指令；关键词匹配的知识 microagent；conversation_instructions RecallObservation (recall_type=WORKSPACE_CONTEXT)
KNOWLEDGE source=USER 或 AGENT（后续消息） 仅关键词匹配的 KnowledgeMicroagent 内容 RecallObservation (recall_type=KNOWLEDGE) 或 NullObservation

两种 recall 均设置 obs._cause = event.id，使 RecallObservation 在 EventStream 中与触发它的 RecallAction 因果关联，之后由 ConversationMemory.process_events() 拼成 LLM prompt 的 system 消息部分。

4c.2 Microagents — 知识注入单元

Microagent 是 Memory 检索的知识载体。系统在初始化时从两个位置加载，后者可被前者覆盖（同名 key 后写优先）：

GLOBAL (公开知识)
OpenHands/skills/
随 OpenHands 代码库分发，包含常用技术栈的领域知识（git、docker、pytest 等）
USER (个人知识)
~/.openhands/microagents/
用户自定义，不随代码库分发；目录不存在时自动创建

Microagent 有两种子类型：

KnowledgeMicroagent

定义一组触发关键词（triggers）。match_trigger(query) 在 query 中检测到关键词时返回该关键词，Memory 将其 content（Markdown 知识文档）打包进 RecallObservation。适合"遇到 pytest/docker/git 时注入对应知识"。

RepoMicroagent

不依赖关键词，始终注入其 content 作为 repo_instructions。等价于 Claude Code 的 CLAUDE.md：存放在用户 workspace 里，每次启动时无条件作为 system prompt 的一部分传入 LLM。也可携带 mcp_tools 配置。

4c.3 Condenser — 历史压缩子系统

当 EventStream 中的事件数量增多，View（传给 LLM 的事件列表）可能超出 context window。Condenser 负责压缩：它接受当前 View，返回 View（直接通过）或 Condensation（触发一次压缩）。

── 每次 agent step ──────────────────────────────────────────
state.view ← View.from_events(all_events) // 过滤 forgotten + 插入摘要
result ← condenser.condense(state.view)
if isinstance(result, Condensation):
    agent returns CondensationAction // 不产生任务 action
    EventStream.add_event(CondensationAction) // forgotten_ids + summary 写入流
    下一 step → View.from_events 重跑 → 被遗忘事件从 View 中消失
else:
    正常 agent step，用 result (View) 构造 LLM 消息

View 的构建逻辑（View.from_events()）：扫描所有事件，收集每个 CondensationAction 的 forgotten id 集合；过滤掉 forgotten 事件；然后把最新的 CondensationAction.summary 作为 AgentCondensationObservation 插入到 summary_offset 位置。最终的 events 列表就是传给 LLM 的完整历史视图。

4c.4 Condenser 类型一览

Condenser触发条件保留策略特点
NoOpCondenser 从不触发 全量保留 baseline / debug 用
RecentEventsCondenser 超出 max_events 前 keep_first 条 + 最近 N 条 无摘要，直接丢弃中间事件
ObservationMaskingCondenser 每次 step 都运行 保留所有事件结构，但 attention_window 外的 Observation 内容替换为 <MASKED> 不丢事件，只遮盖内容；适合 tool-call 密集场景
AmortizedForgettingCondenser 超出 max_size 前 keep_first + 最近 max_size/2 条；直接遗忘中间 无 LLM 调用，零延迟，但无摘要
LLMSummarizingCondenser 超出 max_size（默认 100） 前 keep_first + 摘要事件 + 最近 max_size/2 条 用 LLM 生成结构化摘要（见 4c.5）；信息损失最小
CondenserPipeline 由内层 condenser 决定 多个 condenser 串行，任一返回 Condensation 即停止 可组合策略（如先 Masking 再 Summarizing）

4c.5 LLMSummarizingCondenser · 源码深析

生产环境最常用的 condenser。核心参数：max_size=100，keep_first=1，max_event_length=10_000。

▸ 触发条件（两路）

def should_condense(self, view: View) -> bool:
    return len(view) > self.max_size or view.unhandled_condensation_request

被动触发：view 事件数超出 max_size（默认 100），自然增长超限时触发。
主动触发：unhandled_condensation_request = True，由 View.from_events() 检测到：当 EventStream 中存在比最新 CondensationAction 更新的 CondensationRequestAction 时置 True。这意味着 agent 可以在还没超限时就主动插入"请求压缩"事件，强制触发摘要（例如 agent 自己意识到 token 快满）。

▸ 窗口切割数学（默认参数具体实例）

head             = view[:keep_first]                      # view[0:1] → 1条
target_size      = max_size // 2                          # 100//2 = 50
events_from_tail = target_size - len(head) - 1           # 50-1-1 = 48条
forgotten_zone   = view[keep_first : -events_from_tail]  # 中间 ~51条（排除旧摘要后送LLM）

对一个 101 事件的 view，压缩后结构：

head
1条
ev0 首条事件（通常是初始 UserMessage），永久保留，不参与摘要，不被遗忘
sum
1条
AgentCondensationObservation（旧摘要） 若 view[keep_first] 已是摘要事件则复用；否则创建占位 "No events summarized"。LLM 以此为基础增量更新
fgt
~51条
ev1 … ev52（排除旧摘要事件） 即将被遗忘的中间区段，字符串化后截断至 10,000 字符，打包进 prompt 送给 LLM
tail
48条
ev53 … ev100 最近 48 条，原样保留，完整内容传给主 LLM

压缩后 view = 1 (head) + 1 (新摘要) + 48 (tail) = 50 条。之后随 agent 继续工作 view 重新增长至 101，再次触发，稳定在 50 ↔ 100 的振荡。

▸ 增量摘要（Rolling Summary）——核心设计

每次压缩，LLM 看到的不是从零���始的原始事件，而是 旧摘要 + 新的待遗忘事件。旧摘要作为 <PREVIOUS SUMMARY> 注入 prompt，LLM 在此基础上更新，输出新摘要。整个对话历史被折叠进一个不断滚动的文本块，每轮压缩只需处理 新增 的待遗忘区间。

✓
这是增量式而非全量重新摘要。LLM 的输入规模与 events_from_tail（~51条）成正比，不随 session 总长度增长。若改为每次把全部历史重新摘要，LLM 调用成本会线性增长。

▸ CondensationAction 的 forgotten 编码

# 支持两种编码方式（二选一）
forgotten_event_ids:      list[int] | None   # 显式 ID 列表
forgotten_events_start_id: int | None        # 范围起始
forgotten_events_end_id:   int | None        # 范围终止

# forgotten property：展开为完整列表
@property
def forgotten(self) -> list[int]:
    if self.forgotten_event_ids is not None:
        return self.forgotten_event_ids
    return list(range(self.forgotten_events_start_id, self.forgotten_events_end_id + 1))

LLMSummarizingCondenser 用范围编码：只存 forgotten 事件集的 min/max id。forgotten property 展开为连续 range。隐含假设：该 ID 范围内全部是待遗忘事件（无有效 gap）。若有 gap，gap 对应的 ID 也进入 forgotten 集合，但 View.from_events() 只过滤实际存在于流中的事件，不存在的 ID 不影响结果——用更紧凑的存储换取可��受的假设。

▸ 一次完整压缩的端到端数据流

Step N：view 有 101 条事件
↓ should_condense = True → get_condensation(view)
↓ 构造 prompt（旧摘要 + ~51条待遗忘事件）
↓ LLM call → summary = "USER_CONTEXT: ... CODE_STATE: ..."
↓ return Condensation(CondensationAction(
forgotten_events_start_id=2, forgotten_events_end_id=52,
summary="...", summary_offset=1))
↓ agent 本轮不产生任务 action，直接 return CondensationAction
↓ EventStream.add_event(CondensationAction, id=102)

Step N+1：View.from_events(all 102 events)
↓ 遇到 CondensationAction(id=102) → forgotten_ids = {2,3,...,52}
↓ kept = [ev0, ev53...ev101]（49条）
↓ 在 offset=1 插入 AgentCondensationObservation(summary)
↓ final view = [ev0, SummaryObs, ev53...ev101] = 50条
↓ should_condense = False → 正常 agent step
ℹ
缓存注意：LLMSummarizingCondenser.from_config() 强制设 llm_config.caching_prompt = False。原因：每轮摘要 prompt 都含新增事件，内容不同，缓存写入后极少命中，付出 cache-write token 成本但几乎没有 cache-read 收益，得不偿失。
◆
对轨迹采集的影响：Memory 子系统的两层机制都会在 EventStream 中留下痕迹。① RecallObservation 是事件序列的一部分（cause 链接到 RecallAction），训练时需决定是否将其纳入轨迹（它等价于 system prompt 的动态部分，通常 不应作为 agent 预测目标）。② CondensationAction 同样写入 EventStream；从 EventStream 重放轨迹时，View.from_events() 会自动应用压缩语义，但若直接读取原始 event 列表则会看到已被"遗忘"的事件——两种读法给出不同的历史视图，需根据训练目标明确选择。

Runtime · Docker
容器沙箱
每个 session 启动一个 独立 Docker 容器（基于 nikolaik/python-nodejs 或自定义 base image）。提供完整的 Linux 环境：apt 包、Python 多版本、Node.js、git 等。容器内运行 openhands-runtime 服务（FastAPI），接受来自 Controller 的 action 请求。
Runtime · IPython
Jupyter Kernel
容器内常驻 IPython kernel 进程，维持 Python 状态（变量、import、当前目录）跨 action 持久存在。IPythonRunCellAction 通过 jupyter_client 协议与 kernel 通信，支持 rich output（图像、表格）。
Runtime · Browser
Playwright 浏览器
容器内运行 Chromium headless 浏览器（via Playwright），支持完整 DOM 交互、JavaScript 执行、截图（1280×720 默认分辨率）。截图以 base64 PNG 格式附在 BrowserObservation 中，传给支持 vision 的 LLM。
Runtime · Files
文件系统
容器将 工作目录挂载（volume mount）到宿主机，使 agent 修改的文件可以直接被评测脚本读取。支持 /workspace 作为默认工作目录，也支持将 git repo 复制进容器（SWE 任务标配流程）。
⚠
网络隔离注意：SWE 评测（如 SWE-Bench）要求关闭容器对外网络（防止 agent 直接 git pull 拉取答案）。默认配置网络开放，评测时需设 network_disabled=True。daVinci-Env 在 Reward Hacking 过滤中明确移除了含 git pull 的 bash action。

Runtime 与 Controller 的通信协议（简化）：

runtime / api_schema.py 
# Controller → Runtime: POST /execute_action
{
  "action": "cmd_run",
  "args": {
    "command": "pytest tests/test_foo.py -x",
    "timeout": 120
  }
}

# Runtime → Controller: CmdOutputObservation
{
  "observation": "run",
  "content": "...PASSED 3 tests in 1.24s...",
  "exit_code": 0,
  "command": "pytest tests/test_foo.py -x"
}

CodeAct 范式

CodeAct（arXiv:2402.01030）是 OpenHands 核心 agent 的理论基础，主张用可执行代码替代文本/JSON 形式的工具调用作为 action 的统一表示。

传统范式（ReAct / JSON tool call）
局限性
每个工具单独声明 JSON schema，LLM 必须学习工具的确切参数名称和格式。工具间组合需要多步交互（先查询再写入），表达能力受限于预定义工具集。错误处理几乎不可能——没有 try/except，没有条件分支。
CodeAct 范式
优势
代码天然具备完整表达力：条件判断、循环、错误捕获、变量状态都是 Python 语言的一部分。工具以 Python 函数形式提供，可以随意组合。LLM 对代码语法的训练量远大于任意 JSON schema，泛化更好。
CodeActAgent · 典型单步输出（IPythonRunCellAction） 
# LLM 输出的可执行 Python code
import subprocess, os

# 先定位相关文件
result = subprocess.run(
    ['grep', '-rn', 'def authenticate', 'src/'],
    capture_output=True, text=True
)
print(result.stdout[:2000])  # 只打印前 2000 字符避免截断

# 同时检查测试文件
for f in os.listdir('tests/'):
    if 'auth' in f:
        print(f"Found test file: {f}")
✓
实验结论（CodeAct 论文）：在 M3ToolEval 上，CodeAct agent 比 JSON tool call baseline 平均高 +20.2%；在多步任务上优势更显著（+26.5%），因为代码可以在单步内完成多个操作的组合，减少 LLM 调用轮次。

内置工具集（SWE 任务相关）：

search_files(pattern) view_file(path, start, end) edit_file(path, old, new) run_command(cmd) git_diff() git_apply(patch) run_tests(pattern) browse_url(url) think(reasoning)

作为 Scaffold 使用

"Scaffold"指为 agent 提供环境接口、工具封装、上下文管理的框架层，LLM 在 scaffold 之上运行。OpenHands 作为 scaffold 的角色在多个研究中出现：

daVinci-Env · 难度过滤
目的：估计每个 SWE 实例的难度（agent pass rate）
配置：GLM-4.7 + 200k ctx + 300 max steps
用法：在 OpenHands + SWE-Agent scaffold 上各采样 4 条轨迹，统计通过率
过滤规则：0/4 丢弃（无解），3-4/4 丢弃（太简单），1-2/4 保留
本研究 · Phase 2 VLM 评测
候选角色：ThothVL 的 agentic 能力评测平台
优势：原生支持 vision 输入（截图 → VLM），BrowsingAgent 可直接测 GUI 理解能力
评测路径：WebArena（GUI 导航）+ SWE-Bench（代码 + 视觉理解）
对比基线：SWE-Agent（只有 CodeAct，无 vision 路径）

如何将 VLM 接入 OpenHands：

OpenHands vs SWE-Agent 作为 scaffold 的选择：如果目标是 Phase 2 VLM 的 视觉 + 代码 综合能力评测，OpenHands 更合适（BrowsingAgent + vision 支持）；如果只关注 code agent SWE 性能，SWE-Agent 更专注（prompt 针对 SWE 任务深度优化），两者结果对比有助于衡量 VLM 的视觉理解对 coding 能力的贡献。
09

对本研究的启发

轨迹数据格式直接参考 Event Stream
OpenHands Event Stream 的 (action, observation) 序列是最自然的 agent 轨迹表示。D^env 的轨迹采集可以直接输出 OpenHands 格式的 event log，方便后续在 OpenHands scaffold 上评测和对比。避免自定义格式带来的转换成本。
CodeAct 范式适合 VLM 的 agentic 训练
相比 JSON function call，代码作为 action 对 VLM 更友好——VLM 的预训练数据中代码比例高，且代码的视觉表示（语法高亮、缩进）对图文混合理解��益。D^env 轨迹数据应以 Python/Bash 代码作为主要 action 载体，而非 JSON tool calls。
Vision 路径是 VLM 差异化的评测维度
OpenHands BrowsingAgent 在每步接收页面截图，这是 LLM agent 无法完成但 VLM agent 理论上更擅长的场景。Phase 2 可以通过 BrowsingAgent + GUI 任务（WebArena、Mind2Web）来量化 ThothVL 的 vision 能力对 agentic 性能的实际增益，与 text-only LLM baseline 对比。
difficulty 估���：OpenHands + 弱模型是低成本方案
daVinci-Env 的 difficulty filtering 用 GLM-4.7（非顶级模型）在 OpenHands 上采样 4 次来估计难度，这是一个成本可控的 difficulty 代理。对 D^env 数据集，可以用类似方式（如 Qwen2.5-7B + OpenHands）做初步难度筛选，避免动用昂贵的大模型。
开源 + 活跃社区 = 工程复用价值高
OpenHands 在 GitHub 上有 40k+ stars，持续更新，有活跃的 evaluation harness 维护团队。SWE-Bench、WebArena 等主流基准的 eval harness 都已内置，可以直接复用，不需要从头搭建评测基础设施。Phase 2 VLM 的评测管线建议直接基于 OpenHands 构建，节省工程成本。
