OpenDev
Rust 原生 Agent 脚手架

DoomLoop 检测 · 分级压缩 · Glob 权限系统 · Trace DAG
↗ github.com/opendev-to/opendev Rust / Cargo workspace vs OpenHands (Python)
OpenDev 是一个 Rust 实现的 Agent 脚手架，核心创新在于： (1) DoomLoopDetector——基于工具调用指纹哈希的循环检测，四级渐进式干预； (2) 分级上下文压缩——5 档阈值梯度策略（70%→99%），最后阶段 LLM 介入； (3) Glob 权限系统——Per-subagent 粒度控制 Allow/Deny/Ask，特异性评分； (4) 79 个编译时内嵌模板——消除运行时文件系统依赖； (5) Trace DAG——用图结构（非线性 EventStream）可视化轨迹。 相比 Python 版 OpenHands，设计更偏向工程鲁棒性，而非灵活性扩展。
01

系统概览

OpenDev 是一个 Cargo workspace，主 agent 逻辑在 crates/opendev-agents/，另有若干辅助 crate（opendev-context、opendev-http、opendev-tools-core）。Python 版 OpenHands 采用 asyncio + Pydantic，OpenDev 则选择 Rust async（Tokio）+ serde_json，带来更强的类型安全和零成本抽象。

Architecture · opendev-agents crate
main_agent.rs ──▶ ReactLoop ──▶ DoomLoopDetector
│
├─ apply_staged_compaction() ──▶ LLM (99% stage)
├─ SubagentManager ──▶ SimpleReactRunner / StandardReactRunner
└─ PromptComposer ──▶ 79× embedded .md templates
语言 · 运行时
Rust + Tokio
编译时检查，零成本抽象。相比 Python 的运行时错误，Rust 在类型系统层面消除了大量 agent 状态错误。
模板系统
include_str! 编译时内嵌
79 个 .md 文件在编译时通过 include_str! 打包进二进制，消除运行时 filesystem 依赖，发布产物是单个可执行文件。
事件模型
Vec<Value> 消息列表
直接操作 JSON Value 数组，没有 OpenHands 的持久化 EventStream。更轻量，但失去跨组件订阅能力。
Sub-Agent
AgentMode 枚举 + 权限系统
每个子 agent 有独立的 SubAgentSpec（14 字段），含 glob 权限规则、模型覆盖、颜色标识和隐藏模式。
02

ReAct 主循环架构

主循环在 react_loop/execution.rs（约 1600 行），是整个系统最复杂的模块。每个迭代执行以下步骤：

S1
上下文压缩检查
调用 apply_staged_compaction() 检查当前 context 使用率，按 5 档阈值决定是否执行压缩（见 §04）。99% 档会发起 LLM 调用并 await。
S2
LLM ���用 + 响应解析
通过 LlmCaller 发起请求（含重试、provider 适配、cost 追踪）。解析 tool_calls 数组和 content 文本，提取 IterationEmitter 事件。
S3
DoomLoop 检测
调用 DoomLoopDetector.check(tool_calls)，返回 DoomLoopAction：None / Redirect / Notify / ForceStop。ForceStop 会清空 recent 队列并强制触��上下文压缩（见 §03）。
S4
工具执行（可并行）
PARALLELIZABLE_TOOLS 集合内的工具（read_file、glob、grep 等只读操作）允许并行批量执行。其他工具串行。执行后记录 ArtifactIndex（跨压缩存活）。
S5
停止条件判断 + Observation 续行
若 LLM 返回无工具调用，进入 observation 续行逻辑：必须连续 2 次无工具调用才接受停止（防止过早退出）。第 1 次注入 continuation nudge，第 2 次才真正停止。
S6
最大轮次收尾 + Wind-Down Summary
达到 max_iterations 时，发送 wind-down summary prompt，要求 LLM 汇总已完成工作，给出可执行的后续步骤列表后再退出。
vs OpenHands AgentController：OpenHands 的每步通过 EventStream 分发事件，支持多组件订阅（Memory、Condenser 等独立订阅）。OpenDev 直接操作内存中的 Vec<Value>，更快但耦合度更高。OpenHands 有明确的 AgentDelegate 中断-恢复机制，OpenDev 的 sub-agent 通过 SubagentManager 独立管理，两者的并发模型不同。
03

工具调用执行链 · 写代码全链路

LLM 输出 tool_calls 数组后，OpenDev 通过三层结构处理每次工具执行：ParallelPolicy 分组 → ToolRegistry 10 步流水线 → DiagnosticProvider 回注 LSP 错误。以写代码场景（edit_file + run_command）为例，完整链路如下：

LLM tool_calls 数组
→
ParallelPolicy::partition()
→
ToolRegistry::execute() ×N
→
DiagnosticProvider (LSP)
→
ArtifactIndex::record()
→
tool results → messages

① ParallelPolicy::partition()（parallel.rs）将本轮 tool_calls 分成有序执行组，组内并行，组间串行：

parallel.rs · 分组规则示例
// 本轮 LLM 输出 4 个工具调用：
[read_file("a.rs"), read_file("b.rs"), edit_file("c.rs"), run_command("cargo test")]

// partition() 结果：
Group 1 (并行): [read_file("a.rs"), read_file("b.rs")]   // 所有只读工具
Group 2 (并行): [edit_file("c.rs")]                         // 写不同文件可并行
Group 3 (串行): [run_command("cargo test")]                  // 无 file_path → 独立 group

// 若两次 edit_file 写同一文件，则各占一个独立 group（串行）
// MCP 工具（mcp__xxx__）也走串行，每个独立 group
只读工具（并行）
read_only_tools()
read_file / list_files / search / find_symbol / find_referencing_symbols / analyze_image / fetch_url / web_search / memory_search / list_todos / list_agents 等
写工具（同文件串行/异文件并行）
write_tools()
write_file / edit_file / multi_edit / notebook_edit / insert_before_symbol / replace_symbol_body / apply_patch / memory_write 等；run_command 无 file_path → 始终串行

② ToolRegistry::execute() 10 步流水线（registry/execution.rs）：

T1
resolve_tool() · 模糊名称匹配
精确匹配 → 大小写不敏感 → camelCase→snake_case 转换（如 ReadFile→read_file）。失败则用 Levenshtein 距离（≤3）生成 "Did you mean?" 建议，最多 5 条。
T2
normalize_params() · 参数标准化
相对路径解析为绝对路径（以 working_dir 为基准），camelCase 参数 key 转 snake_case，修复常见 LLM 参数格式问题。
T3
Dedup 检查 · 同轮次去重
dedup_cache: Mutex<HashMap<String, ToolResult>>，key = "{tool_name}:{sorted_args_json_hash}"。命中则直接返回缓存，工具不再执行。每轮 LLM 调用前调 clear_dedup_cache() 重置。spawn_subagent 跳过去重（每次调用必须真实执行）。
T4
JSON Schema validate · 参数校验
对照 BaseTool::parameter_schema() 验证必填字段、类型。工具可覆盖 format_validation_error() 返回定制化错误消息，帮助 LLM 精确定位问题（默认：通用格式）。
T5
middleware.before_execute() · 前置拦截
权限中间件在此拦截。Glob 权限系统（Allow/Deny/Ask）的 Deny 在这里直接返回错误，Ask 暂停等待用户审批。任何中间件返回 Err 则整个调用失败（short-circuit）。
T6
注入 per-tool timeout config
查 tool_timeouts: RwLock<HashMap<String, ToolTimeoutConfig>>，若该工具有独立超时配置（idle_timeout_secs / max_timeout_secs），覆盖 exec_ctx 中的全局配置后再传给工具。
T7
BaseTool::execute() · 实际执行
async fn execute(args: HashMap<String, Value>, ctx: &ToolContext) -> ToolResult。工具实现读取 ctx.working_dir（路径基准）、ctx.cancel_token（UI 中断）、ctx.diagnostic_provider（LSP）、ctx.shared_state（跨迭代状态）。
T8
sanitizer · 输出截断
ToolResultSanitizer::sanitize_with_mcp_fallback() 检测输出是否超过 inline 大小限制，超限则截断并附注 "output truncated"。大型工具输出可溢写到磁盘（overflow_dir），防止撑爆 LLM context。
T9
middleware.after_execute() · 后置 hook
后置中间件错误只打日志（warn!），不改变工具结果——已完成的执行不可撤销，保证结果幂等性。
T10
dedup_cache.insert() · 缓存结果
成功执行后将结果写入 dedup cache，同轮次内相同调用直接命中缓存。duration_ms 在此附加到 ToolResult（由 registry 计算，非工具自报）。

③ edit_file / write_file 后：DiagnosticProvider LSP 回注（OpenHands 无此机制）

traits.rs · DiagnosticProvider + ToolContext 携带
// ToolContext 携带可选的 LSP 诊断 provider
pub struct ToolContext {
    pub working_dir:          PathBuf,
    pub cancel_token:         Option<CancellationToken>,  // UI 中断
    pub diagnostic_provider:  Option<Arc<dyn DiagnosticProvider>>,
    pub shared_state:         Option<Arc<Mutex<HashMap<String, Value>>>>,
    // ...
}

// edit_file 执行成功后，工具内部调用：
if let Some(provider) = &ctx.diagnostic_provider {
    let diags = provider
        .diagnostics_for_file(&file_path, max_severity=1, max_count=10)
        .await;  // severity 1 = Error only
    // 将 LSP 错误直接拼入 tool result：
    // "File modified. Diagnostics: ERROR [12:5] expected `;`"
}

LSP 诊断在 同一次工具调用的返回值 中携带，LLM 在下一轮迭代开始前就能看到编译错误，无需额外发起 run_command("cargo check")，节省 1 个完整 ReAct 轮次。FileDiagnostic 包含行号、列号、severity（1=Error / 2=Warn / 3=Info）和 message。

三层设计的协同效果：ParallelPolicy 尽可能并发（读 N 个文件只算 1 轮）→ ToolRegistry 的 dedup 缓存避免同轮重复 IO → DiagnosticProvider 让编译错误反馈零延迟。对于 long-horizon 写代码任务，这三层协同可以显著减少「写→检查→修」循环的轮次数，也是 DoomLoopDetector（§04）能更早发现问题的前提之一。
04

DoomLoopDetector （OpenHands 无此机制）

OpenDev 特有设计。doom_loop.rs 检测 agent 卡在重复工具调用的循环中，采用滑动窗口 + 指纹哈希，检测 1-步/2-步/3-步循环。

doom_loop.rs · 核心参数
// 最大循环检测长度（检查 1/2/3 步循环）
const MAX_CYCLE_LEN: usize = 3;

// 判定循环所需的重复次数（3 次即触发）
const DOOM_LOOP_THRESHOLD: usize = 3;

// 滑动窗口最大保留指纹数
const MAX_RECENT: usize = 20;

// 指纹格式："{tool_name}:{args_hash:016x}"
fn fingerprint(tool_name: &str, args_str: &str) -> String {
    let mut hasher = DefaultHasher::new();
    args_str.hash(&mut hasher);
    let h = hasher.finish();
    format!("{tool_name}:{h:016x}")
}

触发条件：检测到某长度为 L 的 tool call 序列在滑动窗口尾部重复了 3 次（即连续 3×L 个调用构成周期）。随后按 nudge 计数升级干预等级：

正常
None
无循环
正常继续
第 1 次
Redirect
注入 nudge 消息
"STOP and try something different"
第 2 次
Notify
升级为 StepBack
"stuck in a loop"
通知用户
第 3 次
ForceStop
清空 recent 队列
强制触发上下文压缩
CompactContext

检测支持并行工具调用批次——一次调用传入 tool_calls: &[Value]，多条调用按顺序追加进窗口后再执行检测，批量中 3 个相同调用也能即时触发。

为什么 OpenHands 没有这个？ OpenHands 依赖 LLM 自己识别重复，没有程序级干预。OpenDev 通过哈希指纹在代码层精确检测，成本接近零，且不消耗 token。对 mid-training 的启示：循环行为是 agent 失效的重要特征，在轨迹数据里可以用类似 fingerprint 方法自动标记"卡循环轨迹"，辅助质量过滤。
05

分级上下文压缩

代码在 react_loop/compaction.rs，通过 OptimizationLevel 枚举实现 5 档渐进策略，比 OpenHands 的"超过阈值 → 调用 LLM 压缩"两段式设计更精细。

70%
Warning——仅记录日志警告，不修改消息列表。给 agent 预留空间继续工作。
80%
Mask——调用 mask_old_observations()，将较旧的工具输出替换为紧凑引用（保留最近若干条）。无 LLM 调用，纯文本截断。
85%
Prune——在 Mask 基础上追加 prune_old_tool_outputs()，完全移除（而非截断）旧工具输出消息体。
90%
Aggressive——更激进的 Mask（OptimizationLevel::Aggressive），保留的近期结果更少，同时提前运行 summarize_verbose_tool_outputs()。
99%
Compact（LLM 介入）——build_compaction_payload() 构建 payload，发起真实 LLM 调用（独立 compaction model）生成 summary，替换中间消息段。失败则 fallback 到基础文本压缩。

ArtifactIndex 跨压缩存活：每次工具执行后调用 record_artifact(file_path, operation, details)，记录 read/write/edit 操作。压缩只影响 LLM 消息列表，ArtifactIndex 保持完整，agent 始终能知道自己操作过哪些文件。

compaction.rs · ArtifactIndex 记录
// 工具成功执行后自动追踪文件操作
match tool_name {
    "read_file"  => ("read", format!("{line_count} lines")),
    "write_file" => ("created", format!("{line_count} lines")),
    "edit_file"  => ("modified", "edit".to_string()),
    _             => return, // 非文件操作跳过
}
index.record(file_path, operation, &details); // 压缩后仍存活
vs OpenHands LLMSummarizingCondenser：OpenHands 用滑动窗口（head=1, tail=48，总预算=100 事件），超过阈值整体触发 LLM 压缩，只有两个状态（压缩/不压缩）。OpenDev 的 5 档设计允许在到达 LLM 压缩之前先做多轮低成本降级处理，更节约 token。
06

Sub-Agent 系统

OpenDev 的 sub-agent 通过 SubAgentSpec（14 字段）和 SubagentManager 管理，设计比 OpenHands 的 AgentDelegateAction 更具结构化。

AgentMode 枚举
三种运行模式
Primary：主 agent，完整 ReactLoop
Subagent：降级模式，reasoning effort 从 high→medium
All：不限制，主子 agent 共享配置
SubAgentSpec · 14 字段
结构化 Agent 规格
包含：name、description、system_prompt、model（覆盖主模型）、hidden（不对用户展示）、color（UI 颜色标识）、mode、permission 等。
两种 Runner
SimpleReact vs Standard
SimpleReactRunner：精简版，只读工具并行批量执行，跳过 doom loop/todo/压缩，适合 Explore 类子 agent
StandardReactRunner：封装完整 ReactLoop，含所有检测机制
项目结构自动探测
scan_project_structure()
递归 3 层，跳过 26 类噪音目录（.git/node_modules/target 等），最多捕获 100 条目。自动注入到 sub-agent system prompt。

Glob 权限系统（permissions.rs）是 OpenDev 独有的精细化控制机制：

Allow
直接执行，无需用户审批
Deny
拒绝工具调用，返回错误
Ask
暂停并弹出用户审批请求
permissions.rs · 特异性评分 + Glob 匹配示例
// 特异性规则：更具体的 pattern 优先
// "*"         → specificity = 0
// "git *"     → specificity = 4  (4 个非通配���字符)
// "git status" → specificity = 10 (精确匹配)

// 典型权限配置示例：
{
  "bash": {
    "*": "ask",         // 默认需审批
    "git *": "allow",   // git 命令允许
    "rm -rf *": "deny" // 破坏性操作拒绝
  },
  "edit_file": "deny"  // 只读 agent 禁止编辑
}

Observation-based 续行（SimpleReactRunner）：连续 2 次无工具调用才真正停止——第 1 次发 continuation nudge，确保 agent 有充分机会表达最终结论，避免过早退出。

vs OpenHands：OpenHands 的 sub-agent 通过 AgentDelegateAction 中断父 agent，结果以 AgentDelegateObservation 写回共享 EventStream——父子串行，事件交织。OpenDev 的子 agent 有独立消息列表和权限隔离，多个子 agent 可真正并发（取决于调用层面）。两种 Runner 的分离尤为实用：Explore 类任务用 SimpleReactRunner 跳过所有开销，效率更高。
07

Prompt Composer · 编译时模板系统

提示系统分两层：embedded.rs（compile-time 常量）+ composer/factories.rs（运行时条件组装）。

L1
79 个 include_str! 编译时常量
所有 .md 模板在编译时通过 include_str!("../../templates/...") 打包入二进制。按目录分组：system/（main、init、compaction）、system/main/（action-safety、tools、code-quality…）、subagents/（code-explorer、planner、ask-user…）、memory/（sentiment、topic、update）、generators/（generator-agent/skill）等。
L2
PromptSection · 优先级 + 条件门控
每个 prompt section 有 priority 值（8 个优先级档位）和条件过滤器（provider、model、mode）。create_default_composer() 注册约 20 个 section，按优先级排序后拼接成最终 system prompt。
L3
Cache-Aware 两段拆分
Prompt 被拆分为两部分：静态头部（不随对话变化，可被 prompt caching 缓存）+ 动态尾部（含当前任务、工具列表等）。最大化 provider-side prompt cache 命中率，降低每轮 LLM 调用成本。
L4
5-Part Sub-Agent System Prompt 组装
spawn.rs 为每个子 agent 构建专属 system prompt：① 子 agent 角色定义 ② 项目结构（scan_project_structure 输出）③ 父 agent 提供的 context ④ 可用工具列表 ⑤ 子 agent 自定义 instructions（来自 SubAgentSpec.system_prompt）。
vs OpenHands：OpenHands 的 LLMSummarizingCondenser prompt 硬编码在 Python 文件的多行字符串里（L72-118），无组合机制。OpenDev 的模板系统可在编译时组合不同模板，支持 provider-specific 差异化（如 Anthropic 和 OpenAI 格式差异），架构上更工程化。
08

Trace DAG 可视化

Web UI（web-ui/src/utils/trace/）实现了一套完整的 DAG 流水线，把线性轨迹转化为有向无环图展示，揭示并行工具调用和 sub-agent 分叉关系。

原始 Trace Events
→
buildGraph()
→
mergeToolCallNodes()
→
collapseGraph()
→
layoutGraph()
→
DAG 渲染
buildGraph()
事件 → 图节点
遍历 trace events，按依赖关系（tool_call_id、parent、cause）建立有向边。同一 LLM call 发出的多个工具调用会形成并行分支节点。
mergeToolCallNodes()
工具调用合并
将同批次并行执行的工具调用节点合并为一个 ToolNode 群组，减少视觉噪音，同时保留单独展开的能力。
collapseGraph()
子图折叠
Sub-agent 调用形成独立子图，默认折叠为 CollapsedNode，点击展开。支持 TaskNode（高层任务）和 TraceNode（细粒度操作）两种粒度。
layoutGraph()
自动布局
按 DAG 拓扑排序进行层级布局，时间轴从上到下，并行分支横向展开，使多工具并发关系一目了然。
vs OpenHands：OpenHands 的 Web UI 显示线性 EventStream（时间线列表视图），没有 DAG 结构。OpenDev 的图结构揭示了并行工具调用之间的真实依赖关系，对于理解 agent 决策路径更直观。对 mid-training 的启示：如果我们的轨迹采集工具能输出 DAG 格式，可以更方便地识别"哪些步骤可以并行执行"，辅助数据质量分析。
09

OpenDev vs OpenHands · 创新对比

两者都是 ReAct 模式的 code agent 脚手架，目标相似但工程决策不同。下表聚焦 OpenDev 相对 OpenHands 的显著差异：
