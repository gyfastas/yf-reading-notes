Moatless Tools

Intelligent Code Editing for Large Codebases
⌃ aorwall/moatless-tools ⌃ moatless-tree-search Albert Orwall 70.8% SWE-bench Verified
Moatless Tools 是一个基于 tree-sitter 代码理解的代码编辑框架，专为 SWE-bench 任务设计。其核心创新在于将代码编辑建模为有限状态机（FSM），具有显式状态转换：Search → Identify → PlanToCode → EditCode。每个状态转换被完整记录，天然产出结构化轨迹数据。使用 Claude 4 Sonnet 在 SWE-bench Verified 上达到 70.8%，成本仅 $0.63/instance。同时也是 moatless-tree-search（MCTS for code editing）的基础框架，支持通过蒙特卡洛树搜索探索多条编辑路径，为训练数据生成提供了大规模多样性轨迹的可能。

01

有限状态机架构

Moatless Tools 的核心设计是将代码修复任务分解为显式的状态转换序列。与 OpenHands 的自由 agent loop 或 SWE-agent 的命令行交互不同，Moatless 将整个过程建模为有限状态机，每个状态有明确的输入、输出和转换条件。

FSM 状态转换图
Search —— 找到相关代码？ —→ Identify
                                               ↑ 不够 ←————————↙

Identify —— 上下文充分？ —→ PlanToCode

PlanToCode —→ EditCode （正常编辑）
PlanToCode —→ ClarifyChange （范围太大时收窄）

EditCode —→ Finished / Rejected （终态）
S1
Search — 代码搜索
LLM 通过 function calling 搜索相关代码。可用搜索函数包括：query（语义搜索）、class_name（按类名定位）、function_name（按函数名定位）、code_snippet（按代码片段匹配）。LLM 决定使用哪种搜索以及搜索什么内容，系统返回匹配的 code spans。
S2
Identify — 判断上下文是否充分
LLM 审视当前已收集到的 code spans，判断是否有足够的上下文来制定修改方案。如果不够 → 返回 Search 继续搜索；如果足够 → 进入 PlanToCode。这个"回退循环"确保 agent 在动手修改前确实理解了问题的完整上下文。
S3
PlanToCode — 规划具体修改
将变更请求分解为更小的编辑单元，每个编辑精确定位到特定的 code span。输出结构化修改计划，包括目标文件、目标 span、修改内容描述。如果某个修改涉及的代码范围太大，自动转入 ClarifyChange。
S3b
ClarifyChange — 收窄修改范围
当 PlanToCode 提出的修改涉及过大的代码区域时触发。要求 LLM 进一步明确：具体是哪个类/函数/代码块需要修改？收窄到精确的 code span 后返回 PlanToCode。
S4
EditCode — 执行代码编辑
使用 SEARCH/REPLACE block 格式（灵感来自 Aider）执行实际编辑。关键设计：SEARCH 部分由系统预填充（从已确定的 code span 中提取），LLM 只需要写 REPLACE 部分。这大幅降低了 LLM 的负担，避免了"搜索内容写错导致编辑失败"的常见问题。
T
Finished / Rejected — 终态
Finished：所有计划的编辑已完成，生成最终 git diff patch。Rejected：LLM 判断无法完成修复（例如问题超出代码修改范围，需要更多信息等），终止流程并说明原因。
◆
FSM 的关键优势：每个状态转换都被完整记录到 trajectory.json，天然形成结构化轨迹。相比自由 agent loop 产出的"扁平"action-observation 序列，FSM 轨迹自带阶段标注——每条记录不仅有"做了什么"，还有"处于什么阶段"。这为训练数据提供了更丰富的监督信号：可以分阶段训练（搜索能力、判断能力、编辑能力），而不只是端到端模仿。
02

Code Spans 概念

Code Span 是 Moatless Tools 的核心抽象：代码库中与当前问题相关的特定部分。不是"文件"级别，也不是"行"级别，而是由 tree-sitter 解析得到的语义单元——类定义、函数体、方法签名等。

Core Concept
Code Span 定义
一个 code span = (file_path, span_id)，其中 span_id 由 tree-sitter 解析确定，对应类边界、函数边界、方法边界等语法结构。code span 是 Moatless 中上下文管理和编辑定位的最小单元。
Tree-sitter
语义解析
通过 tree-sitter 将源代码解析为 AST（抽象语法树），提取类边界、函数边界等结构化信息。相比基于行号的切割，code span 天然尊重代码的语义结构，不会在函数中间断开。
Context Mgmt
上下文追踪
系统在整个 FSM 过程中维护一组"当前可见"的 code spans（File Context）。Search 阶段添加 spans，Identify 阶段确认 spans，PlanToCode/EditCode 阶段精确定位到特定 span 进行修改。始终知道 LLM 当前"看到"了什么代码。
Edit Target
编辑定位
在 EditCode 阶段，系统从目标 code span 中提取代码作为 SEARCH block 的内容，预填充给 LLM。LLM 只需要产出 REPLACE 部分。这意味着编辑精度不依赖 LLM 准确复述原始代码——code span 确保了定位的精确性。
EpicSplitter · 层次化分块策略

EpicSplitter 是 Moatless 的代码分块器，为 embedding 索引准备代码块。其设计在 tree-sitter 解析基础上加入了"层次化块合并"逻辑：

750 tokens
目标 chunk 大小
100 ~ 1500
最小 ~ 最大 tokens
6000 max
硬上限（不可突破）
EpicSplitter · 分块流程 
# 1. tree-sitter 解析 → CodeBlock 层次结构
code_blocks = CodeParser(language).parse(source_code)
#    MODULE → CLASS → FUNCTION → nested blocks

# 2. 按 CodeBlock 边界初步切分
raw_chunks = split_by_blocks(code_blocks, target=750)

# 3. 注释处理策略（三种模式）
EXCLUDE   # 注释不纳入 chunk
INCLUDE   # 注释纳入最近的 chunk
ASSOCIATE # 注释与其关联的代码块绑定（跟随函数/类声明）

# 4. Merge pass: 合并过小的 chunk
for chunk in raw_chunks:
    if chunk.tokens < 100:
        merge_with_adjacent(chunk)  # 与相邻 chunk 合并
ℹ
与 Aider RepoMap 的区别：Aider 也使用 tree-sitter，但其 RepoMap 侧重于生成整个仓库的骨架视图（只展示类名、函数签名），用于帮助 LLM 定位。Moatless 的 code span 则是完整保留代码内容的精确切片，用于直接作为编辑上下文。两者互补：RepoMap 回答"代码库长什么样"，Code Span 回答"需要改的那段代码是什么"。
03

代码索引

代码索引是 Moatless Tools 搜索能力的基础。系统在处理代码库时构建多层索引，支持从语义搜索到精确定位的多种查询方式。

L1
CodeParser → CodeBlock 层次结构
tree-sitter 解析器将源代码转换为 CodeBlock 层次树。每个 CodeBlock 记录其类型（MODULE、CLASS、FUNCTION、TEST_SUITE）、起止行号、内容摘要。Python 有完整的 tree-sitter 支持，其他语言也在逐步添加。
L2
EpicSplitter → Embedding-ready Chunks
将 CodeBlock 层次结构切分为适合 embedding 的代码块（750 token 目标），保持语义完整性。每个 chunk 关联其来源 CodeBlock 的元信息，支持从 embedding 搜索结果反向定位到 code span。
L3
Semantic Search → 初始代码发现
通过向量 embedding 实现语义搜索，支持自然语言查询。LLM 在 Search 状态下使用 query 函数时，系统计算 query embedding，在代码块索引中做 cosine similarity 搜索，返回 top-k 匹配结果。
L4
File Context → 运行时 Span 追踪
File Context 维护当前 session 中"可见"的 code spans 集合。每次 Search 返回新结果，相关 spans 被加入 File Context；每次 Identify 确认后，spans 被标记为"已确认"。整个过程中，系统始终知道 LLM 当前看到了哪些代码片段。
CodeBlock 类型层次 
class CodeBlockType(Enum):
    MODULE      = "module"       # 文件顶层
    CLASS       = "class"        # 类定义
    FUNCTION    = "function"     # 函数/方法定义
    TEST_SUITE  = "test_suite"   # 测试类/测试集
    TEST_CASE   = "test_case"    # 单个测试用例
    CONSTRUCTOR = "constructor"  # __init__ 等构造方法
    IMPORT      = "import"       # import 语句块
    COMMENT     = "comment"      # 注释/文档字符串

# 典型层次：
# MODULE
#   ├─ IMPORT
#   ├─ CLASS
#   │   ├─ CONSTRUCTOR
#   │   ├─ FUNCTION (method)
#   │   └─ FUNCTION (method)
#   └─ TEST_SUITE
#       ├─ TEST_CASE
#       └─ TEST_CASE
✓
tree-sitter 是通用基础设施：Moatless、Aider、SWE-agent（v2 的 ACI 中）都使用 tree-sitter 进行代码理解。它的跨语言 AST 能力使得代码结构解析不依赖正则表达式或启发式规则。对于 agentic coding 训练数据生成，tree-sitter 级别的代码理解应被视为标配能力。
04

Agent 实现

Moatless Tools 的 agent 层围绕 ActionAgent 构建，负责在 FSM 各状态中调用 LLM 生成并执行 actions。

4.1  Action 类型
FindClass
在索引中按类名搜索，返回匹配类的 code spans。支持精确匹配和模糊匹配。
FindFunction
在索引中按函数名搜索，支持限定搜索范围（特定文件/类内）。
FindCodeSnippet
按代码片段内容搜索（字符串匹配），用于定位包含特定表达式的代码。
SemanticSearch
通过 embedding 向量执行语义搜索，支持自然语言 query。
ViewCode
查看指定文件的指定 code span 内容，将其加入当前 File Context。
StringReplace
核心编辑操作：SEARCH/REPLACE block。SEARCH 部分由系统从 code span 预填充，LLM 只输出 REPLACE 部分。
CreateFile
创建新文件并写入内容。用于需要添加新模块/测试文件的场景。
AppendString
向文件末尾追加内容，不需要指定 SEARCH block。用于添加新函数/类。
RunTests
执行测试用例验证修改正确性。支持指定测试文件/测试函数。
Finish
标记任务完成，生成最终 git diff patch。触发 FSM 进入 Finished 终态。
Reject
标记任务无法完成，附带拒绝原因。触发 FSM 进入 Rejected 终态。
4.2  完成模式
tool_call 模式
LLM 以 function calling / tool_use 格式输出 action。每个 action 是一个结构化的 tool call（函数名 + JSON 参数）。适合 Claude、GPT 等原生支持 tool calling 的模型。
ReAct 模式
LLM 以 Thought / Action / Observation 格式输出。Action 部分为文本描述，系统解析提取。适合不支持 tool calling 的开源模型。两种模式产出相同的 action 序列。
4.3  Memory · 对话管理

MessageHistoryGenerator 负责将 FSM 状态转换历史转换为 LLM 可消费的消息序列。它不仅记录 action-observation 对，还记录状态转换信息——LLM 在每步决策时，可以看到当前处于 FSM 的哪个状态，这是 Moatless 与自由 agent loop 的本质区别。

4.4  moatless-tree-search · MCTS 探索

moatless-tree-search 是构建在 Moatless Tools 之上的蒙特卡洛树搜索（MCTS）变体，用于探索多条代码编辑路径。

MCTS Core
搜索树结构
FSM 的每个状态转换对应搜索树的一条边。在每个状态节点，MCTS 可以探索多种 action 选择（如不同的搜索策略、不同的编辑方案），而非只采纳一条路径。每条路径最终产出一个 patch，通过测试验证质量。
Training Data
多样性轨迹
核心价值：每个 SWE 问题可以生成数百条不同的解题轨迹。成功的路径提供正样本，失败的路径提供负样本。这是 rejection sampling 和 verifier 训练的理想数据源——SWE-Gym 就是基于 Moatless 轨迹训练 verifier。
◆
MCTS + FSM 的协同效应：FSM 的显式状态结构使 MCTS 能够在语义有意义的节点上分支——而不是在任意 action 上分支。例如，可以固定 Search 阶段的结果，只在 EditCode 阶段探索多种编辑方案。这比在自由 agent loop 上做 MCTS 效率高得多，因为搜索空间通过 FSM 被结构化地限制了。
05

Workspace 系统

Workspace 类是 Moatless Tools 的顶层编排器，协调 Repository、CodeIndex、RuntimeEnvironment 三个子系统的交互。

Core
Workspace 类
统一管理 Repository（git 仓库操作：clone、checkout、diff）+ CodeIndex（代码索引和搜索）+ RuntimeEnvironment（测试执行环境）。三者通过 Workspace 协调，确保索引与仓库状态一致。
Artifacts
Artifact Handlers
多态数据管理：不同类型的产出物（code patches、test results、trajectory logs）通过统一的 Artifact Handler 接口管理。支持序列化/反序列化，方便持久化和跨 session 复用。
Tracking
File Context 追踪
维护编辑前/后的 code span 差异（pre/post edit diffing）。每次 EditCode 执行后，系统自动比对修改前后的 code spans，更新 File Context 中的 span 信息，确保后续编辑步骤基于最新的代码状态。
Persistence
trajectory.json
每次运行产出完整的 trajectory.json，记录从 Search 到 Finished 的所有状态转换、actions、observations、模型元数据。这是评测和训练数据的主要来源，也是与 SWE-bench 评测框架对接的标准接口。
Workspace · 典型初始化流程 
# 1. 从 SWE-bench instance 初始化 workspace
workspace = Workspace.from_swe_instance(
    instance_id="django__django-11099",
    repo_base_dir="/repos",
)

# 2. Workspace 内部自动执行：
#    a) git clone + checkout 到指定 commit
#    b) tree-sitter 解析 → CodeBlock 层次
#    c) EpicSplitter → embedding chunks
#    d) 建立向量索引

# 3. 创建 FSM 并运行
trajectory = workspace.run(
    problem_statement=instance["problem_statement"],
    model="claude-sonnet-4-20250514",
    max_transitions=25,
)

# 4. 产出
trajectory.save("trajectory.json")
patch = workspace.get_diff()  # git diff → SWE-bench submission
06

轨迹格式

Moatless Tools 的轨迹格式是其最重要的产出之一——不仅用于 SWE-bench 评测提交，更是下游训练数据的主要来源。

6.1  trajectory.json 结构
trajectory.json · 顶层结构 
{
  "instance_id": "django__django-11099",
  "model_name_or_path": "claude-sonnet-4-20250514",
  "model_patch": "diff --git a/django/db/... (git diff)",

  "transitions": [
    {
      "state": "SearchCode",
      "actions": [
        {
          "action_type": "FindClass",
          "args": { "class_name": "QuerySet" },
          "observation": { /* matched code spans */ }
        }
      ],
      "model_metadata": {
        "model": "claude-sonnet-4-20250514",
        "tokens_input": 3842,
        "tokens_output": 156,
        "cost": 0.014
      },
      "next_state": "IdentifyCode"
    },
    {
      "state": "IdentifyCode",
      /* ... */
      "next_state": "PlanToCode"
    },
    /* ... more transitions ... */
  ]
}
6.2  SWE-bench 评测对接

Moatless 的 trajectory.json 直接满足 SWE-bench 的提交格式要求：

必填字段
model_name_or_path
模型标识符，如 claude-sonnet-4-20250514。用于 SWE-bench leaderboard 标识提交来源。
必填字段
model_patch
git diff 格式的代码修改。这是 SWE-bench 实际应用并运行测试验证的内容。由 Workspace 在 FSM 结束后自动生成。
必填字段
instance_id
SWE-bench 实例标识符，如 django__django-11099。用于将 patch 与对应的问题对齐。
可选字段
transitions
完整的 FSM 状态转换序列，包含每步的 actions、observations、model metadata。SWE-bench 评测不需要此字段，但它是训练数据采集的核心。
6.3  训练数据用途

Moatless 轨迹已在多个研究中被用作训练数据源：

A
SWE-Gym — Verifier 训练
SWE-Gym 使用 Moatless 生成的大量轨迹（成功 + 失败）来训练 process reward model / outcome verifier。FSM 的结构化状态信息提供了比扁平 action 序列更丰富的监督信号——可以在每个状态转换点评估决策质量。
B
Rejection Sampling — 高质量轨迹筛选
通过 MCTS 为每个问题生成大量候选轨迹，按 reward threshold 过滤，正样本上限为 2 条/instance（防止过拟合某种解法）。筛选后的轨迹用于 SFT/DPO 训练。
C
规模：75,000 轨迹
300 个 SWE 问题 × 每问题 250 条 MCTS 采样 = 约 75,000 条轨迹。这个数据集足够训练中等规模的 policy model 或 verifier，且由于 MCTS 的多样性探索，轨迹之间的策略差异较大。
⚠
轨迹多样性 vs 正确性：MCTS 生成的 75k 轨迹中只有部分通过了测试验证。使用时需注意区分正轨迹（通过测试）和负轨迹（未通过）。负轨迹对 verifier 训练有价值，但不应直接用于 SFT imitation learning。

与其他 Scaffold 的对比

将 Moatless Tools 与其他主流 code agent scaffold 从架构、代码理解、编辑方式、上下文管理、搜索策略五个维度对比：

Moatless SWE-agent Agentless Aider
架构 FSM (状态机) Agent loop Pipeline Chat loop
代码理解 tree-sitter spans ACI commands Skeleton / structure RepoMap (tree-sitter)
编辑方式 预填充 SEARCH/REPLACE Heredoc edit Sample + vote 多种格式
上下文管理 Code span tracking HistoryProcessors Single-pass Token-budgeted map
搜索策略 MCTS (tree-search) Sequential N/A N/A
状态可见性 FSM 阶段显式标注 隐式（在 prompt 中） 流水线阶段 无
轨迹结构 状态转换序列 扁平 action-obs 分阶段产出 对话轮次
关键差异深度解析
MOATLESS vs SWE-AGENT
SWE-agent 让 LLM 在一个类似命令行的环境中自由操作，没有预定义的阶段划分。Moatless 的 FSM 强制了"先搜索再编辑"的纪律，避免 LLM 在没有足够上下文时就开始编辑。代价是灵活性较低——某些任务可能不需要经过完整的 Search → Identify 流程。
MOATLESS vs AIDER
Aider 和 Moatless 都使用 tree-sitter 和 SEARCH/REPLACE，但设计哲学相反：Aider 面向人类开发者（chat loop，人在环中），Moatless 面向全自动化（FSM，无需人类干预）。Aider 的 RepoMap 是全局鸟瞰，Moatless 的 code span 是局部精确切片。
ℹ
编辑方式的关键差异：Moatless 的预填充 SEARCH block 是独特设计。SWE-agent 用 heredoc 编辑（LLM 需要精确指定行号范围），Aider 让 LLM 自己写 SEARCH 和 REPLACE（容易写错 SEARCH 导致匹配失败）。Moatless 通过 code span 系统预先确定了修改位置，LLM 只需要关注"改成什么"而不是"改哪里"。
08

对本研究的启发

FSM 状态 → 结构化轨迹，分解"编码"为不同阶段
Moatless 的 FSM 设计将一个笼统的"修复 bug"任务分解为 Search、Identify、Plan、Edit 四个清晰阶段。每个阶段对应不同的能力要求（信息检索、判断力、规划能力、代码生成）。对训练而言，这意味着可以按阶段提供不同的监督信号，而非端到端模仿一条长轨迹。阶段标注本身就是一种 process supervision。
预填充 SEARCH blocks：降低 LLM 负担，提升编辑精度
Moatless 的 SEARCH block 预填充设计巧妙地将代码编辑分成两个子问题："在哪改"（由 code span 系统确定）和"改成什么"（由 LLM 生成）。这对训练数据质量有直接影响——预填充减少了编辑失败率，意味着更多轨迹能走到最终状态，产出更多有效的正样本。
tree-sitter 是跨 scaffold 的通用代码理解基础设施
Moatless（code spans）、Aider（RepoMap）、SWE-agent（ACI）都依赖 tree-sitter 进行代码结构解析。tree-sitter 已经是 agentic coding 领域的事实标准。对本研究而言，无论最终采用哪个 scaffold 生成训练数据，tree-sitter 级别的代码理解都是基础能力。VLM 的 agentic 训练数据中应当包含结构化的代码表示（AST 级别），而不仅仅是纯文本。
MCTS 探索：大规模多样性轨迹生成的可行路径
moatless-tree-search 的 MCTS 方法证明了从少量问题中生成大量多样性轨迹的可行性（300 问题 × 250 样本 = 75k 轨迹）。对本研究的启示：不需要无限扩大问题集——在有限的高质量问题上通过 MCTS 做深度探索，可以获得足够规模且多样的训练数据。这与 rejection sampling + verifier training 的范式高度匹配。
Code span 是比"文件"更干净的上下文和编辑单元
传统 scaffold（如 SWE-agent）以文件为上下文单元，LLM 需要在整个文件中定位修改位置。Moatless 的 code span 将粒度降到函数/类/方法级别，大幅缩小了 LLM 需要处理的上下文范围。对训练数据而言，code span 级别的标注（"修改了哪个 span"）比文件级别的标注（"修改了哪个文件"）提供了更精确的监督。
