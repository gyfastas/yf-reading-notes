Aider

AI Pair Programming in Your Terminal
⌥ Aider-AI/aider 📖 aider.chat ⭐ 30k+ stars Paul Gauthier · 2023–present
Aider 是目前最流行的开源 AI pair programming 工具，定位于终端内的实时代码协作。其核心设计哲学围绕四个差异化维度展开：(1) Git-native——每次编辑自动提交到 git，保证完整的变更追溯与一键回滚；(2) RepoMap——基于 tree-sitter 的代码库理解系统，通过 PageRank 排序向 LLM 展示最相关的代码结构；(3) 多种编辑格式——针对不同模型能力设计了 whole / diff / SEARCH-REPLACE 等多种代码编辑协议，最大化编辑成功率；(4) Architect/Editor 模式——双模型分离推理与编辑，将"怎么改"和"执行改动"解耦。SWE-bench Lite 单模型 26.3%，Architect 模式在 polyglot 基准上达 85%。与 OpenHands 等全栈 agent 框架不同，Aider 更聚焦于代码编辑本身这一核心能力的极致优化，是 ACI（Agent-Computer Interface）设计的典型范例。

01

RepoMap 系统

RepoMap 是 Aider 最核心的创新之一——一个基于静态分析 + 图排序的代码库理解系统，让 LLM 无需阅读整个代码库就能理解项目结构和依赖关系。

RepoMap 构建流程
tree-sitter Parse (遍历所有源文件，提取 AST)
↓ definitions & references
Directed Graph (nodes=files, edges=dependency references)
↓ personalized PageRank
Relevance Ranking (基于 chat context 的个性化排序)
↓ token budget constraint
Repo Map Output (签名 + 关键标识符，非完整代码)
Step 1
tree-sitter 解析
使用 tree-sitter 解析所有源文件的 AST，提取两类信息：定义（classes, functions, methods 的名称和签名）和引用（代码中对其他符号的使用）。支持 Python, JavaScript, TypeScript, Go, Rust 等数十种语言。
Step 2
有向图构建
以文件为节点、依赖引用为有向边，构建代码库依赖图。如果文件 A 中引用了文件 B 中定义的类或函数，则 A → B 建立一条边。边的权重反映引用频率。
Step 3
个性化 PageRank
对图执行 PageRank with personalization，其中个性化向量基于当前 chat context（用户已添加的文件和讨论主题）。结果：距离当前工作上下文最"近"的文件获得最高分数。
Step 4
Token 预算裁剪
按 PageRank 分数降序排列文件，通过二分搜索找到在 token budget（--map-tokens，默认 1024）内能容纳的最优子集。输出的是函数/类签名和关键标识符，不包含完整代码。
关键配置参数
--map-tokens
RepoMap 的 token 预算，默认 1024 tokens。值越大 LLM 看到的上下文越多，但占用更多 context window。
--map-multiplier-no-files
当用户没有 /add 任何文件时，token 预算自动放大 2–8 倍（默认动态调整），让 LLM 有足够上下文理解代码库全貌。
--map-refresh
RepoMap 刷新策略。auto（默认）在文件变更时重新构建，always 每轮都刷新，files 只在 /add 变更时刷新。

◆
核心洞察：RepoMap 的本质是一种基于图的上下文检索系统。它不需要 embedding 模型或向量数据库，而是利用代码的结构化特性（AST + 引用关系），结合 PageRank 的 "重要性传播" 机制，自动找到与当前任务最相关的代码片段。这是一种可以被 evolved 的 "scaffold intelligence"。
02

编辑格式 (Edit Formats)

Aider 为不同模型能力设计了多种代码编辑协议，这是其 ACI（Agent-Computer Interface）设计的核心。编辑格式的选择直接影响模型性能——同一个模型在不同格式下成功率可以相差 20% 以上。

格式原理Token 消耗适用场景
whole 完整输出整个文件内容替换 最高 最简单最可靠；弱模型 fallback
diff Unified diff 格式（@@ 标记） 低 节省 token；需要模型理解 diff 语法
diff-fenced Diff 包裹在 markdown code fence 内 低 比 diff 更结构化，降低格式错误
udiff Unified diff（legacy 格式） 低 早期版本；已被 diff-fenced 取代
SEARCH/REPLACE 搜索-替换块结构 中等 现代模型首选；最佳平衡
editor-diff Architect 模式下的 diff 低 专用于双模型 Editor 角色
editor-whole Architect 模式下的 whole 最高 弱 Editor 模型的 fallback
SEARCH/REPLACE 格式 · 核心编辑协议

SEARCH/REPLACE 是 Aider 最具代表性的编辑格式，被大多数现代模型（GPT-4, Claude, DeepSeek）作为默认格式使用。其结构简洁，模型容易学习和遵循：

SEARCH/REPLACE Block · 编辑格式示例 
path/to/file.py
<<<<<<< SEARCH
def authenticate(user, password):
    result = check_credentials(user, password)
    if not result:
        return None
=======
def authenticate(user, password):
    result = check_credentials(user, password)
    if not result:
        raise ValueError("Invalid credentials")
>>>>>>> REPLACE

每个编辑块包含三个部分：文件路径（第一行）、SEARCH 部分（要匹配的原始代码）、REPLACE 部分（替换后的新代码）。一次 LLM 回复可以包含多个这样的块，对应多个文件的多处修改。

分层匹配机制 (Layered Matching)

LLM 生成的 SEARCH 块不总是和源代码完全一致（空白、缩进差异等）。Aider 使用一套四层渐进匹配策略来最大化编辑成功率：

L1
Exact Match（精确匹配）
SEARCH 内容与源文件中某段代码完全一致（包括空白字符）。这是最理想的情况，直接替换。
L2
Whitespace-Ignore Match（忽略空白匹配）
忽略行尾空白和空行差异后重新匹配。解决 LLM 生成代码时常见的尾部空格/空行不一致问题。
L3
Indentation-Preserving Match（保留缩进匹配）
当 LLM 输出的缩进级别与源码不同时（如 SEARCH 中用 4 空格但源码用 2 空格），自动检测缩进偏移并在 REPLACE 中保持原始缩进。
L4
Fuzzy Match（模糊匹配，difflib）
使用 Python difflib.SequenceMatcher 进行模糊匹配，找到相似度最高的代码段。这是最后的兜底策略，容忍 LLM 的轻微幻觉（如变量名拼写略有差异）。
⚠
错误反馈机制：当所有四层匹配都失败时，Aider 不是简单地报错，而是生成详细的不匹配解释——哪些行不一致、可能的原因、建议重试策略——然后将这些信息反馈给 LLM，让它在下一轮修正编辑。这种"错误即反馈"的设计显著提升了最终成功率。
◆
ACI 设计决策：编辑格式的选择是一个典型的 ACI 设计决策——同一个底层能力（代码编辑）通过不同的接口协议暴露给模型，模型性能差异巨大。这表明 agent 训练不仅要关注模型能力本身，还要关注与什么样的 scaffold 接口搭配。
03

Git 集成

Aider 的 Git-native 设计是其区别于其他 AI coding 工具的标志性特征。每一次成功的代码编辑都自动产生一个 git commit，使得整个 AI 辅助编码过程成为一条干净的、可回溯的 git 历史。

Auto-Commit
自动提交
每次 LLM 成功应用编辑后，Aider 自动执行 git add + git commit。Commit message 由 LLM 自动生成，格式遵循 conventional commits 风格。默认开启（--auto-commits），可通过 --no-auto-commits 关闭。
Dirty Commits
脏文件提交
当工作目录有未提交的修改时，--dirty-commits（默认开启）会在 Aider 开始编辑前先将这些修改提交为一个独立 commit，确保 Aider 的编辑和用户自己的编辑在 git 历史中清晰分离。
Diff Tracking
变更追踪
每次编辑后自动展示 git diff，用户可以即时看到 LLM 做了什么改动。配合终端语法高亮，修改内容一目了然。这种透明性是用户信任 AI 编辑的基础。
Undo Support
一键回滚
用户可以通过 /undo 命令或简单的 git revert 撤销 Aider 的任何一次编辑。由于每次编辑都是独立 commit，回滚粒度精确到单次 LLM 交互，不影响其他修改。
典型 Git 历史 · Aider session 
# git log --oneline 展示 Aider session 产生的 git 历史

a3f2e1d aider: Add error handling to authenticate() function
b7c4d9a aider: Refactor token validation logic in auth.py
e5a8f3c aider: Add unit tests for new authentication flow
1d9b2ef (user manual commit) WIP: started auth refactor
8f3a6c7 aider: Fix import ordering in test_auth.py

# 每个 "aider:" 前缀的 commit 是自动生成的
# Commit message 由 LLM 根据 diff 内容自动撰写
# 用户 commit 和 aider commit 在历史中清晰分离
✓
训练数据视角：Git-native 工作流意味着每一次 AI 编辑天然就是一个 (instruction, diff, commit_message) 三元组。用户的 chat 消息是 instruction，git diff 是 code change，LLM 生成的 commit message 是 summary。这种数据格式非常适合作为 SFT 训练样本，且清洗成本极低。
04

Architect/Editor 模式

Architect/Editor 是 Aider 的双模型分离方案——将"思考怎么改"和"执行改动"拆分给两个不同的模型，各自发挥优势。这是 Aider 在 polyglot benchmark 上取得 85% 成绩的关键架构。

Architect Model
角色：推理和方案设计

输入：用户请求 + RepoMap + 已添加的文件内容

输出：自然语言描述的解决方案——需要修改哪些文件、修改什么逻辑、注意什么边界条件

典型模型：o1-preview, Claude 3.5 Sonnet, GPT-4

关键特性：不需要遵循任何编辑格式，自由文本输出
Editor Model
角色：将方案转化为精确的代码编辑

输入：Architect 的方案描述 + 相关源文件

输出：结构化的 SEARCH/REPLACE 块（或其他编辑格式）

典型模型：DeepSeek Coder, Claude Haiku, GPT-3.5

关键特性：只需要精确遵循编辑格式，认知负担大幅降低

双模型工作流
⊕
Architect 推理阶段
Architect 模型接收用户请求和代码上下文，输出自然语言方案："需要在 auth.py 的 authenticate() 函数中，将 return None 改为 raise ValueError，同时在 test_auth.py 中添加对应的异常测试用例。"
⊕
Editor 执行阶段
Editor 模型接收 Architect 的方案描述和源文件，输出精确的 SEARCH/REPLACE 块。Editor 不需要理解"为什么改"，只需要精确地将方案转化为可应用的编辑操作。
⊕
应用与提交
Aider 应用 Editor 的编辑块（经四层匹配验证），自动 git commit。若匹配失败，错误信息反馈给 Editor 重试。
◆
对训练的启发：Architect/Editor 分离表明 reasoning trace 和 edit execution 可以被独立训练。一个模型可以先被训练成优秀的"代码方案设计者"（输入: 任务 + 上下文，输出: 自然语言方案），另一个模型训练成精确的"代码编辑执行者"（输入: 方案 + 源码，输出: SEARCH/REPLACE 块）。两个能力的训练数据也应该分别采集。
05

上下文管理

Aider 的上下文管理采用用户手动 + RepoMap 自动的混合策略。与 OpenHands 等全自动框架不同，Aider 将部分控制权交给用户，让用户决定哪些文件应该被 LLM 看到。

MANUAL
用户添加的文件 (/add)
用户通过 /add filename 将文件加入 chat context。这些文件的完整内容会被发送给 LLM，LLM 可以直接读取和编辑它们。这是 Aider 上下文的"第一层"——明确的、完整的文件内容。使用 /drop filename 移除，/clear 清空全部。
AUTO
RepoMap 自动包含
RepoMap 根据 PageRank 自动生成的代码库地图，包含函数签名和关键标识符（但不包含完整代码）。当用户没有 /add 任何文件时，RepoMap 的 token 预算自动放大，提供更多上下文。这是"第二层"——自动的、摘要级别的代码结构。
BUDGET
Token 预算监控
/tokens 命令显示当前 token 使用情况：system prompt、chat history、已添加文件内容、RepoMap 各占多少。用户可以据此决定是否需要 /drop 一些文件或 /clear 历史，避免 context window 溢出。Aider 不会自动压缩历史（不像 OpenHands 的 Condenser），溢出时需用户手动干预。
WEAK MODEL
弱模型限制
对于被 Aider 标记为"弱"的模型（如小参数量的开源模型），RepoMap 默认禁用。原因：弱模型可能无法正确解读代码地图的结构化信息，RepoMap 反而会造成困惑，导致编辑格式错误率上升。此时 Aider 退化为纯手动上下文模式。
上下文相关命令
/add <file>
将文件加入 chat context，LLM 可读取和编辑。支持 glob 模式如 /add src/*.py。
/drop <file>
从 chat context 中移除文件，释放 token 空间。移除后 LLM 无法编辑该文件。
/tokens
显示当前 token 使用详情：system prompt / history / added files / repo map 各部分占比。
/clear
清空 chat 历史（但保留已 /add 的文件）。用于 context window 接近上限时释放空间。
/read-only <file>
将文件加入 context 但标记为只读——LLM 可以阅读但不能编辑。适合提供参考代码。
/map
显示当前 RepoMap 内容。调试用，查看 LLM 实际看到的代码库结构。
ℹ
与 OpenHands 的对比：OpenHands 的上下文完全自动管理（agent 通过 FileReadAction 按需读取，Condenser 自动压缩历史）；Aider 则给用户更多控制权。这反映了两种不同的设计哲学——OpenHands 追求全自动 agent，Aider 追求人机协作的 pair programming。对训练数据而言，Aider 的 /add 模式意味着每条轨迹都有明确的"人为划定的上下文边界"，可以作为训练 context selection 能力的标签。
06

代码结构

Aider 的代码库组织清晰，核心模块围绕 aider/coders/（编辑引擎）和 aider/repomap.py（代码理解）两大支柱展开。

base_coder.py
Base Coder 类——所有编辑器的基类。负责 prompt 组装、LLM 交互、编辑应用、git commit 的完整生命周期管理。约 2000+ 行，是 Aider 最核心的文件。
editblock_coder.py
SEARCH/REPLACE 编辑格式实现。解析 LLM 输出中的 SEARCH/REPLACE 块，执行四层匹配逻辑，处理错误反馈。现代模型的默认编辑格式。
wholefile_coder.py
Whole File 编辑格式实现。解析 LLM 输出中的完整文件内容，直接替换目标文件。最简单但 token 消耗最大的格式。
udiff_coder.py
Unified Diff 编辑格式实现。解析标准 unified diff 格式，应用 patch。需要模型对 diff 语法有精确理解。
architect_coder.py
Architect/Editor 双模型模式实现。协调两个模型的调用链：Architect 推理 → Editor 编辑。管理两个独立的 LLM session。
repomap.py
RepoMap 系统。tree-sitter AST 解析、依赖图构建、PageRank 排序、token 预算裁剪的完整实现。约 800+ 行。
commands.py
用户命令处理。实现 /add, /drop, /tokens, /clear, /undo, /map 等所有用户交互命令。
models.py
模型注册与配置。管理模型元信息（context window 大小、支持的编辑格式、是否为弱模型、默认参数等），支持 100+ 种模型。
核心调用链
User Input → Coder.run()
    ↓ 组装 prompt（system + RepoMap + files + history）
    ↓ LLM.send_message()
    ↓ 解析 LLM response（按 edit format 分派）
    ↓ EditBlockCoder.apply_edits() (或 WholeFileCoder / UDiffCoder)
    ↓ 四层匹配 → 写文件
    ↓ git add + git commit (commit msg 由 LLM 生成)
07

对本研究的启发

RepoMap 的图排序是一种可进化的 "scaffold intelligence"
RepoMap 通过 tree-sitter + PageRank 自动识别与当前任务最相关的代码，这种能力完全由 scaffold 提供，模型不需要训练。但可以更进一步：将 RepoMap 的检索策略作为训练信号——让模型学习"什么代码上下文对当前任务有用"，将 scaffold 的智能内化为模型能力。这是从 "scaffold-dependent" 到 "scaffold-free" 的进化路径。
编辑格式选择是关键的 ACI 设计决策
Aider 的实验明确证明：同一个模型在不同编辑格式下性能差异巨大。这意味着 agent 的训练数据应该针对特定的编辑协议生成——使用 SEARCH/REPLACE 格式训练的模型不应该在 whole-file 格式的 scaffold 上评测，反之亦然。ACI 不是 scaffold 的实现细节，而是影响模型性能的核心设计维度。
Architect/Editor 分离暗示训练数据应分层
Architect/Editor 的成功表明 reasoning trace 和 edit execution 是两种不同的能力，可以（且应该）被独立训练。对 agentic midtrain 而言，这意味着训练数据应该包含两种类型：(1) 推理轨迹（问题分析 → 方案设计），用于训练"architect"能力；(2) 编辑执行轨迹（方案 + 源码 → 精确的代码修改），用于训练"editor"能力。混合训练可能导致两种能力互相干扰。
Git-native 意味着每次编辑都是训练样本
Aider 的 auto-commit 机制使每一次 AI 编辑自动产生一个 (user_message, git_diff, commit_message) 三元组。这是最干净的代码编辑训练数据格式——diff 精确记录了"改了什么"，commit message 概括了"为什么改"，user message 提供了原始意图。大规模采集 Aider session 数据可以低成本获得高质量的 code editing SFT 数据。
分层匹配机制对稳健编辑应用至关重要
Aider 的四层匹配策略（exact → whitespace-ignore → indentation-preserving → fuzzy）展示了一个关键事实：即使是最强的模型也无法保证 100% 精确的代码引用。这对任何使用 SEARCH/REPLACE 或 diff 格式的 agent scaffold 都有启发——必须在应用层实现容错机制。更重要的是，匹配失败时的错误反馈机制（将不匹配信息返回给模型重试）是 self-correction 的有效实现方式。
