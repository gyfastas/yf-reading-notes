CREATOR

Tool Creation for Disentangling Abstract and Concrete Reasoning of LLMs
↗ arXiv:2305.14318 Qian et al. · EMNLP 2023 Findings
CREATOR 将 tool creation 提升为推理管线的第一步，而非事后补充。核心观点：LLM 的推理困难往往来自抽象规划（知道该做什么）和具体执行（精确计算/操作）的纠缠。CREATOR 通过四阶段框架——Creation, Decision, Execution, Rectification——将"创建工具"作为显式步骤，使 LLM 先把具体计算逻辑封装为工具函数，再专注于抽象层面的决策和调度。
01

四阶段框架

CREATOR 的推理过程分为四个显式阶段，每个阶段解决不同层次的问题。

C
Creation — 工具创建
LLM 分析问题，主动创建解决该问题所需的工具函数（Python 代码）。不是从预定义工具库中选择，而是根据问题需求定制化生成。工具封装了具体计算逻辑（数学运算、字符串处理、数据转换等），使后续步骤无需处理低层细节。
D
Decision — 决策调度
LLM 决定如何使用创建的工具：选择哪个工具、传入什么参数、按什么顺序调用。这一步只涉及抽象层面的规划，不涉及具体计算。Decision 和 Creation 的分离是 CREATOR 的核心贡献。
E
Execution — 执行
实际执行工具调用代码，由 Python 解释器完成，不经过 LLM。这确保了计算的精确性——LLM 不擅长的精确算术、复杂逻辑判断都交给确定性代码处理。
R
Rectification — 纠错修正
如果执行失败（异常、结果不合理），LLM 分析错误原因并修正工具代码或调用参数。这形成了一个自我修复循环：Creation 的错误不会导致整体失败，而是通过 Rectification 迭代改进。
*
核心洞察——Disentanglement：传统 CoT 让 LLM 同时做抽象推理和具体计算，两者在 token 序列中交织，互相干扰。CREATOR 通过 Creation 步骤将具体计算提取为可执行代码，LLM 在 Decision 步骤只需做高层规划。这种抽象-具体解耦显著降低了每个步骤的认知负担。
02

与 LATM 对比

CREATOR
每个问题都创建专属工具，tool creation 是推理过程的一部分（在线、per-instance）。不需要工具库或 dispatcher。强调 abstract/concrete 解耦的认知价值。同一个 LLM 同时做 maker 和 user。适合长尾多样化问题。
LATM
一次创建、多次复用。Tool creation 和 usage 解耦为两个独立阶段，使用不同模型（强/弱）。强调成本摊销的经济价值。需要 tool cache 和 dispatcher 机制。适合可归类的批量任务。
i
两者并非互斥。CREATOR 的 per-instance creation 可以与 LATM 的 caching 结合：首次遇到新问题类型时用 CREATOR 流程创建工具，验证有效后存入 LATM cache，后续同类问题直接复用。这正是 scaffold co-evolution 中 scaffold library 的理想运作方式。
03

对本研究的启发

Abstract-Concrete 解耦指导训练数据设计
CREATOR 的 disentanglement 思想对训练数据格式有直接指导：轨迹中应该显式区分"工具创建/scaffold 定义"步骤和"工具使用/任务执行"步骤。在 SFT 数据中标注这两个层次，让 model 学会在不同层次上思考，而不是把 scaffold 设计和任务执行混为一体。
Rectification 循环 = scaffold 的自我修复机制
CREATOR 的 Rectification 阶段表明：工具创建不需要一次成功，iterative refinement 是 tool creation 的自然组成部分。在 scaffold co-evolution 中，新 scaffold 的"验证-修正"循环应被设计为一等公民，而不是异常处理路径。
CREATOR + LATM = scaffold 生命周期的完整模型
结合两篇工作：CREATOR 描述了 scaffold 的诞生过程（按需创建 + 验证 + 修正），LATM 描述了 scaffold 的复用机制（缓存 + 分发 + 成本摊销）。两者拼合就是 scaffold 从创建到成熟的完整生命周期，这正是 scaffold co-evolution 需要实现的。
