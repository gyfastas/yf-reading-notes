EUREKA

Human-Level Reward Design via Coding Large Language Models
↗ arXiv:2310.12931 Ma et al. · UPenn / NVIDIA · ICLR 2024
EUREKA 让 GPT-4 读取环境源代码，生成可执行的 Python reward 函数，然后通过 GPU 加速的 RL rollout 评估 reward 质量，再将 rollout 统计数据反馈给 GPT-4 进行 "reward reflection" 迭代改进。无需任何 task-specific prompt engineering 或预定义 reward template，EUREKA 在 29 个 IsaacGym 机器人操控任务中的 83% 上 超越了人类专家设计的 reward，首次实现了 shadow hand pen spinning 等高难度灵巧操控任务。
01

Reward 生成流程

1
环境源码作为 Context
将目标环境的 Python/YAML 源代码（observation space 定义、action space、物理参数等）直接作为 GPT-4 的输入 context。不做任何手工摘要或 prompt engineering——GPT-4 直接阅读原始代码理解环境语义。
2
生成 Reward 函数代码
GPT-4 输出一段 完整的 Python reward 函数，参数为 environment state，返回标量 reward 值。每次独立生成 K 个候选（采样多样性），形成 reward 种群。
3
GPU 并行 RL 评估
使用 IsaacGym（GPU 并行物理仿真）对每个候选 reward 做完整 RL 训练（PPO），获取 training curve 和 rollout 统计（成功率、reward 分布、各 component 的数值范围）。GPU 并行使评估成本可控。
4
Reward Reflection → 迭代改进
将 rollout 统计摘要反馈给 GPT-4，GPT-4 分析各 reward component 的数值行为（是否饱和、是否有 reward hacking），生成改进版 reward 函数。重复 3-5 轮。
02

Reward Reflection 机制

Reward Reflection 是 EUREKA 的核心创新。传统 reward shaping 是人类手工调参的反复试错；EUREKA 将此过程自动化为 LLM 的 reflection loop。

每轮 reflection 中，GPT-4 接收的信息包括：

- 当前 reward 代码（完整 Python 函数）
- 训练曲线摘要（episode reward 均值/方差随 step 的变化趋势）
- 各 reward component 的统计（均值、方差、极值，哪些 component 饱和或振荡）
- 任务成功率（ground-truth metric，不参与 RL 训练但用于评估）

GPT-4 基于这些数值信号做出决策：调整 component 权重、修改 reward 公式、添加/删除 component 或改变 shaping 策略。这本质上是 LLM 在"reward 程序空间"中做搜索，evaluation signal 是 RL rollout 的实际表现。

i
与 RLHF 的区别：RLHF 用人类偏好数据训练一个 reward model（neural network）；EUREKA 用 LLM 直接生成 reward function（symbolic code）。EUREKA 的 reward 是可解释的、可编辑的、可迭代改进的——而 neural reward model 是黑盒。

04

对本研究的启发

Reward function 是 scaffold 的极端形式
EUREKA 中 LLM 生成的 reward function 本质上就是一种 scaffold——它不直接执行任务，而是塑造 agent 的学习信号。在 scaffold co-evolution 中，reward/evaluation criteria 的自动设计（而不仅仅是 tool 和 prompt）也应被纳入 scaffold 进化的范畴。
源代码作��� context 的有效性
EUREKA 不做 prompt engineering，直接把环境源代码喂给 GPT-4，效果优于人工摘要。这验证了：对于代码空间的搜索任务，让 LLM 阅读原始代码比给它自然语言描述更有效。在 scaffold co-evolution 中，scaffold 的进化也应以代码为主要表示，LLM 直接在代码空间中搜索和修改。
Reflection Loop = scaffold 进化的通用模式
EUREKA 的 "generate → evaluate → reflect → improve" 循环是 scaffold 进化的通用范式。区别仅在于 evaluation signal 的来源：EUREKA 用 RL rollout，我们的场景可以用model 在 benchmark 上的表现作为 evaluation signal，驱动 scaffold 的迭代改进。
