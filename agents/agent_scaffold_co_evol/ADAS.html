ADAS: Discovering Novel Agent Architectures via Code-Space Search
↗ arXiv:2408.08435 Hu, Lu, Clune · UBC / Sakana AI · ICLR 2025
ADAS 提出 Meta Agent Search——一个 meta-agent（GPT-4）在代码空间中搜索新的 agent 架构。meta-agent 维护一个 archive（已发现的架构集合），每轮从 archive 中选择表现最好的架构作为起点，通过代码修改产生新变体，在目标 benchmark 上评估后决定是否加入 archive。实验发现的架构在 ARC、MGSM、MATH 等基准上超越了人类设计的 Chain-of-Thought、Self-Refine、LLM Debate 等框架，验证了 agent 架构搜索的可行性。
01

Meta Agent Search 算法

核心思想：agent 的架构本身可以用代码完整描述（prompt 模板、推理流程、工具调用方式、多轮交互逻辑），因此可以用 LLM 在代码空间中搜索最优架构。

1
从 Archive 采样起点
从当前 archive 中选择 fitness 最高的 K 个架构作为参考。Archive 初始化为几个基础架构（简单 CoT、basic prompting 等）。Meta-agent 读取这些架构的代码和评测分数。
2
Meta-Agent 生成新架构
GPT-4 作为 meta-agent，分析现有架构的优缺点（从代码和评测结果中推断），生成新的 agent 架构代码。可以是修改现有架构（添加 self-verification 步骤、改变 prompt 结构）或组合多个架构的优势。
3
Benchmark 评估
新架构在目标 benchmark（如 ARC 的子集）上运行，获取 accuracy 分数。评估使用与 archive 中所有架构相同的 benchmark 配置，确保可比性。
4
Archive 更新
新架构的代码、评测结果、meta-agent 的设计推理一起加入 archive。Archive 不做淘汰（只增不删），允许搜索历史中的"失败"架构也保留，它们可能启发未来的创新。
02

Archive 机制

Archive 内容
每条记录包含
完整 Python 代码（定义 agent 的推理流程）、benchmark 评测分数、meta-agent 生成时的 thought process（设计推理链）。代码是自包含的——可独立运行，不依赖 archive 中其他条目。
Archive 策略
只增不删
Archive 采用 monotonically growing 策略。所有生成的架构（无论好坏）都保留。这借鉴了 Novelty Search 的理念：当前看起来无用的架构可能是未来突破的 stepping stone。Meta-agent 在参考 archive 时会看到失败案例的教训。
i
与 POET 的 Archive 对比：POET 维护的是 (environment, agent) pair 的种群，有淘汰机制（不满足 minimal criteria 的被剔除）。ADAS 的 archive 更像一个知识库——记录所有尝试过的设计及其效果，供 meta-agent 学习。这种差异反映了搜索策略的不同：POET 是进化选择，ADAS 更接近经验驱动的创造。

04

对本研究的启发

ADAS 验证了 scaffold 进化的可行性和必要性
ADAS 是目前最直接的证据：自动搜索到的 agent scaffold 可以超越人类手工设计。这为 scaffold co-evolution 提供了最强有力的动机——如果固定的人类设计不是最优的，那么 scaffold 就应该与 model 一起进化。
Archive 机制 = scaffold library 的设计参考
ADAS 的 archive（代码 + 评测分数 + 设计推理）为我们的 scaffold library 提供了详细的 schema 设计参考。特别是"只增不删 + 保留设计推理"的策略，使 meta-agent 能从历史尝试中学习，这比简单的"保留最优 + 淘汰最差"更适合开放式搜索。
代码空间搜索 vs 参数空间搜索
ADAS 选择在 代码空间（完整 Python 程序）而非参数空间搜索 agent 架构。这是一个关键设计决策：代码空间的搜索粒度更粗但表达力更强，允许发现结构性创新（新增/删除推理步骤、改变控制流）而不仅仅是参数调优。在 scaffold co-evolution 中，scaffold 的进化也应在代码层面，而不是在连续参数空间中做梯度优化。
