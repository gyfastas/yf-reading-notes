EvoAgent

Towards Automatic Multi-Agent Generation via Evolutionary Algorithms
↗ arXiv:2406.14228 Yuan, Song, Chen et al. · NAACL 2025
EvoAgent 将进化算法的三大算子——变异（mutation）、交叉（crossover）、选择（selection）——应用于 multi-agent 系统的配置优化。每个"个体"是一个完整的 multi-agent 配置（agent 数量、角色定义、交互拓扑），通过进化算子自动搜索最优的 agent 团队组合。引入 LLM 质量检查模块（quality inspection module）作为适应度评估器，同时维护种群的任务表现和多样性。
01

进化算子设计

Operator 1
Mutation
对现有 multi-agent 配置进行随机扰动：修改某个 agent 的角色描述、调整 system prompt、添加/删除一个 agent、改变交互顺序。变异幅度可控——小变异微调 prompt，大变异重构整个团队。LLM 驱动的语义变异，不是随机字符串操作。
Operator 2
Crossover
取两个表现良好的 multi-agent 配置，组合它们的优势：从配置 A 中取 agent 角色定义，从配置 B 中取交互拓扑。LLM 判断哪些组件可以兼容组合，避免语义冲突。产生的后代既继承双亲优点，又可能产生新的涌现行为。
Operator 3
Selection
基于适应度分数（任务表现 + 多样性分数的加权和）进行锦标赛选择（tournament selection）。保留 top-K 个体进入下一代，淘汰低适应度个体。多样性分数防止种群收敛到单一配置。
i
与 POET 的进化类比：POET 进化的是 (环境参数, agent 策略权重)，EvoAgent 进化的是 (agent 角色定义, 交互拓扑, prompt 模板)。POET 操作连续数值向量，EvoAgent 操作自然语言/结构化配置。但核心逻辑一致：mutation 产生变异 → evaluation 评估适应度 → selection 保留最优。
02

适应度评估

EvoAgent 的适应度评估由两部分组成：

F1
Task Performance Score
multi-agent 团队在目标任务上的实际表现（accuracy、解题成功率等）。直接运行 agent 团队 + 收集输出 + 用 ground truth 评分。这是适应度的主要成分。
F2
LLM Quality Inspection Module
一个 LLM-as-judge模块，评估 multi-agent 配置本身的质量：角色定义是否清晰、是否有冗余 agent、交互逻辑是否合理、是否存在潜在的循环依赖。这个模块不运行任务，而是直接审查配置的结构合理性，类似代码 review。
*
双重评估的价值：纯 task performance 评估成本高（需要实际运行 multi-agent 对话）。LLM quality inspection 作为廉价的预筛选，可以快速淘汰明显不合理的配置（如两个 agent 角色完全重叠、交互链断裂），减少昂贵的 task evaluation 次数。
03

实验结果

EvoAgent 在多个需要 multi-agent 协作的任务上评估，包括辩论式推理、协作写作、科学问答。进化后的 agent 团队配置在所有任务上超越了手工设计的 multi-agent baselines（如固定的 Researcher-Writer-Reviewer 三角色、Debate 二角色等）。

关键发现：

- 进化倾向于产生 3-5 个 agent 的团队，过多或过少都不是最优
- 最优角色定义往往比人类设计的更具体（更窄的专业范围，更明确的职责边界）
- 交互拓扑不一定是全连接或线性的，有时星形拓扑（一个 coordinator + 多个 specialist）表现最好

04

对本研究的启发

Multi-Agent 配置 = scaffold 的一种特殊形式
EvoAgent 进化的"multi-agent 配置"（角色定义 + 交互拓扑 + prompt 模板）本质上就是一种 scaffold——它定义了 model 如何组织和协作来解决问题。在 scaffold co-evolution 中，scaffold 的搜索空间不应局限于单 agent 的 tool + prompt，还应包括multi-agent 协作拓扑的设计。
LLM-as-Judge 作为廉价预筛选
EvoAgent 的 LLM quality inspection 模块提供了一个实用策略：在昂贵的 benchmark 评估之前，用 LLM 快速审查 scaffold 配置的结构合理性。这对 scaffold co-evolution 的效率至关重要——不是所有 scaffold 变异都值得做 full evaluation，可以用 LLM 预筛掉明显不合理的。
���化算子可以被 LLM 语义化
EvoAgent 的 mutation 和 crossover 不是随机操作，而是 LLM 驱动的语义操作（理解配置的含义后做有意义的修改/组合）。这比传统进化算法更高效，因为 LLM 的语义理解能力大幅提升了变异的有效性。在 scaffold co-evolution 中也应采用类似的 LLM-guided evolution。
