TileLang: Bridge Programmability and Performance in Modern Neural Kernels

ICLR 2026 · Lei Wang, Yu Cheng, Yining Shi, et al. (PKU / TileAI / MS Research)
论文地址：https://github.com/tile-ai/tilelang
笔记整理于 2026-04-25

一句话总结

TileLang 是一个 tile-level 可编程 DSL，通过显式原语（内存分配、数据搬运、并行调度）和统一的 FTG（Fused Tile-level Dataflow Graph），在保持 Pythonic 简洁性（<70 LOC）的同时，实现了接近手写 CUDA 的性能，在 H100 上平均比 Triton 快 3.02×。

研究背景与动机

现代神经网络对高性能计算 kernel 的需求日益增长，尤其是注意力机制相关的内存密集型算子。但开发者面临两难选择：

  • 手写 CUDA：性能极致，但代码复杂（FlashMLA ~500 LOC）、开发成本高、难以跨硬件移植
  • Triton 等高级 DSL：代码简洁（FlashMLA Triton ~130 LOC），但性能只有手写 CUDA 的 14.2%，因为缺乏对 tile reuse、pipeline scheduling、内存放置的细粒度控制

核心矛盾：programmability vs. performance 之间的鸿沟需要一种新抽象来弥合。

核心方法

1. Tile-Level 编程模型

TileLang 将 tile（张量的超矩形切片）提升为一级公民，提供显式原语让开发者直接操控硬件资源：

原语功能
T.alloc_shared在共享内存（shared memory）分配 tile 缓冲区
T.alloc_fragment在寄存器文件（register file）放置累加器 tile
T.copy orchestrate 数据在内存层级间的搬运
T.annotate_layout / T.use_swizzle定义自定义内存布局，避免 bank conflict
T.Parallelwarp 级别的并行协作策略
T.Pipelined软件流水线调度，重叠计算与数据传输
T.gemmtile 级别的矩阵乘，自动映射到 Tensor Core

2. FTG：统一融合 Tile 级数据流图

TileLang 程序被表示为 FTG（Fused Tile-level Dataflow Graph）：

  • 每个节点 = tile 操作符（gemm、copy、reduce 等）
  • 每条边 = 数据依赖关系
  • FTG 捕获六个关键优化维度：tile size、内存放置、warp 分区、内存布局、软件流水线、向量化

3. 两阶段优化：Recommendation + Inference

基于 FTG，TileLang 提供互补的两阶段工作流：

  1. Tile Recommendation（推荐）：分析 FTG 和部分标注的配置，提供硬件感知的默认值（tile shape、内存放置、warp 分区）。这些推荐是高质量的起点，开发者可接受、调整或进一步微调。
  2. Tile Inference（推理）：通过约束传播自动补全剩余配置（tile size、内存布局、流水线调度、向量化）。自动对齐上下游的缓冲区形状、布局和内存分配，保证一致性和硬件效率。

设计哲学：开发者在 FTG 上描述计算（可选标注优化维度），系统自动完成低层配置的推断和优化。支持全自动和 hint-guided 两种使用模式。

4. FlashMLA 示例（核心案例）

with T.Kernel(batch, heads // min(block_H, kv_group_num), threads=256) as (bx, by):
    # 1. 在目标内存层定义缓冲区
    Q_shared = T.alloc_shared([block_H, dim], dtype)
    S_shared = T.alloc_shared([block_H, block_N], dtype)
    acc_s = T.alloc_fragment([block_H, block_N], accum_dtype)

    # 2. 定义期望的数据布局
    T.annotate_layout({{O_shared: T.make_swizzled_layout(O_shared)}})

    # 3. 数据拷贝
    T.copy(Q[bx, by*block_H:(by+1)*block_H, :], Q_shared)

    # 4. 自动流水线调度
    for k in T.Pipelined(T.ceildiv(seqlen_kv, block_N), num_stages=2):
        T.copy(KV[bx, k*block_N:(k+1)*block_N, cur_kv_head, :], KV_shared)
        T.clear(acc_s)
        # 5. Warp 分区协作
        T.gemm(Q_shared, KV_shared, acc_s, transpose_B=True,
               policy=T.GemmWarpPolicy.FullCol)

代码复杂度

  • 相比手写 CUDA：代码量减少 85.5%（FlashMLA: 73 LOC vs 500 LOC）
  • 相比Triton：代码量更少或持平，但性能显著更高
  • 相比ThunderKittens：性能持平 (0.99–1.11×)，代码量减少 77%

Ablation Study（FlashMLA）

从手动启发式版本开始，逐步启用各组件：

  • H100：tiling (+Tile) 带来 1.31×，warp partitioning (+Partition) 贡献最大（额外 4.34×）
  • MI300X：memory placement (+Alloc) 是主要优化（6.56×），结合 tiling 后再提升 1.75×

说明不同硬件上瓶颈维度不同，TileLang 的成本模型能针对性地推荐最优配置。

vs 最新 DSL（Helion / Gluon / Tilus）

GEMM 上 TileLang 比 Helion 快 1.15–1.62×，比 Gluon 快 1.52–1.83×，比 Tilus 快 1.87–2.12×，同时代码量更少。

与 DeepSeek-V4 的关联

DeepSeek-V4 技术报告（Section 3.2）明确提到使用 TileLang 来开发自定义 kernel：

  • 用于平衡 "development productivity and runtime efficiency"
  • 具体应用包括：MoE fused kernel、attention kernels（CSA + HCA 混合注意力架构）
  • TileLang 的硬件可移植性（NVIDIA + AMD）和 tile-level 控制，使 V4 能在不同部署环境中复用同一套 kernel 代码

Acknowledgement 中，TileLang 作者特别感谢 DeepSeek 的"pioneering adoption"和"significant code contributions"，表明双方是深度合作关系。

编译流程（5 阶段）

  1. 开发者用 Python DSL 编写 tile-level 代码
  2. Parser 将 AST 转换为 TileLang AST
  3. 从 TensorIR 构建 FTG
  4. 应用 TileLang 优化 pass（tile recommendation + inference）和 TVM pass
  5. Codegen 生成 CUDA / HIP / 其他后端代码

TileLang 构建在 TVM 后端之上，但核心贡献（tile 抽象、FTG IR、优化 pass）位于 TVM 之上。

重要 Takeaways

  1. Tile 是正确的抽象粒度——比 scalar 高级（可移植），比 threadblock 低级（可控），在 GPU 内存层级和计算单元之间提供了实用的平衡点。
  2. 显式控制 + 自动推断是最佳组合——开发者决定"做什么"（tile shape、内存放置、warp 策略），系统决定"怎么做"（布局、流水线、向量化）。这比纯自动化（Triton）或纯手动（CUDA）都更高效。
  3. FTG 是统一的优化载体——将 tile recommendation 和 inference 统一到同一张图上，使得跨算子的融合优化成为可能。
  4. 硬件可移植性——同一套 TileLang 代码可在 NVIDIA 和 AMD GPU 上运行，且都能达到接近各平台最优库的性能。
  5. 新兴算法的快速实验——Block Sparse Attention 只需在标准 MHA 基础上加 2 行代码；Vertical Slash Sparse Attention 等新型注意力变体可快速实现。

局限与未来方向

  • tile 级别以下的极细粒度硬件行为（如指令级调度）无法通过稳定可移植的 API 捕获
  • 目前主要面向 GPU，其他加速器（TPU、NPU）的支持有待扩展
  • 成本模型虽准确，但对全新硬件架构需要重新校准
  • 与自动调优（AutoTVM/AutoScheduler）的集成可进一步深化
