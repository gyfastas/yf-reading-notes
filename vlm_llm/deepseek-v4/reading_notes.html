一句话总结

DeepSeek-V4 通过混合注意力架构（CSA + HCA）、流形约束超连接（mHC）和 Muon 优化器，在保留 MoE + MTP 的基础上，实现了原生 100 万 token 上下文的高效支持。在 1M token 场景下，Pro 版本仅需 V3.2 的 27% FLOPs 和 10% KV Cache。

核心问题

推理模型的 Test-Time Scaling 被标准 Attention 的二次计算复杂度根本限制，这在超长上下文和推理过程中形成 prohibitive bottleneck。同时，长时域 Agent 工作流、大规模跨文档分析等场景对超高效长上下文支持提出了刚需。

核心矛盾： 现有开源模型虽然在通用能力上有所进步，但处理超长序列的核心架构低效问题仍未解决，限制了 test-time scaling 的收益和长时域任务的探索。

架构创新

1. Hybrid Attention: CSA + HCA

针对长上下文效率的混合注意力架构：

  • Compressed Sparse Attention (CSA)：沿序列维度压缩 KV cache，然后执行 DeepSeek Sparse Attention (DSA)
  • Heavily Compressed Attention (HCA)：对 KV cache 应用更激进的压缩，但保持 dense attention

两者结合，在不同层/场景下平衡精度和效率。

2. Manifold-Constrained Hyper-Connections (mHC)

增强传统残差连接：

  • 标准 Hyper-Connections (HC) 扩展残差流宽度，但多层堆叠时经常出现数值不稳定
  • mHC 的核心创新：将残差映射矩阵约束到双随机矩阵流形（Birkhoff polytope）
  • 保证谱范数 ≤ 1，残差变换 non-expansive，提升前向/反向传播稳定性
  • 输入/输出变换也通过 Sigmoid 约束为非负有界，避免信号抵消

3. Muon Optimizer

引入 Muon 作为训练优化器，带来更快的收敛速度和更好的训练稳定性。

4. 保留自 V3 的设计

  • DeepSeekMoE：细粒度 routed experts + shared experts
  • Multi-Token Prediction (MTP)：配置与 V3 相同
  • MoE 路由变化：affinity score 从 Sigmoid 改为 Sqrt(Softplus(·))；前若干层用 Hash routing 替代 dense FFN

训练与后训练

预训练

  • Flash: 32T tokens，Pro: 33T tokens
  • 预训练后原生支持 1M-length contexts
  • 基础设施：TileLang DSL、batch-invariant deterministic kernels、FP4 quantization-aware training

后训练：两阶段范式

  1. Domain Specialist 训练：数学、代码、Agent、指令遵循等领域分别独立训练 SFT + GRPO RL
  2. On-Policy Distillation (OPD)：统一模型作为 student，通过 reverse KL loss 向各领域 expert 学习
关键变化： 与 V3.2 不同，混合 RL 阶段被 OPD 完全取代。

Generative Reward Model (GRM)

后训练阶段不再使用传统 scalar reward model：

  • 对于难验证任务，使用 rubric-guided RL 数据 + GRM 评估策略轨迹
  • Actor 网络本身即为 GRM，联合优化生成能力和评估能力
  • 内部推理能力天然融入评估过程，产生高度鲁棒的打分
  • 仅需极少的人类标注即可泛化到复杂任务

三种推理模式

模式特征典型场景
Non-think快速直觉响应日常任务、紧急反应、低风险决策
Think High有意识逻辑分析，较慢但更准复杂问题求解、规划、中等风险决策
Think Max推理推到极限，慢但强大探索模型推理能力边界

Think Max 模式会注入特殊系统指令："Reasoning Effort: Absolute maximum with no shortcuts permitted."

关键基础设施优化

  • MoE 融合 kernel：计算、通信、内存访问完全重叠的单融合 kernel
  • TileLang DSL：平衡开发效率和运行时效率的领域特定语言
  • Batch-invariant deterministic kernels：确保训练和推理的 bit-wise 可复现性
  • FP4 QAT：MoE expert weights 和 indexer QK path 的 FP4 量化感知训练
  • heterogeneous KV cache：带磁盘存储策略的异构 KV cache 结构，支持高效共享前缀复用

重要 Takeaways

  1. 长上下文效率是下一代 LLM 的核心战场——V4 用架构创新（CSA+HCA）而非单纯堆参数，实现了 1M token 的原生高效支持。
  2. Muon + mHC 是训练稳定性和收敛的关键——mHC 的流形约束解决了 HC 的数值不稳定问题，Muon 加速了收敛。
  3. 后训练范式转变：OPD 取代混合 RL—— specialist → distillation 的两阶段范式，GRM 统一生成和评估。
  4. 三种推理模式对应不同计算-精度权衡——从快速直觉到极限推理，用户可按场景选择。
  5. Flash 版本证明架构效率 > 参数规模——13B activated 在多数任务上媲美或超越更大模型。

局限与未来方向

  • 为最小化风险保留了大量已验证的组件和 tricks，导致架构相对复杂，未来需更优雅地蒸馏
  • Anticipatory Routing 和 SwiGLU Clamping 的原理尚未充分理解
  • 将探索更多维度的模型稀疏性（如 sparse embedding modules）
  • 持续研究低延迟架构，使长上下文部署更 responsive
  • 长时域多轮 Agent 任务和 multimodal 能力是未来迭代方向

Q & A

Q1: DeepSeek-V4 的 MoE Pretrain 阶段情况如何？

V4 系列在 V3 预训练数据基础上构建了更多样化、更高质量、更长有效上下文的语料库，总计 32T+ tokens。关键特点：

  • 数据构成：网页数据（过滤批量自动生成和模板内容，缓解模型 collapse）、数学与编程语料（核心，mid-training 加入 agentic 数据）、多语言数据（更大语料，改善长尾知识）、长文档数据（科学论文、技术报告为重点）
  • Tokenizer：基于 V3 的 128K 词表，加入少量特殊 token；继承 token-splitting 和 Fill-in-Middle (FIM) 策略
  • 与 V3 的关键差异：使用 sample-level attention masking（而非 document-level）；不同来源文档打包到合适序列以最小化截断

Flash 训练设置（32T tokens）：

  • 优化器：Muon（大部分参数）+ AdamW（embedding、prediction head、RMSNorm）。Muon momentum=0.95，weight_decay=0.1，update RMS rescale=0.18
  • Batch size scheduling：从小逐步增加到 75.5M tokens，之后保持
  • LR：warmup 2000 steps → peak 2.7×10⁻⁴ → cosine decay 到 2.7×10⁻⁵
  • 序列长度渐进扩展：4K → 16K → 64K → 1M
  • 稀疏注意力引入：前 1T tokens 用 dense attention warmup；64K 时引入 sparse attention（先 short stage warmup lightning indexer）
  • Load balancing：auxiliary-loss-free，bias update speed=0.001；balance loss weight=0.0001（防止单序列内极端不平衡）
  • MTP loss weight：大部分训练 0.3，LR decay 开始时降至 0.1

Pro 训练设置（33T tokens）：与 Flash 基本一致，区别在于最大 batch size 94.4M tokens，peak LR 2.0×10⁻⁴，dense attention warmup 阶段更长。

Q2: 遇到 loss spike 是怎么解决的？

万亿参数 MoE 训练存在显著稳定性挑战。团队 empirically 发现 loss spike 的发生始终与 MoE 层的 outliers 相关，且路由机制本身会加剧异常值的出现。他们从两个维度解决：

① Anticipatory Routing（预判路由）

  • 核心思想：解耦 backbone 网络和路由网络的同步更新。在 step t 用当前参数 θ_t 计算特征，但用历史参数 θ_{t-Δt} 计算和应用路由索引
  • 工程实现：在 step t-Δt 提前获取 step t 的数据，"预判"地计算并缓存路由索引；预计算只需一次前向传播
  • 开销控制：精心编排流水线执行和 Expert Parallelism (EP) 通信重叠，额外 wall-clock 开销约 20%
  • 动态触发：自动检测 loss spike → 短回滚并激活 Anticipatory Routing → 运行一段时间后恢复标准训练。总体额外开销 negligible，不影响模型性能

② SwiGLU Clamping

  • SwiGLU 的线性分量 clamp 到 [-10, 10]
  • Gate 分量的上界限制为 10
  • 有效消除 outliers，大幅稳定训练，且不损害最终性能

论文坦承这两种技术的底层原理"remains an open question"，但实践上非常有效。

Q3: 前 1T tokens 用 dense attention warmup，那怎么切换为 CSA + HCA 的稀疏注意力？

切换遵循一个两阶段渐进策略，与序列长度扩展同步进行：

阶段一：Dense Attention Warmup（前 1T tokens）

  • 前 1T tokens 完全使用标准 dense attention（所有层都是稠密注意力）
  • 此阶段序列长度从 4K 逐步扩展到 16K、64K
  • 目的：让模型先学好基本的注意力模式，避免稀疏机制在早期训练不稳定

阶段二：引入 Sparse Attention（序列长度达到 64K 时）

  • 当训练进入 64K 序列长度阶段时，开始引入稀疏注意力
  • 引入时先设置一个 short stage 来 warmup lightning indexer（CSA 中负责 top-k 稀疏选择的核心组件），让其参数先稳定下来
  • 之后大部分训练使用 CSA + HCA 的混合稀疏注意力

架构层面的交错配置

  • Flash：前 2 层用纯 sliding window attention；后续 41 层中 CSA 和 HCA interleaved（交错）使用
  • Pro：前 2 层用 HCA；后续 59 层 CSA 和 HCA 同样交错使用
  • CSA（压缩率 m=4）负责稀疏注意力 + 压缩；HCA（压缩率 m'=128）负责更激进的压缩但不使用稀疏选择

Pro vs Flash 的差异

  • Pro 的 dense attention warmup 阶段更长（具体 token 数未披露），其余切换策略与 Flash 一致
  • 两版本都遵循同样的 two-stage training method

关键澄清：Dense Attention = 标准全注意力（无压缩、无选择机制）；Sparse Attention = 完整的 CSA+HCA 混合架构（CSA 内含 lightning indexer 做 top-k 选择）。切换是一次性的——在 64K 时间节点上从 dense 直接切到 sparse，不是"逐步引入"选择机制。Warmup lightning indexer 只是给 CSA 的内置组件一个短暂的参数稳定期，而非渐进添加。

Q4: DeepSeek-V4 测了哪些 Agent 榜单？

论文在 Section 5.3.1 中列出了完整的 Agent 评估套件，分为 Code Agent、Search Agent 和通用 Tool Agent 三类：

Code Agent 任务（内部框架，提供 bash + file-edit 工具，最多 500 步交互，最大 512K context）

  • Terminal Bench 2.0 (Merrill et al., 2026) — 也报告了 Verified subset，Pro 约 72.0 分
  • SWE-Verified (OpenAI, 2024e)
  • SWE Multilingual (Yang et al., 2025)
  • SWE-Pro (Deng et al., 2025)

Search Agent 任务（内部 harness，websearch + Python 工具，最多 500 步，最大 512K context）

  • BrowseComp (Wei et al., 2025)
  • HLE w/ tool — 用工具辅助的 HLE 评估

通用 Tool / MCP Agent 任务

  • MCPAtlas Public (Bandi et al., 2026) — 覆盖广泛的工具和 MCP 服务
  • GDPval-AA (AA, 2025; Patwardhan et al., 2025)
  • Tool-Decathlon (Li et al., 2025) — 论文中也简写为 Toolathlon

Formal Math（Agentic 设置）

  • Lean v4.28.0-rc1 — 最多 500 次 tool call，使用 Lean compiler 和 semantic tactic search engine

核心结论：

  • Code Agent：Pro 与 K2.6 / GLM-5.1 持平，但仍落后闭源模型（Claude Opus 4.5）
  • Flash 在 Code Agent 上弱于 Pro，尤其在 Terminal Bench 2.0 上差距明显
  • MCPAtlas 和 Toolathlon 上 Pro 表现突出，说明模型对多样工具和 MCP 服务的泛化能力强，不只是对内部框架过拟合
  • 内部评估中 Pro 超越 Claude Sonnet 4.5，接近 Opus 4.5
